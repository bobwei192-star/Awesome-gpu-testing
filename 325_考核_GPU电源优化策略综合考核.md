### 3.1 电源管理与调度器交互

**题目21**：请简述AMDGPU中调度器与电源管理的协同工作机制（包括调度器触发P-State切换的三种主要场景，以及SMU在其中的中介作用）。（10分）

**参考答案要点**：
- 场景1：命令提交时（amdgpu_cs_ioctl）→ 触发power_up → 提升P-State
- 场景2：页面翻转（Page Flip）→ 提高Display相关域频率
- 场景3：计算任务分派（amdkfd）→ 根据任务量调整GFX频率
- SMU中介作用：驱动通过SMU消息（SMU_MSG_SET_HARD_MIN/MAX）发送频率请求，SMU固件执行实际的P-State切换

### 3.2 自适应功耗管理算法

**题目22**：请解释PID控制器中Kp、Ki、Kd三个参数分别对功耗控制的影响，并说明在GPU场景中如何调优这三个参数。（10分）

**参考答案要点**：
- Kp（比例项）：决定对当前误差的响应速度，过大导致振荡，过小响应慢
- Ki（积分项）：消除稳态误差，过大引起积分饱和，过小无法消除稳态误差
- Kd（微分项）：抑制超调和振荡，对高频噪声敏感
- 调优方法：Ziegler-Nichols法或手动调优，先调Kp使系统临界振荡，再调Ki消除稳态误差，最后调Kd抑制超调

### 3.3 Idle检测与Clock Gating

**题目23**：请对比分析Coarse-Grained Clock Gating（CGCG）和Fine-Grained Clock Gating（FGCG）在实现复杂度、功耗节省、唤醒延迟和适用场景方面的差异。（10分）

**参考答案要点**：

| 维度 | CGCG | FGCG |
|------|------|------|
| 实现复杂度 | 低（全局门控） | 高（每CU独立控制） |
| 功耗节省 | 30-50%动态功耗 | 60-80%动态功耗 |
| 唤醒延迟 | 1-5个时钟周期 | 1-3个时钟周期 |
| 适用场景 | 引擎级空闲 | CU级/单元级空闲 |
| 控制粒度 | 整个GFX引擎 | 单个CU/TA/TD |

### 3.4 Power Gating与域控制

**题目24**：请描述GFXOFF从允许状态到完全关闭电源的完整序列，列出至少5个关键步骤及其目的。（10分）

**参考答案要点**：
1. 等待所有渲染/计算完成（RLC idle check）→ 确保无进行中的操作
2. 触发RLC进入GFXOFF（WREG32 RLC_GCR_CNTL）→ 启动硬件序列
3. 发送SMU消息请求电源门控（SMU_MSG_GFXOFF）→ 通知SMU关闭电源
4. SMU执行PG序列：隔离单元使能 → Header开关关闭 → Footer开关关闭
5. 电源轨关闭（VR输出0V）→ 完全消除泄漏/动态功耗
6. SMU确认PG完成 → 驱动更新状态为GFXOFF_STATE_ACTIVE

### 3.5 综合理解

**题目25**：请解释"功耗管理的盈亏平衡点"概念，并以PG为例，列出计算盈亏平衡点的关键参数和公式。（10分）

**参考答案要点**：
- 盈亏平衡点：进入和退出PG的总开销等于不进入PG时泄漏功耗累积的时间点
- 关键参数：进入能量(E_entry)、退出能量(E_exit)、空闲功耗(P_idle)、泄漏功耗(P_leak)
- 公式：T_break_even = (E_entry + E_exit) / (P_idle - P_leak)
- 典型值：GFXOFF约100μs，VCN约350μs

```c
static int gfx_v11_0_update_gfx_clock_gating(struct amdgpu_device *adev,
                                               bool enable)
{
    uint32_t data, def;

    if (enable && (adev->cg_flags & AMD_CG_SUPPORT_GFX_MGCG)) {
        /* 使能MGCG */
        data = RREG32_SOC15(GC, 0, regRLC_MGCG_CTRL);
        def = data;
        data &= ~RLC_MGCG_CTRL__GFX_MAX_BW_MASK;
        data |= (0x20 << RLC_MGCG_CTRL__GFX_MAX_BW_SHIFT);
        if (def != data)
            WREG32_SOC15(GC, 0, regRLC_MGCG_CTRL, data);
    }

    if (enable && (adev->cg_flags & AMD_CG_SUPPORT_GFX_CGCG)) {
        data = RREG32_SOC15(GC, 0, regRLC_CGCG_CTRL);
        def = data;
        data &= ~RLC_CGCG_CTRL__CGCG_MASK;
        data |= RLC_CGCG_CTRL__CGCG_EN;
        if (def != data)
            WREG32_SOC15(GC, 0, regRLC_CGCG_CTRL, data);
    }

    return 0;
}
```

4. GPU间功耗不均衡，部分GPU温度偏高

**设计要求**：

4. 预期的能效提升目标和验证方法（4分）

**参考答案要点**：
1. 技术策略：
   - 策略1：潮汐感知调度——在夜间低谷期启用更深的电源门控（如BACO），高峰期保持CU活跃
   - 策略2：GPU间负载均衡——根据温度/功耗数据动态分配推理请求
   - 策略3：智能GFXOFF——使用历史推理间隔预测，减少不必要的GFXOFF切换
   - 策略4：CU级精细门控——推理期间只保持必要数量的CU供电

2. GFXOFF诊断方案：
   - 检查deny_reasons位图分布 → 确认阻止GFXOFF主要原因
   - 使用ftrace追踪GFXOFF流程 → 定位失败阶段
   - 检查驱动版本 → 确认已知Bug修复状态
   - 检查BIOS/UEFI设置 → 确认未禁用ASPM/PG
   - 常见修复：驱动升级、禁用冲突的debugfs接口、调整空闲超时阈值

3. 数学模型：
   P_total = Σ(P_dynamic_i + P_leakage_i)
   P_dynamic_i = α_i * C_eff_i * V^2_i * f_i
   where α_i ∈ {0, 0.3, 0.7, 1.0} 对应 PG/CG/轻载/重载
   Performance_constraint: latency_p99 < 10ms
   Objective: minimize E_total = ∫P_total(t)dt
   Decision: for each idle interval T_idle:
     if T_idle > T_break_even_pg → enter PG
     elif T_idle > T_break_even_cg → enter CG
     else → stay active

4. 能效目标：idle功耗从45W降至10W（降幅78%）
   验证方法：A/B测试、长期监控（7天）、功耗采样频率1Hz

**问题要求**：
1. 设计针对视频播放场景的电源管理策略（重点关注VCN和Display域）（5分）
2. 设计从办公切换到游戏场景时，电源状态转换的最优路径（5分）
3. 当CPU和GPU同时存在功耗竞争时，如何分配TDP预算？请给出分配算法（5分）

**参考答案要点**：
1. 视频播放策略：
   - VCN进入D3Hot（解码完成后立即PG）
   - 显示刷新率动态降低（从60Hz到48Hz或30Hz）
   - GFX进入GFXOFF（无需渲染时）
   - Display域降低PHY电压（eDP低功耗模式）
   - 禁用不必要的CG引擎（SDMA、JPEG）

2. 办公→游戏转换：
   GFXOFF → GFX_POWER_ON → VMIN提升 → freq提升
   路径：GFXOFF_STATE_ALLOWED → GPU active → SMU消息FREQ_HIGH
   总延迟要求：<50ms（避免卡顿）

3. TDP分配算法：
   - 静态基线：CPU 5W + GPU 5W + SoC 5W = 15W
   - 动态分配：基于实时负载和温度
   - 公式：P_GPU_allowed = min(P_TDP - P_CPU - P_SoC, P_GPU_demand)
   - 优先级：游戏时GPU优先(70%)，办公时CPU优先(60%)

### 5.3 场景三：诊断与排错（15分）

**场景描述**：
用户报告在RDNA3 GPU上运行游戏时出现间歇性卡顿，卡顿发生时伴随以下现象：
1. GPU利用率瞬时从95%降至10%
2. 功耗瞬时从200W降至30W
3. GFXOFF计数器显示频繁的进入/退出（每秒约50次）
4. 卡顿间隔约500ms-2s，无明显规律
5. dmesg中未发现错误或警告

**问题要求**：
1. 分析可能的原因，解释为什么GFXOFF每秒50次的进入/退出会导致卡顿（5分）
2. 设计一个诊断方案，列出需要检查的5个关键指标和对应的检查命令（5分）
3. 提出至少2种解决方案（包括短期临时方案和长期修复方案）（5分）

**参考答案要点**：
1. 原因分析：
   - GFXOFF每进入/退出一次延迟约132μs，50次/秒→约6.6ms/s的延迟开销
   - GFXOFF退出时需要重新初始化状态、恢复寄存器，导致渲染管线排空
   - 唤醒后的前几帧性能不稳定（电压/频率尚未恢复到稳态）
   - 卡顿的根本原因：GFXOFF进入条件过于激进，导致频繁在渲染间隙进入深度休眠

2. 诊断方案：
   - 检查GFXOFF deny_reasons：cat /sys/class/drm/card0/device/gfxoff_status
   - 检查CG状态：cat /sys/class/drm/card0/device/cg_state
   - 检查GFXOFF计数器：cat /sys/class/drm/card0/device/gfxoff_count
   - 检查Power Profile：cat /sys/class/drm/card0/device/power_dpm_force_performance_level
   - 检查频率曲线：cat /sys/class/drm/card0/device/pp_od_clk_voltage
   - ftrace追踪GFXOFF事件：trace-cmd record -e amdgpu:amdgpu_gfxoff_entry

3. 解决方案：
   - 临时方案：禁用GFXOFF（echo manual > power_dpm_force_performance_level）
   - 临时方案：增加GFXOFF空闲超时阈值（修改idle_threshold参数）
   - 长期方案：实现自适应GFXOFF策略，根据历史帧间隔动态调整进入条件
   - 长期方案：增加游戏场景检测，在3D全屏模式下降低GFXOFF激进程度
   - 长期方案：实现CU级精细门控替代全GFXOFF，保留部分CU供电

### 8.1 实验一：GFXOFF状态观察

### 附录C：常见问题FAQ

**Q1: GFXOFF进入失败如何处理？**

A: 首先检查deny_reasons位图，确认被阻止的原因。常见原因包括：
- MMIO访问进行中：等待当前访问完成
- 寄存器编程中：确认驱动初始化完成
- 中断处理中：等待中断处理完成
- 帧缓冲器访问中：等待显示刷新完成

使用 `cat /sys/kernel/debug/dri/0/amdgpu_gfxoff_deny_reasons` 查看具体原因。

**Q2: 如何判断CG是否生效？**

A: 通过以下方法判断：
1. 观察CG寄存器值（确认使能位已置位）
2. 测量同场景下CG启用/禁用的功耗差异
3. 查看CG residency统计（确认CG状态时间占比）
4. 使用示波器测量时钟树上的时钟活动

**Q3: 为什么禁用GFXOFF后功耗反而降低？**

A: 这可能是因为GFXOFF过于频繁的进入/退出产生了较大的延迟开销。禁用GFXOFF后避免了频繁的状态切换，减少了切换开销。这通常发生在空闲间隔非常短的工作负载中（如办公应用），此时建议使用时钟门控替代电源门控。

**Q4: 如何优化视频播放的功耗？**

A: 优化策略：
1. 确保VCN在解码完成后进入D3Hot/D3Cold
2. 显示刷新率与视频帧率同步（如24fps视频使用48Hz刷新率）
3. GFX保持GFXOFF状态
4. Display域降低PHY驱动强度
5. 禁用不必要的SDMA和JPEG引擎

**Q5: 数据中心场景下如何管理大量GPU？**

A: 建议措施：
1. 使用统一的监控框架（如Prometheus + Node Exporter）采集所有GPU的功耗数据
2. 基于功耗数据实现负载均衡调度
3. 在低负载时段启用BACO（Bus Alive Chip Off）
4. 设置GFXOFF自适应策略，根据推理间隔动态调整阈值
5. 定期收集和分析GFXOFF/PG统计信息，发现异常的GPU

```c
static int smu_send_message(struct amdgpu_device *adev, uint32_t msg, uint32_t param)
{
    struct smu_context *smu = adev->smu;
    uint32_t resp_reg;
    int ret;
    unsigned long flags;

    spin_lock_irqsave(&smu_msg_lock, flags);

    if (RREG32(smu->msg_reg_offset) & SMU_MSG_BUSY) {
        spin_unlock_irqrestore(&smu_msg_lock, flags);
        dev_err(adev->dev, "SMU busy\n");
        return -EBUSY;

    WREG32(smu->param_reg_offset, param);
    wmb();
    WREG32(smu->msg_reg_offset, msg);

    spin_unlock_irqrestore(&smu_msg_lock, flags);

    ret = readx_poll_timeout(RREG32, smu->resp_reg_offset,
                              resp_reg, resp_reg & SMU_RESP_MASK,
                              10, 10000);
    if (ret < 0) {

        return -ETIMEDOUT;
    }

    return 0;
}
```

- 超时时间设计依据：SMU处理典型PG/CG消息约需1-5ms，设置10ms（10000μs）为典型值的2倍，在保证不误判的同时避免等待过长时间；太短（<1ms）会导致正常消息被误判为超时，太长（>100ms）会严重影响用户体验

2. 自适应阈值更新函数：
```c
static void gfxoff_adaptive_update(struct adaptive_gfxoff_ctrl *ctrl)
{
    uint32_t i, bin;
    uint32_t histogram[GFXOFF_NUM_BINS] = {0};
    uint32_t total = min(ctrl->history_count, GFXOFF_HISTORY_SIZE);
    uint32_t p25, p50, p75;
    uint32_t false_wake_rate;
    uint32_t new_threshold;

    spin_lock(&ctrl->lock);

    /* 构建空闲间隔直方图 */
    for (i = 0; i < total; i++) {
        bin = ctrl->history[i] / (ctrl->max_threshold_ns / GFXOFF_NUM_BINS);
        if (bin >= GFXOFF_NUM_BINS)
            bin = GFXOFF_NUM_BINS - 1;
        histogram[bin]++;
    }

    /* 计算百分位数 */
    uint32_t cumulative = 0;
    for (i = 0; i < GFXOFF_NUM_BINS; i++) {
        cumulative += histogram[i];
        if (cumulative >= total * 25 / 100 && p25 == 0)
            p25 = i * (ctrl->max_threshold_ns / GFXOFF_NUM_BINS);
        if (cumulative >= total * 50 / 100 && p50 == 0)
            p50 = i * (ctrl->max_threshold_ns / GFXOFF_NUM_BINS);
        if (cumulative >= total * 75 / 100 && p75 == 0)
            p75 = i * (ctrl->max_threshold_ns / GFXOFF_NUM_BINS);
    }

    /* 计算误唤醒率 */
    false_wake_rate = ctrl->total_entry_count ?
        (ctrl->false_wake_count * 100 / ctrl->total_entry_count) : 0;

    /* 根据策略调整阈值 */
    switch (ctrl->policy) {
    case GFXOFF_POLICY_CONSERVATIVE:
        new_threshold = max(p75, ctrl->break_even_ns * 2);
        break;
    case GFXOFF_POLICY_BALANCED:
        new_threshold = max(p50, ctrl->break_even_ns);
        break;
    case GFXOFF_POLICY_AGGRESSIVE:
        new_threshold = max(p25 / 2, ctrl->break_even_ns / 2);
        break;
    }

    /* 误唤醒率过高时自动提升阈值 */
    if (false_wake_rate > 30) {
        new_threshold = min(new_threshold * 12 / 10, ctrl->max_threshold_ns);
        ctrl->consecutive_false_wake++;
    } else {
        ctrl->consecutive_false_wake = 0;
    }

    ctrl->idle_threshold_ns = clamp(new_threshold,
                                     ctrl->min_threshold_ns,
                                     ctrl->max_threshold_ns);

    spin_unlock(&ctrl->lock);
}
```

3. 进入决策函数：
```c
static bool gfxoff_should_enter(struct adaptive_gfxoff_ctrl *ctrl,
                                 uint32_t current_idle_ns)
{
    bool should_enter;
    uint32_t effective_threshold;

    spin_lock(&ctrl->lock);

    /* 将历史样本左移，记录当前空闲间隔 */
    ctrl->history[ctrl->history_head] = current_idle_ns;
    ctrl->history_head = (ctrl->history_head + 1) % GFXOFF_HISTORY_SIZE;
    if (ctrl->history_count < GFXOFF_HISTORY_SIZE)
        ctrl->history_count++;

    /* 应用策略偏移 */
    effective_threshold = ctrl->idle_threshold_ns;

    /* 如果连续误唤醒，额外增加磁滞 */
    if (ctrl->consecutive_false_wake > 3)
        effective_threshold = effective_threshold * 15 / 10;

    /* 决策：当前空闲时间是否超过有效阈值且超过盈亏平衡点 */
    should_enter = (current_idle_ns >= effective_threshold) &&
                   (current_idle_ns >= ctrl->break_even_ns);

    spin_unlock(&ctrl->lock);
    return should_enter;
}
```
