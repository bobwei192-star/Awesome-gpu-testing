#### 1.3 热插拔（HPD）处理流程

```
热插拔事件处理完整流程：
══════════════════════════════════════════════════════════════════════════════════════════════

  阶段 1: 硬件检测
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  硬件事件: 显示器插入/拔出 → HPD 引脚电平变化 (DP: IRQ_HPD 或 HPD)                │
  │  DCN: DMCUB 检测到 HPD 信号变化 → 通知驱动                                         │
  └────────────────────────────────────────────────────────────────────────────────────┘

  阶段 2: 中断处理
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  IRQ Handler → schedule_work(hpd_work) ─→ 工作队列处理                              │
  │                                                                                    │
  │  hpd_work_handler() {                                                              │
  │      // 1. 锁住所有连接器                                                           │
  │      mutex_lock(&aconnector->hpd_lock);                                            │
  │                                                                                    │
  │      // 2. 延迟 100ms (去抖动 Debounce)                                            │
  │      msleep(100);                                                                  │
  │                                                                                    │
  │      // 3. 读取 HPD 引脚状态                                                        │
  │      hpd_high = dc_link_detect_sink(link);                                         │
  │                                                                                    │
  │      // 4. 如果 HPD 为高: 检测显示器                                                │
  │      if (hpd_high) {                                                               │
  │          // 4a. 读取 EDID                                                          │
  │          edid = drm_get_edid(connector);                                           │
  │          if (!edid) {                                                              │
  │              // EDID 读取失败, 可能是显示器未准备就绪                                │
  │              schedule_delayed_work(&hpd_work, HZ/2);                               │
  │              return;                                                               │
  │          }                                                                         │
  │                                                                                    │
  │          // 4b. 解析 EDID, 更新连接器状态                                            │
  │          drm_connector_update_edid_property(connector, edid);                      │
  │          connector->display_info = drm_display_info_from_edid(edid);               │
  │                                                                                    │
  │          // 4c. 链路训练 (DP) 或 FRL 训练 (HDMI 2.1)                               │
  │          dp_lt_train(link);                                                        │
  │                                                                                    │
  │          // 4d. 设置连接器状态为 connected                                           │
  │          connector->status = connector_status_connected;                           │
  │                                                                                    │
  │          // 4e. 发送 uevent 通知用户空间                                              │
  │          drm_kms_helper_hotplug_event(dev);                                        │
  │                                                                                    │
  │      } else {                                                                      │
  │          // 5. 如果 HPD 为低: 显示器已断开                                           │
  │          connector->status = connector_status_disconnected;                        │
  │          drm_connector_update_edid_property(connector, NULL);                      │
  │          drm_kms_helper_hotplug_event(dev);                                        │
  │      }                                                                             │
  │  }                                                                                 │
  └────────────────────────────────────────────────────────────────────────────────────┘

  阶段 3: 用户空间处理
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  /lib/udev/rules.d/ 中的 udev 规则捕获 DRM uevent                                   │
  │  ├─ Xorg: 通过 xf86-input 监听 uevent, 重新配置 xrandr                              │
  │  ├─ Wayland: Weston/GNOME/KDE 合成器处理 uevent                                     │
  │  └─ modetest: 通过 drmModeGetConnector() 轮询连接器状态                              │
  │                                                                                    │
  │  uevent 内容:                                                                       │
  │  CHANGE@/devices/pci0000:00/0000:00:03.1/0000:03:00.0/drm/card0                     │
  │  ACTION=CHANGE                                                                      │
  │  DEVTYPE=drm_minor                                                                  │
  │  HOTPLUG=1                                                                          │
  │  CONNECTOR=25                                                                       │
  │  STATUS=connected                                                                   │
  └────────────────────────────────────────────────────────────────────────────────────┘
══════════════════════════════════════════════════════════════════════════════════════════════
```

#### 1.4 DCN 多管道资源分配

```c
// DCN 管道资源分配策略

struct pipe_ctx *dc_resource_acquire_pipe(struct dc *dc,
                                          struct dc_stream_state *stream)
{
    struct pipe_ctx *pipe = NULL;
    int i;

    // 1. 检查可用的物理管道
    for (i = 0; i < dc->res_pool->pipe_count; i++) {
        if (!dc->current_state->res_ctx.pipe_ctx[i].stream) {
            pipe = &dc->current_state->res_ctx.pipe_ctx[i];
            break;
        }
    }

    if (!pipe) {
        DC_ERR("No available pipes for new stream\n");
        return NULL;
    }

    // 2. 根据带宽需求选择管道
    // 高带宽需求 (如 4K@120Hz HDR) 使用索引较小的管道
    if (stream->timing.pix_clk_100hz > PIX_CLK_HIGH_THRESHOLD) {
        // 优先使用 Pipe 0 或 1 (有更好的调度优先级)
        for (i = 0; i < 2; i++) {
            if (!dc->current_state->res_ctx.pipe_ctx[i].stream) {
                pipe = &dc->current_state->res_ctx.pipe_ctx[i];
                break;
            }
        }
    }

    // 3. 分配 HUBP/DPP/MPC/OPP/OTG 资源
    pipe->plane = dc_plane_state_alloc(dc);
    pipe->stream = stream;
    pipe->plane_res.hubp = dc->res_pool->hubps[i];
    pipe->plane_res.dpp = dc->res_pool->dpps[i];
    pipe->stream_res.opp = dc->res_pool->opps[i];
    pipe->stream_res.tg = dc->res_pool->timing_generators[i];

    DC_LOG_DC("Pipe %d allocated for stream %p\n", i, stream);
    return pipe;
}

// 多显示器带宽预算计算
bool dc_validate_bandwidth(struct dc *dc, struct dc_state *context)
{
    int total_bw = 0;
    int i;

    // 计算所有已启用管道的带宽需求
    for (i = 0; i < MAX_PIPES; i++) {
        struct pipe_ctx *pipe = &context->res_ctx.pipe_ctx[i];
        if (pipe->stream && pipe->plane) {
            total_bw += pipe->stream->timing.pix_clk_100hz * 100;

            // DCC 压缩使带宽需求降低
            if (pipe->plane->dcc.enable)
                total_bw /= 2;

            // HDR 10bit 增加带宽
            if (pipe->plane->format == SURFACE_PIXEL_FORMAT_GRPH_ARGB2101010)
                total_bw = total_bw * 10 / 8;
        }
    }

    // 检查是否超过总可用带宽
    if (total_bw > dc->caps.max_total_bandwidth) {
        DC_LOG_WARNING("Bandwidth budget exceeded: %d > %d\n",
                       total_bw, dc->caps.max_total_bandwidth);
        return false;
    }

    return true;
}
```

### 3. 案例分析

#### 3.1 Bug 分析: 三显示器配置中断分配失败

**Bug 信息**：
- 来源：GitLab issue #22145
- 标题：Third display shows no signal on RX 7900 XTX DCN 3.1
- 链接：https://gitlab.freedesktop.org/drm/amd/-/issues/22145

**问题描述**：
RX 7900 XTX 在连接三个 4K@144Hz 显示器时，第三个显示器无信号。前两个正常工作。

**根因**：
即使 DCN 3.1 有 4 个管道，但在 4K@144Hz HDR 配置下，每个管道需要大量带宽。DCN 的内部数据总线（DCN Data Fabric）带宽达到饱和。

```
带宽计算:
══════════════════════════════════════════════════════════════════════════════════════════════

  每个 4K@144Hz 10bit HDR 流:
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  像素时钟: 1440 MHz (4K@144Hz CVT-RB)                                              │
  │  每像素: 32bit (ARGB2101010)                                                       │
  │  每管道带宽: 1440 MHz × 32bit = 46.08 Gbps                                        │
  │                                                                                    │
  │  3 管道合计: 138.24 Gbps                                                           │
  │  DCN Data Fabric 最大带宽: 128 Gbps (DCN 3.1)                                     │
  │  → 超限! 第 3 个显示器无法获得足够带宽                                              │
  └────────────────────────────────────────────────────────────────────────────────────┘

  解决方案:
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  1. 降低一个显示器的刷新率 (如 144Hz → 60Hz)                                       │
  │  2. 使用 DSC 压缩 (3:1 压缩比 → 带宽降低到 15.36 Gbps/管道)                        │
  │  3. 降低色度采样 (RGB → YCbCr 4:2:0 → 带宽减半)                                   │
  └────────────────────────────────────────────────────────────────────────────────────┘
```

#### 3.2 Bug 分析: 热插拔导致显示配置丢失

**Bug 信息**：
- 来源：GitLab issue #17893
- 标题：Display configuration lost after HPD toggle on DCN 3.0
- 链接：https://gitlab.freedesktop.org/drm/amd/-/issues/17893

**问题描述**：
当显示器短暂断开后重新连接（如切换显示器输入源），桌面环境丢失了该显示器的配置，需要用户手动重新配置。

**根因**：
HPD 去抖动延时不足。当显示器切换输入源时产生短暂的 HPD 低电平，驱动立即检测到断开事件并清除了显示配置。但显示器在 200ms 后重新 HPD 高电平，驱动重新检测为新的显示器，没有复用先前的配置。


## 今日小结

今日我们学习了多显示器配置与管理的核心技术：

1. **配置模式**：扩展桌面（每个 CRTC 独立输出）、镜像模式（同一帧缓冲多 OTG 输出）、混合模式。

2. **资源分配**：DCN 使用管道（Pipe）抽象管理硬件资源，每个显示器消耗一个 CRTC 和一组管道资源。4 管道 DCN 最大支持 4 个显示器。

3. **热插拔处理**：三阶段流程（硬件检测 → 中断处理/EDID读取 → 用户空间 uevent）。去抖动时间是关键参数。

4. **带宽预算**：多显示器配置需要总带宽不超过 DCN Data Fabric 的限制。DSC 压缩和色度采样降低可以缓解带宽压力。

5. **EDID 管理**：EDID 告知驱动显示器的能力。EDID 覆盖机制支持调试和测试场景。
