## 第一部分：概念辨析（25分）

### 1.1 简答题（每题3分，共15分）

**题目1：AMD GPU 代号演变**
请按照时间顺序列出从 GCN 1.0 到 RDNA 4 的主要架构代号及其对应的核心架构特性变化。

**参考答案要点**：
- GCN 1.0 (Southern Islands, 2012)：统一着色器架构，VLIW4->SIMD16
- GCN 2.0 (Sea Islands, 2013)：改进的几何引擎，TrueAudio
- GCN 3.0 (Volcanic Islands, 2014)：改进的DMA引擎，完整64位虚拟地址支持
- GCN 4.0 (Polaris, 2016)：14nm FinFET工艺，改进的电源管理
- GCN 5.0 (Vega, 2017)：HBM2支持，Primitive Shaders，Raven Ridge APU中的首代
- RDNA 1 (Navi 10, 2019)：全新架构设计，SIMD->SIMD32，Wave32引入
- RDNA 2 (Navi 21, 2020)：硬件光线追踪，Infinity Cache，VSR
- RDNA 3 (Navi 31, 2022)：Chiplet设计，双计算单元（WGP），DPP指令集增强
- RDNA 4 (Navi 4x, 预计2024+)：进一步优化的光线追踪，改进的AI加速器

**题目2：DRM 子系统的核心组件**
请简述 DRM 子系统中 GEM、KMS、DMA-BUF、PRIME 四个核心组件的功能。

**参考答案要点**：
- GEM（Graphics Execution Manager）：内存管理对象抽象，管理 Buffer Object 的分配、释放和共享
- KMS（Kernel Mode Setting）：显示模式设置，负责分辨率、刷新率、平面配置等显示输出控制
- DMA-BUF：跨设备缓冲区共享机制，允许不同驱动/设备间共享 DMA 缓冲区
- PRIME：基于 DMA-BUF 的 DRM 缓冲区共享扩展，支持 GPU 间和 GPU-Display Controller 间共享

**题目3：AMDGPU 内存管理层次**
请说明 VRAM、GTT、Doorbell 三种内存类型在 AMDGPU 驱动中的用途和差异。

**参考答案要点**：
- VRAM：显存（GPU本地内存），高带宽，用于存储帧缓冲、纹理、顶点数据等 GPU 频繁访问的数据
- GTT（Graphics Translation Table）：系统内存中映射给 GPU 使用的部分，通过页表访问，速度低于 VRAM，可用于存储不常访问的数据
- Doorbell：用于用户空间向 GPU 发送命令通知的特殊内存区域，通过 PCIe BAR 映射，实现低延迟命令提交

**题目4：GPU 调度器架构**
请说明 DRM Scheduler 中 Job、Fence、Scheduler Entity 的关系。

**参考答案要点**：
- Job：封装一次 GPU 命令提交的工作单元，包含命令缓冲区地址、依赖关系和优先级信息
- Fence（dma_fence）：同步原语，用于追踪 GPU 命令执行完成状态，支持跨引擎和跨进程同步
- Scheduler Entity：调度实体，将 Job 映射到硬件队列，负责排队、优先级排序和依赖解析
- 关系：Job 提交到 Scheduler Entity 排队，Entity 按依赖关系和优先级将 Job 推送到硬件 Ring，Fence 标记 Job 完成状态

**题目5：Display Core Next (DCN) 显示管线组件**
请列出 DCN 显示管线中的 OTG、DPP、MPC、OPP 四个组件及其功能。

**参考答案要点**：
- OTG（Output Timing Generator）：生成显示时序信号，控制像素时钟、同步信号和空白期间
- DPP（Display Pipe and Plane）：处理显示平面的像素数据，包含缩放、色彩转换和格式转换
- MPC（Multiple Pipe Combined）：多管线的合成，组合多个 DPP 的输出
- OPP（Output Plane Processor）：输出前的最终处理，包括 dithering、gamma 校正和像素格式打包

### 1.2 判断题（每题1分，共10分）

判断以下陈述是否正确，并说明理由。

1. AMDGPU 驱动中，amdgpu_device 结构体是驱动的核心数据结构，包含芯片特性、IP block 列表和调度器引用。**(正确)**
2. PSP（Platform Security Processor）仅负责固件加载，不参与 GPU 电源管理。**(错误——PSP 还参与安全启动、密钥管理和电源状态转换认证)**
3. TTM（Translation Table Manager）子系统仅用于 VRAM 管理，不涉 GTT。**(错误——TTM 管理包括 VRAM、GTT 在内的所有 GPU 内存类型)**
4. MES（Micro Engine Scheduler）固件运行在 GPU 的调度器微引擎上，负责用户模式队列的管理。**(正确)**
5. GFXCLK 和 SOCCLK 属于独立的时钟域，调整其中一个不会影响另一个。**(部分正确——它们虽然是独立时钟域，但在某些电源状态切换时需要协同调整)**
6. AMDGPU 的 SR-IOV 实现中，VF（Virtual Function）可以完全控制 GPU 硬件寄存器。**(错误——VF 只能访问模拟的寄存器空间，实际的硬件控制由 PF 管理)**
7. DMUB（Display Manager Microcontroller Unit）固件负责处理热插拔检测和显示链路训练。**(正确)**
8. AMDGPU 支持的 BACO（Bus Active Card Off）状态允许 GPU 在 PCIe 总线活跃时关闭核心供电。**(正确)**
9. amdgpu_gem.c 实现了 DRM GEM 接口，包括 gem_create_object 和 gem_prime_import 等回调。**(正确)**
10. KFD（Kernel Fusion Driver）与 AMDGPU 共享内存管理基础设施，通过 amdgpu_amdkfd_gpuvm.c 集成。**(正确)**

## 第四部分：综合设计（25分）

### 4.1 场景设计题（15分）

**场景**：你收到报告称，在某款 RDNA 3 GPU（Navi 31）上运行 Vulkan 计算应用时，长时间运行后出现 GPU hang。hang 前内核日志中出现 "amdgpu: ring gfx timeout" 消息，但在此之前没有任何错误征兆。

**问题**：请设计排查方案，涵盖以下方面：

1. 如何确认这次 GPU hang 的确切原因？（列出需要检查的日志和工具）
2. 如何利用 AMDGPU 的恢复机制进行测试？
3. 如何编写 IGT 测试来复现该问题？

**参考答案要点**：

1. **排查方案**：
   - 检查完整 dmesg 日志，重点关注 ring timeout 前后的消息
   - 检查 `amdgpu_gfxoff` 状态：`cat /sys/kernel/debug/dri/0/amdgpu_gfxoff`
   - 使用 umr 读取 GPU 寄存器状态：`umr -r -O WAIT_ON_ADDRESS -O SHADER -R` 检查 GPU 是否真正挂起
   - 检查 GPU 温度：`cat /sys/class/drm/card0/device/hwmon/hwmon*/temp1_input`
   - 检查电源状态：`cat /sys/class/drm/card0/device/pp_dpm_sclk`
   - 使用 ftrace 追踪 fence 信号：`trace_cmd record -e amdgpu:* -e dma_fence:*`
   - 检查是否存在内存压力导致 BO 驱替异常

2. **使用恢复机制测试**：
   - 触发手动 recovery：`echo 1 > /sys/class/drm/card0/device/amdgpu_job_timeout`
   - 测试 soft reset：检查 `amdgpu_gpu_recover()` 函数路径
   - 测试 mode1 reset：通过 BACO 入口测试完整重置
   - 验证恢复后的 GPU 功能完整性：运行基本测试 gem_basic、amd_cs_nop

3. **IGT 测试复现**：
   - 参考 amdgpu/amd_cs_nop.c 编写计算负载测试
   - 添加长时间循环执行计算 shader 的测试
   - 在每次迭代中检查 fence 超时
   - 监控 GPU 温度和频率变化
   - 在测试中同时注入内存压力（mempressure 工具）
