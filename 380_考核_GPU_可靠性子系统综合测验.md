## 第二部分：判断题（每题 2 分，共 10 分）

**11.** RAS 子系统中的 CE（Correctable Error）不需要任何处理，因为硬件已自动纠正。

**12.** umr 工具可以通过 --perf 参数读取 GPU 性能计数器，但不能用于 RAS 错误检测。

**13.** GPU 恢复流程中，软重置（soft reset）失败后会自动尝试硬重置（hard reset）。

**14.** 在 ECC 错误注入测试中，注入 CE 错误后必须通过实际内存访问（如 GPU 拷贝操作）来触发检测。

**15.** dmesg 日志中，amdgpu 相关的 RAS 错误日志可以通过 `dmesg | grep -i "ras\|ecc\|amdgpu.*err"` 进行过滤筛选。

### 选择题

| 题号 | 答案 | 解析 |
|------|------|------|
| 1 | **B** | UE 错误表示数据已损坏，必须触发 GPU Recovery 来恢复运行状态 |
| 2 | **B** | 检测 → 分类（CE/UE/Fatal）→ 决策（软/硬重置）→ 重置 → 后恢复验证 |
| 3 | **B** | SECDED 对 128B 数据需 8B ECC，效率为 128/(128+8) ≈ 94.1% |
| 4 | **A** | INJECT_ADDR 设地址、INJECT_DATA 设数据、INJECT_CTRL 设错误类型和触发 |
| 5 | **B** | 0xA0000002 = bit1 CP_BUSY + bit29 NOT_GUI_IDLE → GUI Active + CP Busy |
| 6 | **C** | gpu_freq 是频率切换事件，不属于恢复日志分析范畴 |
| 7 | **B** | CE 超过阈值后将引发 UE 级别的系统响应，触发页面隔离或恢复 |
| 8 | **C** | 典型值：crit = 105°C，emergency = 110°C |
| 9 | **B** | 温度升高 → 触发降频 → 降频不足导致 hang，这是典型的异常链 |
| 10 | **B** | ras_features 用于查询和设置 RAS 功能启用状态 |

### 简答题要点

**16. 严重性级别：**
- INFO：正常事件记录，仅日志
- WARN：CE 超阈值或轻微异常，触发告警
- ERROR：UE 或 recoverable 错误，触发软恢复
- CRIT：多次恢复失败或硬件故障，触发硬恢复
- PANIC：不可恢复错误，触发系统级响应

**17. ECC 注入流程：**
1. 选择方法（debugfs/硬件寄存器/软件模拟）
2. 配置 INJECT_ADDR（目标地址）
3. 配置 INJECT_DATA（注入数据模式）
4. 配置 INJECT_CTRL（CE: ERR_TYPE=0, UE: ERR_TYPE=1）
5. 触发内存访问（GPU copy 或 compute）
6. 验证：CE → ras_ecc_cnt CE 计数 +1；UE → ras_ecc_cnt UE 计数 +1 且触发恢复

**18. 日志分析：**
- 异常类型：UMC UE（不可纠正 ECC 错误）
- 关键信息：地址 0xffff0000，syndrome 0xabcd，channel 3
- 恢复流程：检测 UE → 启动恢复 → 软重置 → 1.002s 成功完成
- 评估：恢复速度快（~1s），单次成功，无需硬重置

### 实践题要点

**19. RAS 健康检查脚本：**
- CE == 0 且 UE == 0：正常（绿色）
- CE > 0 且 UE == 0：警告（黄色），需关注趋势
- UE > 0：严重（红色），需立即处理
- recent_bad_pages 非空：严重，已存在坏页
- ras_features 未启用对应功能：警告

**20. 诊断方案：**
1. 收集：dmesg 日志、ras_ecc_cnt 计数、温度历史、GRBM_STATUS 寄存器值、应用负载特征
2. 排除流程：检查温度 → 检查 ECC → 检查 GRBM_STATUS → 分析 hang 前操作 → 对比已知 bug
3. 区分：ECC 导致 → ras_ecc_cnt 有计数；温度导致 → 温度日志超阈值；驱动 bug → 特定负载复现
4. 验证：修复后 72h 压力测试 + 回归测试 + 对比修复前后 hang 频率
