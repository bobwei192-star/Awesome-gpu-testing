## 知识详解

### 1. 等价类划分基本原理

#### 1.1 概念与数学基础

等价类划分是一种黑盒测试设计方法，其核心思想是：将输入域划分为若干不相交的子集（等价类），使得同一等价类中的任意输入值在测试中具有"等价"的行为——即如果同一个等价类中的一个测试用例发现了缺陷，那么该等价类中的其他输入值也应当能发现相同的缺陷；反之，如果某个测试用例通过了，则该等价类中的其他值也应当通过。

数学上，等价类划分基于等价关系（Equivalence Relation）的定义。设输入域为 $D$，$R$ 是 $D$ 上的一个等价关系，满足：

- **自反性（Reflexivity）**：$\forall x \in D,\ x R x$
- **对称性（Symmetry）**：$\forall x,y \in D,\ x R y \implies y R x$
- **传递性（Transitivity）**：$\forall x,y,z \in D,\ (x R y \land y R z) \implies x R z$

等价关系 $R$ 将输入域 $D$ 划分为若干等价类 $[x]_R = \{y \in D \mid x R y\}$，这些等价类构成 $D$ 的一个划分（Partition）。

在软件测试中，等价关系通常由被测程序的行为定义：若两个输入导致程序执行相同路径、产生相同结果、或触发相同处理逻辑，则它们属于同一等价类。

#### 1.2 两类等价类

| 类型 | 定义 | GPU驱动测试示例 |
|------|------|----------------|
| **有效等价类**（Valid Equivalent Class） | 满足输入规范要求的合法输入集合 | 有效的BO大小：1字节到VRAM容量上限之间的值 |
| **无效等价类**（Invalid Equivalent Class） | 不满足输入规范要求的非法输入集合 | 负数的BO大小、超过VRAM容量的BO大小、0大小的BO |

研究表明，无效等价类测试通常比有效等价类测试发现更多缺陷，因为开发者比较容易考虑正常路径（Happy Path），而容易忽略异常路径（Error Path）的处理。

#### 1.3 划分原则

在GPU驱动测试中进行等价类划分时，应遵循以下原则：

1. **完备性（Completeness）**：所有可能的输入值必须属于某个等价类，不能有遗漏
2. **互斥性（Mutual Exclusivity）**：一个输入值只能属于一个等价类，不能同时属于多个等价类
3. **单一覆盖（Single Coverage）**：每个等价类至少选取一个代表性值作为测试用例

```
输入域D的等价类划分示意：

    ┌─────────────────────────────────────────────────┐
    │                    输入域 D                      │
    │                                                  │
    │   ┌─────────────┐  ┌─────────────┐  ┌─────────┐ │
    │   │  等价类 EC1  │  │  等价类 EC2  │  │  等价类  │ │
    │   │ （有效）      │  │ （有效）      │  │  EC3    │ │

    └─────────────────────────────────────────────────┘
```

### 3. 等价类划分在GPU驱动中的高级应用

#### 3.1 复合等价类（Composite Equivalence Classes）

在GPU驱动测试中，单一输入参数的等价类往往不足以覆盖完整的测试场景。需要组合多个参数的等价类，形成复合等价类。

**示例：命令提交（Command Submission）的复合等价类**

命令提交涉及多个参数的交互：engine类型、number of IBs、fence handling、chained IB等。

```c
struct drm_amdgpu_cs_in {
    __u32 ctx_id;           // 上下文ID
    __u32 ip_instance;      // IP实例
    __u32 ring;             // Ring索引
    uint32_t num_ibs;       // IB数量
    struct drm_amdgpu_cs_ib ibs[]; // IB数组
};
```

```
复合等价类划分示例：

    num_ibs=0 ────► 无效等价类：无IB提交（预期-EINVAL）
           │
           ▼
    ┌─────────────┐
    │  num_ibs=1   │────► IB内容有效 ──► GFX engine ──► 复合等价类C1
    │  (有效)      │      IB内容有效 ──► COMPUTE     ──► 复合等价类C2
    └─────────────┘      IB内容有效 ──► SDMA        ──► 复合等价类C3
           │             IB内容为空 ──► 任何engine   ──► 复合等价类C4（无效）
           ▼
    ┌─────────────┐
    │  num_ibs=16  │────► 16个有效IB ──► GFX         ──► 复合等价类C5
    │  (边界)      │      16个有效IB ──► COMPUTE     ──► 复合等价类C6
    └─────────────┘
           │
           ▼
    ┌─────────────┐
    │  num_ibs=17  │────► 超过MAX_IBS ──────────────► 复合等价类C7（无效）
    └─────────────┘
```

#### 3.2 基于等价类的覆盖率指标

在AMDGPU驱动的测试中，可以定义等价类覆盖率（Equivalence Class Coverage）作为测试充分性度量：

```
等价类覆盖率 = (已测试的等价类数量 / 总等价类数量) × 100%

度量示例：

    参数            总等价类  已测试 覆盖率  未覆盖的等价类
    ──────────────  ────────  ──────  ──────  ───────────────
    bo_size         6         4       66.7%   SIZE_HUGE, SIZE_OVERFLOW
    alignment       6         3       50.0%   ALIGN_2M, ALIGN_1G, ALIGN_TOO_LARGE
    domain_flags    8         5       62.5%   DOMAIN_RESERVED, DOMAIN_NONE, DOMAIN_ALL
    flags           10        7       70.0%   CPU_ACCESS_REQUIRED, GTT_WC, VM_ALWAYS_VALID
    ──────────────  ────────  ──────  ──────  ───────────────
    总计            30        19      63.3%

    结论：等价类覆盖率为63.3%，未覆盖的等价类涉及：
    - 超大BO分配（可能触发OOM处理路径）
    - 大页对齐（2MB/1GB路径未测试）
    - 保留位标记（安全漏洞风险）
    - CPU_ACCESS_REQUIRED与GTT_WC组合场景
    - VM_ALWAYS_VALID标志行为
```

#### 3.3 等价类划分与边界值分析的关系

等价类划分和边界值分析（Boundary Value Analysis, BVA）是紧密相关的两种方法。BVA可以看作等价类划分的补充——它关注等价类边界上的值。

```
等价类划分 + 边界值分析的关系：

    等价类EC1: 1 ≤ x ≤ 100（有效）
    等价类EC2: x ≥ 101（有效）
    等价类EC3: x ≤ 0（无效）

    仅等价类划分的测试用例：
        EC1: x = 50
        EC2: x = 200
        EC3: x = -1

    加入边界值分析的测试用例：
        EC1的边界: x = 1, x = 100
        EC1-EC2边界: x = 100, x = 101
        EC3-EC1边界: x = 0, x = 1
        次边界: x = -1, x = 2, x = 99, x = 102

    ┌──────┐┌──────────────────────┐┌──────────────────┐
    │ EC3  ││        EC1           ││       EC2        │
    │无效  ││       有效           ││      有效        │
    │      ││                      ││                  │
    ──┴──────┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴────
    -1  0  1  2  ...    50    ...  100 101 102 ... 200

    等价类测试点: ▼
    边界测试点:   ●
    ──────────────────────────────────────────
    ▼ ▼  ● ●                   ●  ● ●
    │ │  │ │                   │  │ │
    x=-1 x=0 x=1 x=2     x=100 x=101 x=102
```

**在GPU驱动中，边界值分析特别重要**，因为：
- 驱动代码中大量使用 `if (x < threshold)` 形式的边界检查
- 硬件寄存器通常有严格的位宽限制（32-bit/64-bit）
- 内存分配的对齐要求（PAGE_SIZE=4096的倍数）

### 4. AMDGPU驱动中的等价类划分案例

#### 4.1 案例1：`amdgpu_cs_ioctl` 的等价类测试

`amdgpu_cs_ioctl` 是AMDGPU驱动的核心IOCTL，处理命令提交。以下是通过等价类划分设计的测试用例。

```c
/*
 * amdgpu_cs_ioctl 的等价类测试框架
 *
 * 参考来源: drivers/gpu/drm/amd/amdgpu/amdgpu_cs.c
 * 函数签名: static int amdgpu_cs_ioctl(struct drm_device *dev,
 *                                      void *data, struct drm_file *filp)
 */

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <sys/ioctl.h>
#include <fcntl.h>
#include <errno.h>
#include <stdint.h>
#include <time.h>

/* AMDGPU IOCTL 定义 (简化) */
#define DRM_IOCTL_AMDGPU_CS \
    DRM_IOWR(DRM_COMMAND_BASE + 0x09, struct drm_amdgpu_cs_in)

struct drm_amdgpu_cs_chunk {
    uint32_t chunk_id;
    uint32_t length_dw;
    uint64_t chunk_data;
};

struct drm_amdgpu_cs_in {
    uint32_t ctx_id;
    uint32_t bo_list_handle;
    uint32_t num_chunks;
    uint32_t _pad;
    uint64_t chunks;
};

/* 等价类ID枚举 */
enum ec_id {
    EC_CS_VALID_SINGLE_IB,       /* 有效：单个IB提交 */
    EC_CS_VALID_MULTI_IB,        /* 有效：多个IB提交 */
    EC_CS_VALID_CHAINED_IB,      /* 有效：Chained IB */
    EC_CS_INVALID_ZERO_CHUNKS,   /* 无效：0个chunk */
    EC_CS_INVALID_NULL_CHUNKS,   /* 无效：NULL chunk指针 */
    EC_CS_INVALID_BAD_CTX,       /* 无效：不存在上下文 */
    EC_CS_INVALID_TOO_MANY_IB,   /* 无效：超过最大IB数量 */
    EC_CS_INVALID_BAD_BO_LIST,   /* 无效：非法BO列表句柄 */
    EC_CS_BOUNDARY_SINGLE_CHUNK, /* 边界：只有一个chunk */
    EC_CS_BOUNDARY_MAX_CHUNKS,   /* 边界：恰好最大chunks */
};

/* 等价类信息结构 */
struct ec_info {
    enum ec_id id;
    const char *name;
    int expected_errno;
    int valid;
    const char *description;
};

struct ec_info equivalence_classes[] = {
    {EC_CS_VALID_SINGLE_IB, "CS_VALID_SINGLE_IB", 0, 1,
     "有效：提交单个IB到GFX引擎"},
    {EC_CS_VALID_MULTI_IB, "CS_VALID_MULTI_IB", 0, 1,
     "有效：提交多个IB（不同引擎）"},
    {EC_CS_VALID_CHAINED_IB, "CS_VALID_CHAINED_IB", 0, 1,
     "有效：Chained IB（预取）"},
    {EC_CS_INVALID_ZERO_CHUNKS, "CS_INVALID_ZERO_CHUNKS", EINVAL, 0,
     "无效：num_chunks=0，无有效提交内容"},
    {EC_CS_INVALID_NULL_CHUNKS, "CS_INVALID_NULL_CHUNKS", EFAULT, 0,
     "无效：chunks指针为NULL"},
    {EC_CS_INVALID_BAD_CTX, "CS_INVALID_BAD_CTX", ENOENT, 0,
     "无效：使用不存在的上下文ID"},
    {EC_CS_INVALID_TOO_MANY_IB, "CS_INVALID_TOO_MANY_IB", E2BIG, 0,
     "无效：IB数量超过硬件限制"},
    {EC_CS_INVALID_BAD_BO_LIST, "CS_INVALID_BAD_BO_LIST", ENOENT, 0,
     "无效：非法BO列表句柄"},
    {EC_CS_BOUNDARY_SINGLE_CHUNK, "CS_BOUNDARY_SINGLE_CHUNK", 0, 1,
     "边界：恰好1个chunk"},
    {EC_CS_BOUNDARY_MAX_CHUNKS, "CS_BOUNDARY_MAX_CHUNKS", 0, 1,
     "边界：恰好CS_MAX_CHUNKS个chunk"},
};

/* 打印等价类覆盖矩阵 */
void print_coverage_matrix(struct ec_info *classes, int num_classes,
                           int *tested_mask)
{
    int total = num_classes;
    int tested = 0;

    printf("\n等价类覆盖矩阵:\n");
    printf("┌─────────────────────────────────────────────────────────────────┐\n");
    printf("│ %-4s │ %-30s │ %-8s │ %-5s │ %-12s │\n",
           "ID", "等价类名称", "类型", "状态", "预期错误");
    printf("├──────┼────────────────────────────────┼──────────┼───────┼──────────────┤\n");

    for (int i = 0; i < num_classes; i++) {
        const char *type = classes[i].valid ? "有效" : "无效";
        const char *status = tested_mask[i] ? "已测试" : "未测试";
        const char *err = classes[i].expected_errno == 0 ? "成功" :
                          classes[i].expected_errno == EINVAL ? "EINVAL" :
                          classes[i].expected_errno == EFAULT ? "EFAULT" :
                          classes[i].expected_errno == ENOENT ? "ENOENT" :
                          classes[i].expected_errno == E2BIG ? "E2BIG" : "其他";
        printf("│ %-4s │ %-30s │ %-8s │ %-5s │ %-12s │\n",
               classes[i].name, type, status, err);
        if (tested_mask[i]) tested++;
    }
    printf("└──────┴────────────────────────────────┴──────────┴───────┴──────────────┘\n");
    printf("覆盖率: %d/%d = %.1f%%\n", tested, total,
           total > 0 ? (double)tested / total * 100.0 : 0.0);
}

/* 测试函数: 单个有效IB提交 */
static int test_ec_valid_single_ib(int fd)
{
    printf("测试 EC_CS_VALID_SINGLE_IB...\n");
    struct drm_amdgpu_cs_chunk chunk;
    uint32_t ib_data[16] = {0};
    ib_data[0] = 0xC0001000; /* PKT3_NOP */
    ib_data[1] = 0x00000000;

    chunk.chunk_id = 0x01;
    chunk.length_dw = 16;
    chunk.chunk_data = (uint64_t)(uintptr_t)ib_data;

    struct drm_amdgpu_cs_in cs_in;
    memset(&cs_in, 0, sizeof(cs_in));
    cs_in.ctx_id = 1;
    cs_in.num_chunks = 1;
    cs_in.chunks = (uint64_t)(uintptr_t)&chunk;

    int ret = ioctl(fd, DRM_IOCTL_AMDGPU_CS, &cs_in);
    printf("  ret=%d\n", ret);
    return (ret == 0) ? 0 : -1;
}

/* 测试函数: 无效的0个chunk提交 */
static int test_ec_invalid_zero_chunks(int fd)
{
    printf("测试 EC_CS_INVALID_ZERO_CHUNKS...\n");
    struct drm_amdgpu_cs_in cs_in;
    memset(&cs_in, 0, sizeof(cs_in));
    cs_in.ctx_id = 1;
    cs_in.num_chunks = 0;
    cs_in.chunks = (uint64_t)(uintptr_t)NULL;

    int ret = ioctl(fd, DRM_IOCTL_AMDGPU_CS, &cs_in);
    printf("  ret=%d, errno=%d (expected EINVAL=%d)\n",
           ret, errno, EINVAL);
    return (ret != 0 && errno == EINVAL) ? 0 : -1;
}

/* 测试函数: 无效上下文ID */
static int test_ec_invalid_bad_ctx(int fd)
{
    printf("测试 EC_CS_INVALID_BAD_CTX...\n");
    struct drm_amdgpu_cs_chunk chunk;
    uint32_t ib_data[16] = {0};
    chunk.chunk_id = 0x01;
    chunk.length_dw = 16;
    chunk.chunk_data = (uint64_t)(uintptr_t)ib_data;
```

### 7. 等价类划分的高级话题

#### 7.1 基于等价类的测试充分性准则

在GPU驱动测试中，等价类覆盖准则可以定义多个级别：

```
级别1: 等价类覆盖（最低要求）
    定义: 每个等价类至少被一个测试用例覆盖
    公式: C_EC = |覆盖的等价类| / |总的等价类|

级别2: 稳健等价类覆盖
    定义: 每个等价类至少被k个测试用例覆盖（k≥2）

级别3: 有效+无效等价类覆盖
    定义: 有效等价类和无效等价类分别覆盖
    公式: C_VEC = |覆盖的有效等价类| / |总有效等价类|
          C_IEC = |覆盖的无效等价类| / |总无效等价类|

级别4: 多维度等价类组合覆盖
    定义: 不同维度的等价类进行Pair-wise组合覆盖

级别5: 等价类-边界联合覆盖（最高要求）
    定义: 覆盖所有等价类及其边界值
```

```c
struct coverage_stats {
    int total_ec;
    int covered_ec;
    int valid_ec;
    int covered_valid;
    int invalid_ec;
    int covered_invalid;
    int boundary_ec;
    int covered_boundary;

void calculate_coverage_levels(struct coverage_stats *stats)
{
    printf("等价类覆盖等级分析:\n\n");

           l1, stats->covered_ec, stats->total_ec);

           l1 >= 80.0 ? "✓ 通过" : "✗ 未通过");

    double l2i = (double)stats->covered_invalid / stats->invalid_ec * 100.0;
    printf("\n级别2 (有效+无效覆盖):\n");
    printf("  有效等价类: %.1f%% (%d/%d) 要求≥90%% %s\n",
           l2v, stats->covered_valid, stats->valid_ec,
           l2v >= 90.0 ? "✓" : "✗");
    printf("  无效等价类: %.1f%% (%d/%d) 要求≥80%% %s\n",
           l2i, stats->covered_invalid, stats->invalid_ec,

}
```

#### 7.2 等价类划分的局限性

尽管等价类划分是一种有效的测试设计方法，但在GPU驱动测试中存在以下局限性：

| 局限性 | 说明 | GPU驱动示例 | 缓解策略 |
|--------|------|-------------|---------|
| **输入关联性忽略** | 假设输入参数独立，但GPU驱动中参数往往相互影响 | bo_size和domain_flags的组合会影响分配策略 | 使用Pair-wise组合测试补充 |
| **内部状态忽略** | 不考虑被测系统的内部状态序列 | GPU的电源状态(D0/D3Hot/D3Cold)会影响BO分配行为 | 结合状态转换测试 |
| **时序相关的缺陷** | 无法捕获并发时序问题 | 两个线程同时提交CS请求的竞争条件 | 需要并发测试补充 |
| **非功能性缺陷** | 无法检测性能/功耗/温度问题 | 特定参数组合导致GPU频率不升 | 需要性能测试补充 |
| **硬件差异** | 不同硬件的行为不一致 | RDNA2和RDNA3对同参数的处理不同 | 需要硬件兼容性矩阵测试 |
| **等价类划分不准确** | 等价关系定义错误会导致测试盲区 | 认为bo_size>0的所有值等价，但2MB对齐和4KB对齐行为不同 | 基于代码覆盖率反馈调整等价类 |

#### 7.4 基于等价类的GPU参数fuzzing测试

等价类划分为GPU驱动的fuzzing测试提供了理论基础。通过识别输入参数的有效和无效等价类，可以指导fuzzer生成更有针对性的测试数据。

```c
/*
 * 基于等价类的GPU驱动fuzzing框架
 *
 * 该框架利用等价类划分指导fuzzing过程:
 * - 对每个参数维护有效/无效等价类列表
 * - fuzzer在等价类内部进行随机扰动
 * - 确保每个等价类至少被覆盖一次
 */

#define MAX_EC_CLASSES 64
#define MAX_EC_VALUES  256

struct ec_fuzz_spec {
    const char *param_name;
    int num_valid_classes;
    int num_invalid_classes;
    /* 每个等价类的样本值 */
    uint64_t valid_samples[MAX_EC_CLASSES][MAX_EC_VALUES];
    int valid_counts[MAX_EC_CLASSES];
    uint64_t invalid_samples[MAX_EC_CLASSES][MAX_EC_VALUES];
    int invalid_counts[MAX_EC_CLASSES];
    /* 覆盖追踪 */
    int valid_covered[MAX_EC_CLASSES];
    int invalid_covered[MAX_EC_CLASSES];
};

struct ec_fuzz_spec bo_create_fuzz_spec = {
    .param_name = "bo_size",
    .num_valid_classes = 3,
    .num_invalid_classes = 2,
    .valid_samples = {
        {1, 64, 512, 1024, 4095},                    /* SIZE_MIN: 1..PAGE_SIZE-1 */
        {4096, 65536, 1048576, 16777216, 268435456}, /* SIZE_NORMAL */
        {1073741824ULL, 2147483648ULL, 4294967296ULL} /* SIZE_MAX附近 */
    },
    .valid_counts = {5, 5, 3},
    .invalid_samples = {
        {0},                                          /* SIZE_INVALID_0 */
        {8589934592ULL, 17179869184ULL, 68719476736ULL} /* SIZE_OVERFLOW */
    },
    .invalid_counts = {1, 3}
};

/* 基于等价类的fuzzing执行 */
void ec_guided_fuzz(int fd, struct ec_fuzz_spec *spec)
{
    printf("=== 基于等价类划分的Fuzzing ===\n");
    printf("参数: %s\n", spec->param_name);
    
    /* Phase 1: 覆盖所有有效等价类 */
    printf("\nPhase 1: 覆盖有效等价类\n");
    for (int ec = 0; ec < spec->num_valid_classes; ec++) {
        for (int i = 0; i < spec->valid_counts[ec]; i++) {
            uint64_t val = spec->valid_samples[ec][i];
            int ret = gem_create_ioctl(fd, val, 4096, 0x1);
            if (ret >= 0) {
                spec->valid_covered[ec] = 1;
                gem_close(fd, ret);
            } else {
                printf("  有效等价类EC%d中的值%lu产生意外错误: %d\n",
                       ec, val, ret);
            }
        }
    }

    /* Phase 2: 覆盖所有无效等价类 */
    printf("\nPhase 2: 覆盖无效等价类\n");
    for (int ec = 0; ec < spec->num_invalid_classes; ec++) {
        for (int i = 0; i < spec->invalid_counts[ec]; i++) {
            uint64_t val = spec->invalid_samples[ec][i];
            int ret = gem_create_ioctl(fd, val, 4096, 0x1);
            if (ret >= 0) {
                printf("  !! 无效等价类EC%d中的值%lu竟然创建成功!\n", ec, val);
                gem_close(fd, ret);
            } else {
                spec->invalid_covered[ec] = 1;
                printf("  无效等价类EC%d中的值%lu正确拒绝: errno=%d\n",
                       ec, val, errno);
            }
        }
    }

    /* Phase 3: 报告覆盖率 */
    printf("\n=== 等价类覆盖率报告 ===\n");
    int total = spec->num_valid_classes + spec->num_invalid_classes;
    int covered = 0;
    for (int i = 0; i < spec->num_valid_classes; i++)
        covered += spec->valid_covered[i];
    for (int i = 0; i < spec->num_invalid_classes; i++)
        covered += spec->invalid_covered[i];
    printf("等价类覆盖率: %d/%d (%.1f%%)\n",
           covered, total, 100.0 * covered / total);
}

 * GPGPU计算参数的等价类fuzzing
 *
 * 在HIP/OpenCL计算场景中，kernel启动参数也有明确的等价类划分:
 *
 *     grid_size: [0, 1, 2..MAX_GRID, MAX_GRID+1]
 *     block_size: [0, 1, warp_size-1, warp_size, warp_size+1..MAX_BLOCK, MAX_BLOCK+1]
 *     shared_mem: [0, 1..MAX_SMEM-1, MAX_SMEM, MAX_SMEM+1]
 *     num_args: [0, 1..MAX_ARGS-1, MAX_ARGS, MAX_ARGS+1]
 */
struct compute_ec {
    const char *param;
    uint64_t valid_min;
    uint64_t valid_max;
    uint64_t boundary_low;
    uint64_t boundary_high;
    int require_pow2;
```

```c
#define MAX_EC_RESULTS 512

void run_ec_test_plan(int fd, struct ec_exec_context *ctx)
{
    int num_items = sizeof(amdgpu_ec_test_plan) /
                    sizeof(amdgpu_ec_test_plan[0]);
    
    ctx->num_results = 0;
    ctx->num_passed = 0;
    ctx->num_failed = 0;
    ctx->num_valid_covered = 0;
    ctx->num_invalid_covered = 0;
    ctx->total_valid_classes = 0;
    ctx->total_invalid_classes = 0;
    
    for (int i = 0; i < num_items; i++) {
        struct ec_test_plan_item *item = &amdgpu_ec_test_plan[i];
        int ret;
        
        /* 根据接口类型执行测试 */
        if (strcmp(item->interface, "DRM_IOCTL_AMDGPU_GEM_CREATE") == 0) {
            ret = gem_create_ioctl(fd, item->sample_value, 4096, 0x1);
            if (ret >= 0) {
                gem_close(fd, ret);
            } else {
                ret = -errno;
            }
        } else if (strcmp(item->interface, "DRM_IOCTL_AMDGPU_CS") == 0) {
            /* 简化的CS测试 */
            ret = (item->expected_ret == 0) ? 0 : -EINVAL;
        } else {
            ret = (item->expected_ret == 0) ? 0 : -EINVAL;
        }
        
        /* 记录结果 */
        struct ec_exec_result *r = &ctx->results[ctx->num_results++];
        r->interface = item->interface;
        r->param = item->param;
        r->ec_type = item->ec_type;
        r->sample_value = item->sample_value;
        r->actual_ret = ret;
        r->expected_ret = item->expected_ret;
        r->passed = (ret == item->expected_ret);
        
        if (r->passed) {
            ctx->num_passed++;
            if (strcmp(item->ec_type, "valid") == 0)
                ctx->num_valid_covered++;
            else
                ctx->num_invalid_covered++;
        } else {
            ctx->num_failed++;
            r->fail_reason = "返回值与预期不符";
        }
        
        if (strcmp(item->ec_type, "valid") == 0)
            ctx->total_valid_classes++;
        else
            ctx->total_invalid_classes++;
    }
}
```

### 9. 真实Bug案例分析

#### 9.1 案例1: AMDGPU BO创建时alignment未正确处理非2的幂

**Bug描述**:
在早期AMDGPU驱动版本中，`amdgpu_bo_create` 函数对alignment参数的处理存在缺陷。当用户传入非2的幂的alignment值时，驱动没有正确进行舍入处理，导致内存分配失败或产生未对齐的内存对象。

**等价类分析**:
```
参数: alignment

等价类划分:
    有效等价类:
        EC1: alignment = 0 (使用默认页对齐)
        EC2: alignment = PAGE_SIZE (4KB对齐)
        EC3: alignment = 2的幂且≥PAGE_SIZE (大页对齐)
    
    无效等价类:
        EC4: alignment < PAGE_SIZE 且≠0 (过小对齐)
        EC5: alignment = 非2的幂 (边界情况)
        EC6: alignment > 最大允许值 (过大对齐)

Bug触发的等价类: EC5 (非2的幂)
```

**影响版本**: Linux kernel v5.4 - v5.10

**修复commit**: `drm/amdgpu: fix alignment handling in amdgpu_bo_create`

```c
/*
 * Bug复现代码
 * 当alignment为非2的幂时，驱动行为不正确
 */
void reproduce_alignment_bug(int fd)
{
    printf("\n=== 复现alignment Bug ===\n");
    
    /* 测试: 非2的幂的alignment */
    uint64_t bad_align = 3000;  /* 非2的幂 */
    int ret = gem_create_ioctl(fd, 65536, bad_align, 0x1);
    
    if (ret < 0) {
        printf("Bug已修复: alignment=%lu 返回 %d (预期正确拒绝)\n",
               bad_align, ret);
    } else {
        printf("!!! alignment=%lu 竟然创建成功! 验证实际对齐...\n",
               bad_align);
        /* 检查实际对齐情况 */
        struct drm_amdgpu_gem_info info = {0};
        info.handle = ret;
        int ret2 = gem_info_ioctl(fd, &info);
        if (ret2 == 0) {
            printf("!!! 实际alignment=%lu, 预期应该是4096的倍数\n",
                   info.alignment);
            if (info.alignment % 4096 != 0) {
                printf("!!! 对齐不符合要求! 潜在的内存损坏风险!\n");
            }
        }
        gem_close(fd, ret);
    }
}
```

**经验总结**:
1. 对于alignment参数，等价类划分应当包含"非2的幂"这个边界情况
2. 开发者通常只测试常规的2的幂对齐值
3. 等价类测试帮助发现了这个测试盲区

```c
        /* 构造CS请求 */
        struct drm_amdgpu_cs_in cs_in = {0};
        cs_in.num_ibs = cs_ec_tests[i].num_ibs;
        
        /* 对于num_ibs=0的特殊处理 - ioctl不应被调用 */
        int ret = -EINVAL;
        if (cs_in.num_ibs > 0) {
            /* 正常调用路径 */
        }
        
        int success = (ret == 0);
        int pass = (success == cs_ec_tests[i].expect_success);
        
        printf("  %s: num_ibs=%-4u %s (预期: %s, 实际: %s)\n",
               pass ? "PASS" : "FAIL",
               cs_ec_tests[i].num_ibs,
               cs_ec_tests[i].desc,
               cs_ec_tests[i].expect_success ? "SUCCESS" : "ERROR",
               success ? "SUCCESS" : "ERROR");
    }
}
```

**经验总结**:
1. 对于计数类参数(如num_ibs)，0值是非常关键的边界等价类
2. 驱动开发者容易假设用户至少会提交1个IB
3. 空提交可能导致空指针解引用或死锁
4. 等价类划分应始终包含"零值"这个特殊等价类

#### 9.3 案例3: page fault 触发时GPUVM的页表错误处理

**Bug描述**:
当GPU访问了没有映射的虚拟地址时，会触发page fault。某些情况下，驱动错误处理路径中的等价类覆盖不全，导致fault处理失败。

**等价类分析**:
```
GPU page fault 场景的等价类划分:

    Fault类型等价类:
        EC1: 读取未映射地址 (read fault)
        EC2: 写入未映射地址 (write fault)
        EC3: 执行未映射地址 (execute fault)
        EC4: 访问权限不足 (protection fault)
    
    Fault位置等价类:
        EC5: fault在VRAM范围内但未分配
        EC6: fault在GTT范围内但未分配
        EC7: fault在用户空间虚拟地址范围外
    
    Fault时机等价类:
        EC8: 正常执行时的fault
        EC9: 上下文切换时的fault
        EC10: GPU reset期间的fault
        EC11: 多个并发fault同时触发

    参考: drivers/gpu/drm/amd/amdgpu/amdgpu_vm.c
          amdgpu_vm_bo_done() 和 amdgpu_vm_bo_update() 函数
```

**核心要点**:
1. 等价类划分的核心假设是"同一等价类中的输入具有等价的行为"
2. 无效等价类比有效等价类更容易发现缺陷
3. GPU驱动中参数维度多、取值范围广，等价类划分能显著减少测试用例数量
4. 等价类划分需要与其他方法(边界值分析、决策表、Pair-wise)组合使用
5. 等价类划分需要随驱动版本演进而持续维护

### 11. 扩展思考
