## 2. 决策表理论基础

### 2.1 核心概念

决策表是一种系统性的测试设计方法，适用于具有多个输入条件和多个动作结果的逻辑密集型系统。它将复杂的业务逻辑以表格形式展开，确保测试覆盖所有条件组合。

```
决策表的组成部分：

┌─────────────────────────────────────────────────────────────┐
│  决策表结构                                                   │
│                                                              │
│  ┌──────────────┬──────┬──────┬──────┬──────┬──────┐        │
│  │ 条件桩       │ R1   │ R2   │ R3   │ R4   │ ...  │        │
│  │ (Condition   │      │      │      │      │      │        │
│  │  Stub)       │      │      │      │      │      │        │
│  ├──────────────┼──────┼──────┼──────┼──────┼──────┤        │
│  │ C1: 条件1    │ Y    │ Y    │ N    │ N    │      │        │
│  │ C2: 条件2    │ Y    │ N    │ Y    │ N    │      │        │
│  ├──────────────┼──────┼──────┼──────┼──────┼──────┤        │
│  │ 动作桩       │      │      │      │      │      │        │
│  │ (Action      │      │      │      │      │      │        │
│  │  Stub)       │      │      │      │      │      │        │
│  ├──────────────┼──────┼──────┼──────┼──────┼──────┤        │
│  │ A1: 动作1    │ X    │      │      │ X    │      │        │
│  │ A2: 动作2    │      │ X    │ X    │      │      │        │
│  └──────────────┴──────┴──────┴──────┴──────┴──────┘        │
│                                                              │
│  条件桩：列出所有输入条件                                     │
│  动作桩：列出所有输出动作                                     │
│  规则列 (R1-Rn)：每种条件组合及对应的动作                     │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 决策表的数学基础

```
决策表的完备性分析：

n个布尔条件 → 2^n 种组合 → 2^n 条规则

条件数(n)    规则数(2^n)    备注
────────────────────────────────────
1            2             最简单的决策表
2            4             常用
3            8             适中
4            16            稍复杂
5            32            需要合并简化
6            64            考虑使用扩展条目
7            128           建议拆分
8            256           必须简化

当 n > 5 时，通常需要以下优化策略：
1. 使用"不关心"条目（Don't Care）合并规则
2. 将大表拆分为多个小表
3. 使用扩展条目（非布尔条件）
```

### 2.3 四种决策表类型

**2.3.1 有限条目决策表（Limited Entry Decision Table）**

```
所有条件均为布尔值（Y/N），动作也为布尔值（发生/不发生）

示例：GPU显示模式切换

条件：
  C1: 外部显示器已连接？
  C2: 当前为扩展模式？
  C3: 用户请求镜像模式？

动作：
  A1: 切换到镜像模式
  A2: 重新配置显示控制器
  A3: 通知窗口管理器
  A4: 保持当前配置

┌──────────────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 条件         │ R1   │ R2   │ R3   │ R4   │ R5   │ R6   │ R7   │ R8   │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ C1: 显示器   │ Y    │ Y    │ Y    │ Y    │ N    │ N    │ N    │ N    │
│ C2: 扩展模式 │ Y    │ Y    │ N    │ N    │ Y    │ Y    │ N    │ N    │
│ C3: 镜像请求 │ Y    │ N    │ Y    │ N    │ Y    │ N    │ Y    │ N    │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ A1: 镜像模式 │ X    │      │ X    │      │      │      │      │      │
│ A2: 重配置   │ X    │ X    │ X    │      │      │      │      │      │
│ A3: 通知WM   │ X    │ X    │ X    │      │      │      │      │      │
│ A4: 保持配置 │      │      │      │ X    │ X    │ X    │ X    │ X    │
└──────────────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘

分析：
  R1: 有外部显示器、当前扩展、请求镜像 → 切换到镜像
  R2: 有外部显示器、当前扩展、不请求镜像 → 重新配置（维持在扩展）
  R4-R8: 无外部显示器 → 保持当前配置（动作不变，可以合并）
```

**2.3.2 扩展条目决策表（Extended Entry Decision Table）**

```
条件可以是多值或非布尔类型

示例：GPU频率管理策略

条件：
  C1: 工作负载类型 ∈ {游戏, 计算, 视频播放, 空闲}
  C2: 温度状态 ∈ {正常, 偏高, 过热}
  C3: 电源状态 ∈ {插电, 电池, 节能模式}

动作：
  A1: 设置高频区间（>2000MHz）
  A2: 设置中频区间（1200-2000MHz）
  A3: 设置低频区间（<1200MHz）
  A4: 触发降频保护

┌──────────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│ 条件         │ R1       │ R2       │ R3       │ R4       │ R5       │
├──────────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ C1: 负载     │ 游戏     │ 计算     │ 视频播放  │ 空闲     │ -        │
│ C2: 温度     │ 正常     │ 正常     │ 正常     │ 正常     │ 过热     │
│ C3: 电源     │ 插电     │ 插电     │ 任意     │ 任意     │ -        │
├──────────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ A1: 高频     │ X        │ X        │          │          │          │
│ A2: 中频     │          │          │ X        │          │          │
│ A3: 低频     │          │          │          │ X        │          │
│ A4: 降频保护 │          │          │          │          │ X        │
└──────────────┴──────────┴──────────┴──────────┴──────────┴──────────┘

注意：R5中C1和C3为"不关心"(-)，表示无论什么负载和电源状态，
     只要温度过热就触发降频保护。
```

**2.3.3 混合条目决策表（Mixed Entry Decision Table）**

```
同时包含有限条目和扩展条目

示例：GPU内存分配

条件：
  C1: 请求大小 > 可用VRAM？（布尔）
  C2: 请求域 ∈ {VRAM, GTT, CPU}
  C3: 允许GTT回退？（布尔）
  C4: 对齐要求 ∈ {4KB, 2MB, 1GB}

动作：
  A1: 分配VRAM
  A2: 分配GTT
  A3: 分配系统内存
  A4: 返回ENOMEM

┌──────────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│ 条件         │ R1       │ R2       │ R3       │ R4       │ R5       │
├──────────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ C1: 超VRAM   │ N        │ Y        │ Y        │ Y        │ Y        │
│ C2: 请求域   │ VRAM     │ VRAM     │ GTT      │ CPU      │ VRAM     │
│ C3: GTT回退  │ -        │ N        │ -        │ -        │ Y        │
│ C4: 对齐     │ 4KB      │ 2MB      │ 4KB      │ 4KB      │ 1GB      │
├──────────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ A1: 分配VRAM │ X        │          │          │          │          │
│ A2: 分配GTT  │          │          │ X        │          │ X        │
│ A3: 分配系统 │          │          │          │ X        │          │
│ A4: ENOMEM   │          │ X        │          │          │          │
└──────────────┴──────────┴──────────┴──────────┴──────────┴──────────┘

R2分析：请求VRAM但超过容量且不允许GTT回退 → 返回ENOMEM
R4分析：请求CPU域（系统内存） → 分配系统内存
```

**2.3.4 不关心条目决策表（Decision Table with Don't Care Conditions）**

```
使用"-"表示条件不影响结果，用于合并规则减少测试用例

示例：GPU上下文创建

条件：
  C1: 驱动已加载？（布尔）
  C2: GPU硬件可用？（布尔）
  C3: 权限足够？（布尔）
  C4: 资源足够？（布尔）

动作：
  A1: 创建上下文成功
  A2: 返回错误码

完整决策表（无合并）：2^4 = 16条规则

┌──────────────┬──────┬──────┬──────┬──────┬──────┐
│ 条件         │ R1   │ R2   │ R3   │ R4   │ R5   │
├──────────────┼──────┼──────┼──────┼──────┼──────┤
│ C1: 驱动加载 │ Y    │ N    │ Y    │ Y    │ Y    │
│ C2: 硬件可用 │ Y    │ -    │ N    │ Y    │ Y    │
│ C3: 权限足够 │ Y    │ -    │ -    │ N    │ Y    │
│ C4: 资源足够 │ Y    │ -    │ -    │ -    │ N    │
├──────────────┼──────┼──────┼──────┼──────┼──────┤
│ A1: 创建成功 │ X    │      │      │      │      │
│ A2: 返回错误 │      │ X    │ X    │ X    │ X    │
└──────────────┴──────┴──────┴──────┴──────┴──────┘

合并效果：16条 → 5条
R2: C1=N → 驱动未加载（无论其他条件）
R3: C1=Y, C2=N → 硬件不可用（无论权限和资源）
R4: C1=Y, C2=Y, C3=N → 权限不足（无论资源）
R5: C1=Y, C2=Y, C3=Y, C4=N → 资源不足
```

## 3. 因果图理论基础

### 3.1 核心概念

因果图（Cause-Effect Graph）是决策表的图形化前身，用于可视化展示条件（原因）与动作（结果）之间的逻辑关系。

```
因果图的基本符号：

因果关系：
  ┌───┐       ┌───┐
  │ C │───────│ A │       C → A: 如果C为真则A发生
  └───┘       └───┘

恒等关系（Identity）：
  ┌───┐       ┌───┐
  │ C │───────│ A │       C = A: A当且仅当C
  └───┘       └───┘

非关系（NOT）：
  ┌───┐   ○   ┌───┐
  │ C │───────│ A │       C = NOT A: A当且仅当C为假
  └───┘       └───┘

与关系（AND）：
  ┌───┐
  │ C1│──┐
  └───┘  │  ∧   ┌───┐
         ├──────│ A │       A = C1 ∧ C2: 两个条件都为真时A发生
  ┌───┐  │      └───┘
  │ C2│──┘
  └───┘

或关系（OR）：
  ┌───┐
  │ C1│──┐
  └───┘  │  ∨   ┌───┐
         ├──────│ A │       A = C1 ∨ C2: 至少一个条件为真时A发生
  ┌───┐  │      └───┘
  │ C2│──┘
  └───┘
```

### 3.2 因果图的约束关系

```
原因之间的约束（输入约束）：

互斥约束（Exclusive / E）：最多一个原因为真
  ┌───┐
  │ C1│──┐
  └───┘  │  E
         ├───
  ┌───┐  │
  │ C2│──┘
  └───┘

示例：GPU电源状态
  C1: 设置为P-State 0（最高性能）
  C2: 设置为P-State 1（中等性能）
  C3: 设置为P-State 2（节能）
  约束：E → 最多只能设置一个P-State

包含约束（Inclusive / I）：至少一个原因为真
  ┌───┐
  │ C1│──┐
  └───┘  │  I
         ├───
  ┌───┐  │
  │ C2│──┘
  └───┘

示例：GPU上下文创建
  C1: 创建计算上下文
  C2: 创建图形上下文
  约束：I → 至少需要一个上下文类型

唯一约束（One and Only One / O）：恰有一个原因为真
  ┌───┐
  │ C1│──┐
  └───┘  │  O
         ├───
  ┌───┐  │
  │ C2│──┘
  └───┘

示例：GPU BO分配域
  C1: 分配在VRAM
  C2: 分配在GTT
  C3: 分配在系统内存
  约束：O → 恰有一个分配域

结果之间的约束（输出约束）：

屏蔽约束（Mask / M）
  ┌───┐       ┌───┐
  │ A1│───────│ A2│       A1为真时A2被屏蔽（不能为真）
  └───┘       └───┘

示例：GPU频率设置
  A1: 频率设置成功
  A2: 返回频率错误
  约束：M → 成功和错误不能同时发生
```

### 3.3 从因果图到决策表

```
转换流程：

步骤1：识别原因（输入条件）
  ┌────────────────────────────────────────────┐
  │ 原因列表                                    │
  │  ├── C1:  温度 > 85°C                      │
  │  ├── C2:  功耗 > TDP                       │
  │  ├── C3:  用户请求降频                      │
  │  └── C4:  高优先级计算任务运行中             │
  └────────────────────────────────────────────┘

步骤2：识别结果（输出动作）
  ┌────────────────────────────────────────────┐
  │ 结果列表                                    │
  │  ├── A1: 触发降频保护（降低到安全频率）      │
  │  ├── A2: 限制最大频率（soft limit）          │
  │  ├── A3: 保持当前频率                        │
  │  └── A4: 通知系统管理员                      │
  └────────────────────────────────────────────┘

步骤3：绘制因果图

  C1 ──┐
       ├── ∧ ──┐
  C4 ──┘      │
              ├── ∨ ── A1（降频保护）
  C2 ─────────┘
       
  C2 ──────┐
           ├── ∨ ── A2（限制频率）
  C3 ──────┘
       
  NOT(A1) ∧ NOT(A2) ──── A3（保持当前频率）
  
  A1 ── M ── A4（通知管理员）

步骤4：转换为决策表

┌──────────────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 条件         │ R1   │ R2   │ R3   │ R4   │ R5   │ R6   │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ C1: 高温     │ Y    │ Y    │ N    │ N    │ N    │ Y    │
│ C2: 高功耗   │ Y    │ N    │ Y    │ N    │ N    │ Y    │
│ C3: 用户请求 │ -    │ -    │ Y    │ N    │ N    │ N    │
│ C4: 高优先级 │ Y    │ N    │ -    │ -    │ -    │ N    │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ A1: 降频保护 │ X    │ X    │      │      │      │      │
│ A2: 限制频率 │      │      │ X    │      │      │ X    │
│ A3: 保持频率 │      │      │      │ X    │ X    │      │
│ A4: 通知管理 │ X    │ X    │      │      │      │      │
└──────────────┴──────┴──────┴──────┴──────┴──────┴──────┘

步骤5：导出测试用例
  R1: 高温 + 高功耗 + 高优先级任务 → 降频保护 + 通知
  R2: 高温 + 非高功耗 + 非高优先级 → 降频保护 + 通知
  R3: 非高温 + 高功耗 + 用户请求 → 限制频率
  R4: 所有条件都不满足 → 保持频率
  R5: 只有用户请求 → 保持频率（无约束时忽略用户请求？注意检查逻辑）
  R6: 高温 + 高功耗 + 非高优先级 → 限制频率
```

## 4. GPU驱动中的决策表应用

### 4.1 KMS显示配置决策表

KMS（Kernel Mode Setting）包含复杂的条件逻辑，是决策表应用的理想场景。

**场景：GPU显示输出配置**

```
条件分析：
  C1: 显示器 EDID 读取成功？
  C2: 显示器分辨率在GPU支持范围内？
  C3: 带宽足够支持当前配置？
  C4: 显示管线有足够的时钟？
  C5: 用户指定强制模式？

动作：
  A1: 应用最佳分辨率（来自EDID）
  A2: 应用用户指定模式
  A3: 降级到可用分辨率
  A4: 报告连接失败
  A5: 启用DSC（显示流压缩）
  A6: 减少颜色深度（10bit → 8bit）

因果图：

  C1 ──┐
       ├── ∧ ──┐
  C2 ──┘      │
              ├── ∨ ── A5（启用DSC）
  C3 ─────────┘

  C1 ──┐
       ├── ∧ ──┐
  C5 ──┘      │
              ├── A2（用户指定模式）
  C3 ─────────┘

  C1 ──┐
       ├── ∧ ──┐
  C2 ──┘      │
              ├── ∧ ──┐
  C4 ─────────┘      │
                     ├── A1（最佳模式）
  NOT(C3) ───────────┘

  C1 ── N ── A4（报告失败）
  
  NOT(A1) ∧ NOT(A2) ── A3（降级）

决策表：

┌──────────────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 条件         │ R1   │ R2   │ R3   │ R4   │ R5   │ R6   │ R7   │ R8   │ R9   │ R10  │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ C1: EDID     │ Y    │ Y    │ Y    │ Y    │ Y    │ N    │ N    │ Y    │ Y    │ Y    │
│ C2: 范围     │ Y    │ Y    │ Y    │ Y    │ N    │ -    │ -    │ Y    │ Y    │ Y    │
│ C3: 带宽     │ Y    │ N    │ Y    │ Y    │ Y    │ -    │ -    │ N    │ Y    │ Y    │
│ C4: 时钟     │ Y    │ Y    │ N    │ N    │ Y    │ -    │ -    │ Y    │ Y    │ N    │
│ C5: 强制模式 │ N    │ N    │ N    │ Y    │ N    │ Y    │ N    │ N    │ Y    │ N    │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ A1: 最佳模式 │ X    │      │      │      │      │      │      │      │      │      │
│ A2: 用户指定 │      │      │      │ X    │      │      │      │      │ X    │      │
│ A3: 降级     │      │      │      │      │ X    │      │      │      │      │ X    │
│ A4: 连接失败 │      │      │      │      │      │ X    │ X    │      │      │      │
│ A5: 启用DSC  │      │ X    │      │      │      │      │      │ X    │      │      │
│ A6: 减少色深 │      │      │ X    │      │      │      │      │ X    │      │      │
└──────────────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘

规则分析：
  R1: EDID读取成功+分辨率在范围内+带宽充足+时钟充足+无强制模式 → 应用最佳分辨率
  R2: EDID读取成功+分辨率在范围内+带宽不足+时钟充足+无强制模式 → 启用DSC压缩
  R3: EDID读取成功+分辨率在范围内+带宽充足+时钟不足+无强制模式 → 减少颜色深度
  R4: EDID读取成功+分辨率在范围内+带宽充足+时钟不足+用户强制模式 → 应用用户指定模式
  R5: EDID读取成功+分辨率不在范围内+带宽充足+时钟充足+无强制模式 → 降级到可用分辨率
  R6: EDID读取失败+用户指定强制模式 → 尝试用户指定模式（可能失败）
  R7: EDID读取失败+无强制模式 → 报告连接失败
  R8: EDID读取成功+带宽不足+时钟不足 → 启用DSC+减少色深（双重压缩）
  R9: EDID读取成功+用户指定强制模式+带宽充足 → 强制执行用户配置（即使不是最佳）
  R10: EDID读取成功+时钟不足+无强制模式 → 降级分辨率

优化：通过"不关心"条目将原始 2^5 = 32 条规则简化为 10 条有效规则

### 4.2 GPU电源管理决策表

GPU电源管理包含多个条件和动作的复杂逻辑组合。

场景：AMDGPU PowerPlay频率调节

```
条件分析：

  C2: GPU温度 > 温度阈值T2（95°C）？
  C3: GPU功耗 > PPT（Package Power Target）？
  C4: 当前负载类型为GPU密集型？
  C5: 当前工作负载为计算任务（非图形）？

动作：
  A1: 降低GFXCLK到安全频率（<1500MHz）
  A2: 降低GFXCLK到最低频率（<500MHz）
  A3: 限制最大功耗（PPT硬限制）
  A4: 保持当前P-State
  A5: 提升MCLK（内存频率）以维持吞吐量
  A6: 触发紧急降频保护（Throttle）

因果图：

  C2 ──── A2（最低频率）
  
  C1 ──┐
       ├── ∨ ── A1（安全频率）
  C3 ──┘
  
  C1 ──┐
       ├── ∧ ──┐
  C3 ──┘      │
              ├── ∨ ── A6（紧急降频）
  C2 ─────────┘

  C1 ──┐
       ├── ∨ ── A3（功耗限制）
  C3 ──┘

  NOT(A1) ∧ NOT(A2) ∧ NOT(A6) ── A4（保持P-State）

  C4 ∧ C5 ── A5（提升MCLK）

决策表：

┌──────────────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 条件         │ R1   │ R2   │ R3   │ R4   │ R5   │ R6   │ R7   │ R8   │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ C1: T>T1     │ Y    │ Y    │ Y    │ N    │ N    │ N    │ N    │ Y    │
│ C2: T>T2     │ Y    │ N    │ N    │ N    │ N    │ N    │ N    │ N    │
│ C3: 超PPT    │ Y    │ Y    │ N    │ Y    │ N    │ N    │ Y    │ N    │
│ C4: GPU负载  │ -    │ Y    │ N    │ -    │ Y    │ N    │ -    │ Y    │
│ C5: 计算任务 │ -    │ N    │ -    │ -    │ Y    │ -    │ -    │ N    │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ A1: 安全频率 │      │ X    │ X    │      │      │      │      │ X    │
│ A2: 最低频率 │ X    │      │      │      │      │      │      │      │
│ A3: 功耗限制 │      │ X    │      │ X    │      │      │ X    │      │
│ A4: 保持状态 │      │      │      │      │      │ X    │      │      │
│ A5: 提升MCLK │      │      │      │      │ X    │      │      │      │

└──────────────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘

规则分析：
  R1: 温度>T2（即严重过热）+ 超PPT → 紧急降频+最低频率（最危险情况）
  R2: T1<温度<T2 + 超PPT + GPU负载（非计算） → 安全频率+功耗限制
  R3: T1<温度<T2 + 非超PPT + 非GPU负载 → 安全频率（仅因温度降频）
  R4: 温度正常 + 超PPT + 任意负载 → 功耗限制（功耗过高但温度可控）
  R5: 温度正常 + 非超PPT + GPU负载 + 计算任务 → 保持P-State+提升MCLK（高性能计算场景）
  R6: 温度正常 + 非超PPT + 非GPU负载 → 保持P-State（理想状态）
  R7: 温度正常 + 超PPT → 功耗限制（独立于负载类型）
  R8: T1<温度<T2 + 非超PPT + GPU负载（非计算） → 安全频率

### 5.2 KMS显示配置的IGT测试实现

以下示例演示如何使用IGT框架测试KMS显示配置的决策表规则。

```c
/*
 * IGT测试示例：KMS显示配置决策表验证
 * 对应决策表规则R1-R10
 *
 * 编译：在IGT源码目录下添加此测试，或使用以下命令独立编译：
 * gcc -o test_kms_decision_table test_kms_decision_table.c \
 *     `pkg-config --cflags --libs libdrm igt` -I/usr/include/igt
 */

#include "igt.h"

#define MAX_CONNECTORS 6
#define MAX_RESOLUTIONS 32

/* 显示配置状态结构体 */
struct display_config {
    bool edid_valid;
    bool resolution_supported;
    bool bandwidth_sufficient;
    bool clock_sufficient;
    bool force_mode;
    uint32_t preferred_width;
    uint32_t preferred_height;
    uint32_t forced_width;
    uint32_t forced_height;
};

/* 动作枚举 */
enum display_action {
    ACTION_BEST_MODE,       /* 应用最佳分辨率 */
    ACTION_USER_SPECIFIED,  /* 应用用户指定模式 */
    ACTION_DEGRADE,         /* 降级到可用分辨率 */
    ACTION_FAILURE,         /* 报告连接失败 */
    ACTION_DSC_ENABLE,      /* 启用DSC */
    ACTION_REDUCE_BPC       /* 减少颜色深度 */
};

/*
 * 决策表执行引擎：根据条件组合返回预期动作
 *
 * 对应决策表：
 * ┌──────────────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
 * │ 条件         │ R1   │ R2   │ R3   │ R4   │ R5   │ R6   │ R7   │ R8   │ R9   │ R10  │
 * ├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
 * │ C1: EDID     │ Y    │ Y    │ Y    │ Y    │ Y    │ N    │ N    │ Y    │ Y    │ Y    │
 * │ C2: 范围     │ Y    │ Y    │ Y    │ Y    │ N    │ -    │ -    │ Y    │ Y    │ Y    │
 * │ C3: 带宽     │ Y    │ N    │ Y    │ Y    │ Y    │ -    │ -    │ N    │ Y    │ Y    │
 * │ C4: 时钟     │ Y    │ Y    │ N    │ N    │ Y    │ -    │ -    │ Y    │ Y    │ N    │
 * │ C5: 强制模式 │ N    │ N    │ N    │ Y    │ N    │ Y    │ N    │ N    │ Y    │ N    │
 * ├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
 * │ A1: 最佳模式 │ X    │      │      │      │      │      │      │      │      │      │
 * │ A2: 用户指定 │      │      │      │ X    │      │      │      │      │ X    │      │
 * │ A3: 降级     │      │      │      │      │ X    │      │      │      │      │ X    │
 * │ A4: 连接失败 │      │      │      │      │      │ X    │ X    │      │      │      │
 * │ A5: 启用DSC  │      │ X    │      │      │      │      │      │ X    │      │      │
 * │ A6: 减少色深 │      │      │ X    │      │      │      │      │ X    │      │      │
 * └──────────────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
 */
int evaluate_kms_decision(struct display_config *cfg)
{
    int actions = 0;

    /* C1: EDID无效 → 连接失败 */
    if (!cfg->edid_valid) {
        if (cfg->force_mode) {
            actions |= (1 << ACTION_USER_SPECIFIED);
            return actions;
        }
        actions |= (1 << ACTION_FAILURE);
        return actions;
    }

    /* EDID有效的情况 */
    if (cfg->force_mode) {
        /* R4, R9: 用户强制模式 */
        actions |= (1 << ACTION_USER_SPECIFIED);
        return actions;
    }

    /* 分辨率不在范围内 */
    if (!cfg->resolution_supported) {
        /* R5: 降级 */
        actions |= (1 << ACTION_DEGRADE);
        return actions;
    }

    /* 带宽不足且时钟不足 */
    if (!cfg->bandwidth_sufficient && !cfg->clock_sufficient) {
        /* R8: 双重压缩 */
        actions |= (1 << ACTION_DSC_ENABLE);
        actions |= (1 << ACTION_REDUCE_BPC);
        return actions;
    }

    /* 带宽不足 */
    if (!cfg->bandwidth_sufficient) {
        /* R2: 启用DSC */
        actions |= (1 << ACTION_DSC_ENABLE);
        return actions;
    }

    /* 时钟不足 */
    if (!cfg->clock_sufficient) {
        /* R3, R10: 减少色深或降级 */
        if (cfg->bandwidth_sufficient) {
            actions |= (1 << ACTION_REDUCE_BPC);
        } else {
            actions |= (1 << ACTION_DEGRADE);
        }
        return actions;
    }

    /* R1: 所有条件满足 → 最佳模式 */
    actions |= (1 << ACTION_BEST_MODE);
    return actions;
}

/*
 * IGT测试：验证KMS显示配置决策表
 */
igt_main
{
    struct display_config cfg;

    igt_subtest("kms_decision_R1_best_mode") {
        /* R1: EDID有效, 范围支持, 带宽充足, 时钟充足, 无强制 */
        cfg.edid_valid = true;
        cfg.resolution_supported = true;
        cfg.bandwidth_sufficient = true;
        cfg.clock_sufficient = true;
        cfg.force_mode = false;

        int result = evaluate_kms_decision(&cfg);
        igt_assert(result & (1 << ACTION_BEST_MODE));
        igt_assert(!(result & (1 << ACTION_FAILURE)));
    }

### 5.3 GPU电源管理的决策表测试

```c
/*
 * 电源管理决策表测试框架
 */

enum pm_action {
    PM_SAFE_FREQ,       /* 安全频率 */
    PM_LOWEST_FREQ,     /* 最低频率 */
    PM_PPT_LIMIT,       /* 功耗限制 */
    PM_KEEP_STATE,      /* 保持状态 */
    PM_BOOST_MCLK,      /* 提升MCLK */
    PM_THROTTLE         /* 紧急降频 */
};

struct pm_conditions {
    bool temp_over_t1;   /* 温度 > 85°C */
    bool temp_over_t2;   /* 温度 > 95°C */
    bool over_ppt;       /* 超过功耗目标 */
    bool gpu_load;       /* GPU密集型负载 */
    bool compute_task;   /* 计算任务 */
};

int evaluate_pm_decision(struct pm_conditions *pm) {
    int actions = 0;

    /* R1: 严重过热 + 超PPT */
    if (pm->temp_over_t2 && pm->over_ppt) {
        actions |= (1 << PM_LOWEST_FREQ);
        actions |= (1 << PM_THROTTLE);
        return actions;
    }

    /* R2: T1<温度<T2 + 超PPT + GPU负载 */
    if (pm->temp_over_t1 && !pm->temp_over_t2 &&
        pm->over_ppt && pm->gpu_load && !pm->compute_task) {
        actions |= (1 << PM_SAFE_FREQ);
        actions |= (1 << PM_PPT_LIMIT);
        return actions;
    }

    /* R3: T1<温度<T2 + 非超PPT + 非GPU负载 */
    if (pm->temp_over_t1 && !pm->temp_over_t2 &&
        !pm->over_ppt && !pm->gpu_load) {
        actions |= (1 << PM_SAFE_FREQ);
        return actions;
    }

    /* R5: 温度正常 + 非超PPT + GPU负载 + 计算任务 */
    if (!pm->temp_over_t1 && !pm->over_ppt &&
        pm->gpu_load && pm->compute_task) {
        actions |= (1 << PM_KEEP_STATE);
        actions |= (1 << PM_BOOST_MCLK);
        return actions;
    }

    /* R6: 理想状态 */
    if (!pm->temp_over_t1 && !pm->over_ppt && !pm->gpu_load) {
        actions |= (1 << PM_KEEP_STATE);
        return actions;
    }

    /* R4, R7: 超PPT（独立于温度和负载） */
    if (pm->over_ppt) {
        actions |= (1 << PM_PPT_LIMIT);
        return actions;
    }

    /* R8: T1<温度<T2 + 非超PPT + GPU负载 */
    if (pm->temp_over_t1 && !pm->temp_over_t2 &&
        !pm->over_ppt && pm->gpu_load) {
        actions |= (1 << PM_SAFE_FREQ);
        return actions;
    }

    return actions;
}
```

## 6. 实践案例：AMDGPU BO泄漏Bug的决策表分析

### 6.1 Bug背景

参考AMDGPU公开Bug报告：freedesktop.org Bugzilla #110421 - "amdgpu: BO memory leak when GEM_FLINK used with exported dma-buf"

```
Bug描述：
┌─────────────────────────────────────────────────────────────┐
│ Bug ID: 110421                                               │
│ 标题: amdgpu: BO memory leak when GEM_FLINK used with       │
│        exported dma-buf                                      │
│ 组件: DRM/AMDGPU-Driver                                      │
│ 影响: 4.19+ 内核版本                                          │
│ 现象: 使用GEM_FLINK获取相同BO的多个fd时，BO引用计数           │
│       未正确增加，导致提前释放或内存泄漏                        │
│ 复现: 进程A导出dma-buf → 进程B通过GEM_FLINK打开 →            │
│       进程A关闭fd → BO被释放 → 进程B访问非法内存               │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 决策表分析

```
BO引用计数管理的条件与动作分析：

条件：
  C1: BO当前引用计数 > 0？
  C2: BO被多个进程共享（通过GEM_FLINK或dma-buf）？
  C3: 关闭操作来自最后一个引用者？
  C4: BO关联了有效的dma-buf句柄？
  C5: BO正在被GPU使用（有未完成的fence）？

动作：
  A1: 释放BO（回收到空闲列表）
  A2: 减少引用计数
  A3: 标记BO为延迟释放（等待GPU完成）
  A4: 触发WARNING（引用计数异常）
  A5: 保留BO但移除fd映射

正确的决策表（修复前）：

┌──────────────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 条件         │ R1   │ R2   │ R3   │ R4   │ R5   │ R6   │ R7   │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ C1: ref>0    │ Y    │ Y    │ Y    │ N    │ N    │ N    │ Y    │
│ C2: 共享     │ Y    │ Y    │ N    │ -    │ -    │ -    │ N    │
│ C3: 最后引用 │ Y    │ N    │ Y    │ -    │ -    │ -    │ N    │
│ C4: dma-buf  │ Y    │ N    │ -    │ -    │ -    │ -    │ Y    │
│ C5: GPU使用  │ Y    │ -    │ N    │ -    │ -    │ -    │ Y    │
├──────────────┼──────┼──────┼──────┼──────┼──────┼──────┼──────┤
│ A1: 释放BO   │      │      │ X    │      │      │      │      │
│ A2: 减少ref  │      │ X    │ X    │      │      │      │      │
│ A3: 延迟释放 │ X    │      │      │      │      │      │ X    │
│ A4: WARNING  │      │      │      │ X    │ X    │ X    │      │
│ A5: 保留映射 │      │ X    │      │      │      │      │      │
└──────────────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘

Bug分析：修复前的代码在R2场景中（共享+非最后引用）未能正确处理
dma-buf关联的引用计数，导致引用计数少加1。当另一个进程通过
GEM_FLINK打开时，内核代码路径如下：

1. 进程A创建BO: ref=1
2. 进程A导出dma-buf: ref=2 (dma_buf_get增加ref)
3. 进程B通过GEM_FLINK打开: 期望ref=3, 实际ref=2 (Bug!)
4. 进程A关闭BO fd: ref从2减到1 (期望3→2)
5. 进程A关闭dma-buf: ref从1减到0, BO被释放 (Bug! B还在使用)
6. 进程B访问已释放的BO → use-after-free

修复方案：
  在GEM_FLINK路径中添加：if (bo->dma_buf) drm_gem_object_get(bo);
```

```

## 8. 决策表与因果图的局限性

```
局限性分析：

1. 条件爆炸问题
   条件数(n)与规则数(2^n)呈指数关系
   n=10 → 1024条规则
   简化策略：规则合并、表拆分、扩展条目

2. 仅适用于离散条件
   对连续值（频率、温度、电压）需要先离散化
   离散化可能丢失边界附近的缺陷

3. 不考虑时序依赖
   因果图是静态的，不描述条件的时间顺序
   GPU驱动中的竞态条件无法用决策表完全描述
   补充方法：状态转换图、时序图

4. 组合爆炸优化
   当存在大量"不关心"条目时，规则合并可能过度简化
   需要在测试覆盖率和效率之间权衡

5. 条件间依赖性
   某些条件可能隐含依赖（如C1和C2互斥）
   需要在因果图中显式标注约束关系（E/I/O/M）
```

## 9. 自动化决策表生成工具

### 9.1 决策表最佳实践与常见陷阱

```
最佳实践清单：

┌─────────────────────────────────────────────────────────────────┐
│ 实践1: 保持条件粒度一致性                                         │
│   每个条件应该是原子性的布尔判断，避免复合条件                      │
│   错误示例: "温度>85°C且功耗<PPT"（复合条件）                     │
│   正确示例: "温度>85°C" 和 "功耗<PPT"（两个独立条件）              │
│                                                                 │
│ 实践2: 标注"不关心"条目时要谨慎                                     │
│   只有明确知道该条件不影响结果时才使用"-"                           │
│   过度使用不关心条目可能导致遗漏关键测试场景                        │
│                                                                 │
│ 实践3: 规则合并的边界验证                                          │
│   合并前: 分别测试每个规则                                       │
│   合并后: 验证合并后的规则覆盖了原始规则的所有情况                  │
│                                                                 │
│ 实践4: GPU驱动中的禁忌条件                                          │
│   某些条件组合在硬件层面不可能发生（如频率同时为0且非0）            │
│   需要在因果图中使用E/I/O/M约束显式标注                             │
│                                                                 │
│ 实践5: 决策表版本管理                                              │
│   每次驱动逻辑变更后更新决策表                                    │
│   保持决策表与IGT测试用例的同步                                   │
│   在代码注释中引用决策表规则编号                                   │
└─────────────────────────────────────────────────────────────────┘

常见陷阱：

陷阱1: 遗漏"不可能发生"的组合
  后果: 测试用例覆盖了硬件不可达路径，浪费测试资源
  案例: 在BO分配中，"请求VRAM为真"和"请求GTT为真"同时成立，
        但驱动逻辑是互斥的（只能选择一种分配域）
  解决: 在因果图中使用E约束（互斥）标记

陷阱2: 条件顺序依赖
  后果: 决策表忽略了条件评估顺序对结果的影响
  案例: GPU上下文创建中，"权限检查"和"资源检查"的先后顺序
        可能导致不同的错误码（EPERM vs EBUSY）
  解决: 结合场景法，验证不同条件评估顺序下的行为

陷阱3: 边界附近的规则失效
  后果: 决策表规则在条件边界附近可能被违反
  案例: "温度 > 85°C"的决策边界，温度恰好等于85°C时
        可能触发错误的分支
  解决: 在决策表测试中配合边界值分析，验证每个条件边界

陷阱4: 同步问题导致的条件瞬间无效
  后果: 并发场景下，决策条件在读取和动作执行之间发生变化
  案例: 检查"VRAM空闲"为真后，在分配BO前其他线程占用了VRAM
  解决: 在并发测试中，使用决策表+压力测试组合方法
```

### 9.2 大规模决策表的拆分策略

当条件数量超过5个时，考虑表拆分：

```
拆分策略示例：AMDGPU显示配置（8个条件）

原始表：2^8 = 256条规则 → 不可管理

策略1：按条件层次拆分
  表A: EDID相关条件 → 确定是否连接
  ┌────────────────────────────────────────────┐
  │  条件: EDID有效, 热插拔信号, I2C通信正常    │
  │  动作: 连接成功, 连接失败, 检测中            │
  └────────────────────────────────────────────┘
  
  表B: 分辨率相关条件 → 确定显示模式
  ┌────────────────────────────────────────────┐
  │  条件: 分辨率在范围内, 带宽充足, 时钟充足    │
  │  动作: 最佳模式, 降级, 启用DSC              │
  └────────────────────────────────────────────┘
  
  表C: 用户偏好相关条件 → 确定最终输出
  ┌────────────────────────────────────────────┐
  │  条件: 强制模式, 色彩深度偏好, 刷新率偏好   │
  │  动作: 应用用户配置, 自动适配, 保持默认      │
  └────────────────────────────────────────────┘

策略2：按功能模块拆分
  表A: 初始连接阶段（4个条件） → 8条规则
  表B: 模式设置阶段（3个条件） → 4条规则
  表C: 输出验证阶段（3个条件） → 4条规则

策略3：使用扩展条目代替有限条目
  将某些布尔条件改为枚举条件，减少条件数量
  例如: "温度状态" = {正常, 偏高, 临界} 代替两个布尔条件

#!/usr/bin/env python3
"""
决策表驱动的测试用例生成器
适用于AMDGPU驱动测试的自动化测试设计
"""

from enum import Enum
from itertools import product
from typing import List, Dict, Tuple, Optional


class ConditionType(Enum):
    BOOLEAN = "boolean"     # Y/N
    ENUM = "enum"           # 枚举值
    RANGE = "range"         # 范围值


class DecisionTable:
    """决策表类"""
    
    def __init__(self, name: str):
        self.name = name
        self.conditions: List[Dict] = []  # 条件列表
        self.actions: List[str] = []      # 动作列表
        self.rules: List[Dict] = []       # 规则列表
    
    def add_condition(self, name: str, 
                      cond_type: ConditionType = ConditionType.BOOLEAN,
                      values: Optional[List] = None):
        """添加条件"""
        cond = {
            'name': name,
            'type': cond_type,
            'values': values or [True, False]
        }
        self.conditions.append(cond)
    
    def add_action(self, name: str):
        """添加动作"""
        self.actions.append(name)
    
    def generate_full_table(self) -> List[Dict]:
        """生成完整决策表（2^n条规则）"""
        bool_conds = [c for c in self.conditions 
                     if c['type'] == ConditionType.BOOLEAN]
        enum_conds = [c for c in self.conditions 
                     if c['type'] == ConditionType.ENUM]
        
        bool_combinations = list(product([True, False], 
                                        repeat=len(bool_conds)))
        enum_values = [c['values'] for c in enum_conds]
        enum_combinations = list(product(*enum_values)) if enum_values else [()]
        
        rules = []
        for bc in bool_combinations:
            for ec in enum_combinations:
                rule = {}
                for i, cond in enumerate(bool_conds):
                    rule[cond['name']] = bc[i]
                for i, cond in enumerate(enum_conds):
                    rule[cond['name']] = ec[i]
                rules.append(rule)
        
        return rules
    
    def merge_rules(self, rules: List[Dict]) -> List[Dict]:
        """使用Don't Care合并规则"""
        # 简化实现：相同动作的规则合并
        merged = []
        seen = set()
        
        for rule in rules:
            action_key = tuple(sorted(rule.get('action', {}).items()))
            if action_key not in seen:
                seen.add(action_key)
                merged.append(rule)
        
        return merged
    
    def to_test_cases(self, rule_mapping: Dict) -> List[str]:
        """生成测试用例描述"""
        test_cases = []
        
        for rule_name, conditions in rule_mapping.items():
            tc = f"  {rule_name}:\n"
            tc += f"    条件: {', '.join(f'{k}={v}' for k, v in conditions.items())}\n"
            test_cases.append(tc)
        
        return test_cases

class CauseEffectGraph:
    """因果图类"""
    
    def __init__(self, name: str):
        self.name = name
        self.causes: List[str] = []
        self.effects: List[str] = []
        self.edges: List[Tuple] = []  # (from, to, type)
        self.constraints: List[Tuple] = []  # (node1, node2, type)
    
    def add_cause(self, name: str):
        """添加原因节点"""
        self.causes.append(name)
    
    def add_effect(self, name: str):
        """添加结果节点"""
        self.effects.append(name)
    
    def add_edge(self, from_node: str, to_node: str, 
                 edge_type: str = "identity"):
        """添加边: identity, not, and, or"""
        self.edges.append((from_node, to_node, edge_type))
    
    def add_constraint(self, node1: str, node2: str, 
                       constraint_type: str):
        """添加约束: E, I, O, M"""
        self.constraints.append((node1, node2, constraint_type))
    
    def generate_truth_table(self) -> List[Dict]:
        """从因果图生成真值表"""
        n = len(self.causes)
        truth_table = []
        
        for bits in product([0, 1], repeat=n):
            row = {}
            for i, cause in enumerate(self.causes):
                row[cause] = bool(bits[i])
            truth_table.append(row)
        
        return truth_table

ceg.add_cause("temp_critical")  # C2: 温度 > T2

ceg.add_edge("temp_high", "safe_freq", "or")
ceg.add_edge("over_ppt", "safe_freq", "or")

class DecisionTableValidator:
    """决策表验证器：检查完备性、一致性和无冗余性"""
    
    def __init__(self):
        self.rules = []
        self.warnings = []
        self.errors = []
    
    def add_rule(self, rule_id: str, conditions: Dict, 
                 expected_actions: List[str]):
        """添加规则到验证器"""
        self.rules.append({
            'id': rule_id,
            'conditions': conditions,
            'expected_actions': expected_actions
        })
    
    def check_completeness(self, all_conditions: List[str],
                          condition_values: Dict[str, List]) -> List[str]:
        """检查完备性：是否所有条件组合都有对应规则"""
        missing = []
        
        # 生成所有可能组合
        all_values = [condition_values[c] for c in all_conditions]
        for combo in product(*all_values):
            combo_dict = dict(zip(all_conditions, combo))
            
            # 检查是否有规则覆盖此组合
            covered = False
            for rule in self.rules:
                matches = True
                for cond, val in combo_dict.items():
                    if cond in rule['conditions']:
                        if rule['conditions'][cond] is not None:
                            if rule['conditions'][cond] != val:
                                matches = False
                                break
                if matches:
                    covered = True
                    break
            
            if not covered:
                missing.append(combo_dict)
        
        return missing
    
    def check_consistency(self) -> List[Tuple[str, str, str]]:
        """检查一致性：同一条件组合不会映射到不同的动作"""
        conflicts = []
        
        for i, rule1 in enumerate(self.rules):
            for j, rule2 in enumerate(self.rules):
                if j <= i:
                    continue
                
                # 检查条件是否一致
                conditions_match = True
                for cond in set(list(rule1['conditions'].keys()) + 
                               list(rule2['conditions'].keys())):
                    v1 = rule1['conditions'].get(cond)
                    v2 = rule2['conditions'].get(cond)
                    if v1 is not None and v2 is not None and v1 != v2:
                        conditions_match = False
                        break
                
                if conditions_match:
                    # 条件一致但动作不同 → 冲突
                    if rule1['expected_actions'] != rule2['expected_actions']:
                        conflicts.append((
                            rule1['id'], rule2['id'],
                            f"相同条件映射到不同动作: "
                            f"{rule1['expected_actions']} vs "
                            f"{rule2['expected_actions']}"
                        ))
        
        return conflicts
    
    def generate_test_report(self) -> str:
        """生成验证报告"""
        report = []
        report.append("=" * 60)
        report.append("决策表验证报告")
        report.append("=" * 60)
        report.append(f"规则总数: {len(self.rules)}")
        
        conflicts = self.check_consistency()
        if conflicts:
            report.append(f"\n[错误] 发现 {len(conflicts)} 个冲突:")
            for c in conflicts:
                report.append(f"  {c[0]} vs {c[1]}: {c[2]}")
        else:
            report.append("\n[通过] 一致性检查: 无冲突")
        
        if self.warnings:
            report.append(f"\n[警告] {len(self.warnings)} 个:")
            for w in self.warnings:
                report.append(f"  {w}")
        
        report.append("\n" + "=" * 60)
        return "\n".join(report)

class TestCaseGenerator:
    """从决策表生成具体测试用例"""
    
    def __init__(self, dt: DecisionTable):
        self.dt = dt
        self.test_cases = []
    
    def generate_igt_test_cases(self, rule_mapping: Dict) -> List[str]:
        """生成IGT测试用例代码骨架"""
        test_cases = []
        
        for rule_name, conditions in rule_mapping.items():
            tc = f"""
/*
 * IGT Test Case: {self.dt.name} - {rule_name}
 * 条件: {conditions}
 */
static void
test_{rule_name.lower()}(igt_display_t *display)
{{
    /* 设置fixture */
    igt_fixture {{
        /* 初始化显示状态 */
    }}

    /* 应用条件组合 */
    {self._generate_condition_setup(conditions)}

    /* 执行操作 */
    {self._generate_action_verification(conditions)}

    /* 验证结果 */
    igt_assert_f(true, "{rule_name} 验证通过");
}}
"""
            test_cases.append(tc)
        
        return test_cases
    
    def _generate_condition_setup(self, conditions: Dict) -> str:
        """生成条件设置代码"""
        setup_lines = []
        for cond, value in conditions.items():
            if isinstance(value, bool):
                setup_lines.append(
                    f"    /* {cond} = {value} */"
                )
        return "\n".join(setup_lines)
    
    def _generate_action_verification(self, conditions: Dict) -> str:
        """生成动作验证代码"""
        return """    /* 执行操作 */
    int ret = ioctl(fd, DRM_IOCTL_AMDGPU_GEM_CREATE, &args);
    igt_assert_eq(ret, 0);"""

## 11. 今日小结

决策表和因果图是处理多条件逻辑场景的强大利器，在GPU驱动测试中有广泛的应用价值。今天的内容总结如下：

1. **决策表四类型**：有限条目（布尔条件）、扩展条目（多值条件）、混合条目（两类混合）和不关心条目（规则合并），适用于不同复杂度的场景。

2. **因果图转换**：因果图是决策表的图形化前身，使用Identity/NOT/AND/OR四种基本关系和E/I/O/M四种约束关系来描述条件与动作的逻辑连接。

3. **GPU驱动应用**：在KMS显示配置、电源管理频率调节、BO内存分配和上下文创建等场景中，决策表能够系统性地覆盖所有条件组合，避免测试盲区。

4. **IGT实践**：将决策表规则直接映射为IGT测试用例，建立从规则到代码的清晰对应关系，提高测试的可维护性和可追溯性。

5. **局限性认识**：决策表面临条件爆炸、连续值离散化、时序依赖缺失等挑战，需要与其他测试设计方法（边界值分析、状态转换图）配合使用。

**关键洞察**：决策表的价值不仅在于设计测试用例，更在于帮助测试人员系统化地理解驱动逻辑。绘制因果图的过程本身就能发现需求文档中的模糊点和逻辑矛盾。在GPU驱动测试中，决策表与等价类划分、边界值分析形成了"三位一体"的测试设计基础方法体系。
