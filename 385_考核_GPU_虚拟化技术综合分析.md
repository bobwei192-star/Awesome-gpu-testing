## 参考答案要点

### 选择题

| 题号 | 答案 | 解析 |
|------|------|------|
| 1 | **B** | PF 有完整配置空间和全部硬件功能；VF 有独立配置空间但功能是 PF 的子集 |
| 2 | **C** | VF 通过 Mailbox（MMIO 寄存器）与 PF 通信，不能直接访问 PF 的控制接口 |
| 3 | **C** | SR-IOV 性能可达原生的 90-95%，API 转发 50-80%，全虚拟化 1-5% |
| 4 | **C** | 上下文切换的主要开销来自保存和恢复 Wavefront 的执行状态 |
| 5 | **B** | 每 VF 有独立的 GART 页表，GPU 地址翻译硬件保证不能访问其他 VF 的地址范围 |
| 6 | **C** | VF 驱动可提交命令和分配内存（通过 PF 代理），但不能控制时钟/电源等全局资源 |
| 7 | **C** | Token Bucket 是带宽控制算法，限制 VF 的内存/PCIe 带宽使用 |
| 8 | **D** | Jain's Index = 1.0 表示完全公平，>0.95 为可接受 |
| 9 | **C** | GPU 核心频率不受虚拟化方案影响，主要瓶颈在模拟层开销 |
| 10 | **B** | 优先级继承防止低优先级 VF 持有高优先级 VF 需要的资源时导致的优先级反转 |

### 判断题

| 题号 | 答案 | 解析 |
|------|------|------|
| 11 | **×** | SR-IOV 的关键优势就是错误隔离，一个 VF hang 不影响其他 VF |
| 12 | **√** | VirGL 将 API 调用编码为命令流，通过 virtio 通道发给 Host 渲染 |
| 13 | **×** | VF 显存是静态分区，调整大小需销毁 VF 后重建 |
| 14 | **√** | IOMMU 提供 DMA 地址翻译和隔离，是 SR-IOV 的必需组件 |
| 15 | **√** | VF 不能直接访问 RAS 寄存器，错误需通过 Mailbox 报告给 PF 处理 |

### 简答题要点

**16. 三种方案对比：**
- 全虚拟化：性能 1-5%，密度高，无需硬件，兼容性好，隔离高 → 虚拟桌面控制台
- API 转发：性能 50-80%，密度高，无需硬件，部分 API，隔离中 → 云游戏
- SR-IOV：性能 90-95%，密度中，需硬件支持，完全 API，隔离高 → AI 训练

**17. 时间片调度：**
- 分配策略：Round Robin + 优先级权重
- 切换步骤：时间片耗尽 → 保存 WF 上下文 → 选择下一 VF → 恢复 WF 上下文 → 执行
- 开销组成：WF 状态保存(60%) + 页表切换(20%) + 缓存污染(20%)
- 抖动排查：检查切换延迟(大 WF 数)、时间片配置、优先级配置、带宽限制

**18. Mailbox 通信：**
- VF 侧：写入消息类型+参数到 Mailbox 寄存器 → 写入 TRIGGER → 轮询等待响应
- PF 侧：Mailbox 中断触发 → 读取消息 → 处理请求 → 写入响应
- 显存分配示例：ALLOC_VRAM → PF 分配 VF 分区内空间 → 返回地址偏移
- 超时处理：5 秒超时，超时后 VF 重试或报告错误

### 实践题要点

**19. 隔离性测试：**
- 显存隔离：跨 VF 地址访问验证
- 计算隔离：单 VF vs 多 VF 性能对比
- 错误隔离：VF hang 不影响其他 VF
- 带宽隔离：配置带宽限制后验证实际带宽

**20. GPU 共享方案：**
- 推荐 SR-IOV（AI 训练需要性能隔离）
- 配置 4 个 VF，显存分配 A:32GB, B:8GB, C:8GB, D:4GB + PF 保留
- 优先级：A=7(高), B=4(中), C=4(中), D=1(低)
- 权重：A=8, B=4, C=4, D=2
- 监控：每 VF 的 GPU 利用率、显存使用、延迟、错误计数
