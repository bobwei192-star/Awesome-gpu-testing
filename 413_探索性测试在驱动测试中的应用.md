## 知识详解

### 1. 探索性测试基本概念

#### 1.1 定义与起源

探索性测试（Exploratory Testing, ET）由 Cem Kaner 在1983年提出，是一种同时进行测试设计、测试执行和测试学习的测试方法。与传统的脚本化测试不同，探索性测试强调测试人员的主动性和创造性。

```
脚本化测试 vs 探索性测试:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  脚本化测试 (Scripted Testing)                                       │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │  设计 ──────────→ 编写脚本 ──────────→ 执行 ──────────→ 报告    │  │
  │  │  (提前完成)      (固定步骤)        (按部就班)    (对比预期)     │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │                          vs                                          │
  │                                                                     │
  │  探索性测试 (Exploratory Testing)                                    │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │  学习 ───→ 设计 ───→ 执行 ───→ 分析 ───→ 学习 (循环迭代)       │  │
  │  │  (理解系统)  (形成假设)  (操作验证)  (观察结果)  (更新认知)    │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  关键差异:                                                            │
  │    ├─ 脚本化测试: 测试用例 = 设计的产物                                │
  │    └─ 探索性测试: 测试过程 = 设计与执行同时进行                        │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

来源: Kaner et al., "Lessons Learned in Software Testing", 2001

#### 1.2 探索性测试在GPU驱动中的价值

GPU驱动测试中，探索性测试特别有价值，原因包括:

```
为什么GPU驱动需要探索性测试:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  1. 硬件状态空间巨大                                                 │
  │     ├─ GPU有数百个寄存器、数十个时钟域、多种电源状态                   │
  │     ├─ 组合状态数量 = 天文数字, 无法全覆盖                            │
  │     └─ 探索性测试可以发现意外的状态交互                                │
  │                                                                     │
  │  2. 非确定性行为                                                     │
  │     ├─ GPU调度是异步的, 测试结果受时序影响                            │
  │     ├─ 竞态条件在多线程/多进程场景下难以用脚本覆盖                      │
  │     └─ 探索性测试通过变化时序发现隐藏的Race Condition                 │
  │                                                                     │
  │  3. 硬件-软件交互复杂                                                │
  │     ├─ 驱动行为取决于固件版本、硬件步进、BIOS设置                     │
  │     ├─ 这些组合在脚本化测试中容易被忽略                                │
  │     └─ 探索性测试通过主动变化环境配置发现兼容性问题                    │
  │                                                                     │
  │  4. 缺乏完整规范                                                     │
  │     ├─ AMDGPU驱动的部分行为依赖于硬件白皮书或逆向工程                 │
  │     ├─ 脚本化测试只能验证已知行为                                     │
  │     └─ 探索性测试可以发现未定义行为或规范遗漏                          │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

#### 1.3 探索性测试的四个核心原则

```
探索性测试的四大原则:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  原则1: 基于学习的测试                                               │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ 每一次测试操作都在增加对系统的理解                              │  │
  │  │ 学习驱动后续测试设计, 形成正向反馈循环                          │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  原则2: 同时进行设计、执行和学习                                      │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ 没有固定的测试脚本                                               │  │
  │  │ 测试人员在执行过程中动态决定下一步操作                            │  │
  │  │ 测试本身就是设计活动                                             │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  原则3: 基于启发式的探索                                              │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ 使用经验法则指导测试方向                                         │  │
  │  │ 例如: "先测试边界", "变化一个参数"                              │  │
  │  │ 启发式方法使探索性测试更有效率                                    │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  原则4: 基于会话的结构化管理                                          │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ 虽然自由探索, 但需要结构化记录                                   │  │
  │  │ 使用基于会话的测试管理 (SBTM) 进行追踪                          │  │
  │  │ 每个会话有明确的时间盒和探索目标                                  │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

### 2. 基于会话的测试管理 (SBTM)

#### 2.1 SBTM核心概念

基于会话的测试管理（Session-Based Test Management）由 Jonathan Bach 提出，为探索性测试提供了结构化管理框架。

```
SBTM框架:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  测试会话 (Test Session)                                            │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │  时间盒: 60-120分钟                                            │  │
  │  │  结构:                                                         │  │
  │  │    ├─ 宪章 (Charter): 探索目标和范围定义                        │  │
  │  │    ├─ 探索 (Explore): 实际操作和观察                            │  │
  │  │    ├─ 记录 (Record): 笔记、Bug、覆盖区域                        │  │
  │  │    └─ 汇报 (Debrief): 会话总结和发现分享                        │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  测试宪章模板:                                                       │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │  探索 [目标区域]                                               │  │
  │  │  使用 [测试资源/工具]                                          │  │
  │  │  发现 [预期发现]                                               │  │
  │  │  约束: [边界条件]                                              │  │
  │  │  参考: [相关文档/Bug链接]                                      │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

#### 2.3 探索性测试会话类型

```
在GPU驱动测试中, 探索性测试会话分为以下几类:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  1. 基于功能的探索 (Feature-based)                                   │
  │     ├─ 宪章: 探索新功能 [X] 的正确性                                 │
  │     ├─ 示例: "探索 FreeSync 在 4K@144Hz 下的 VRR 范围"              │
  │     └─ 适用: 新功能验证初期                                          │
  │                                                                     │
  │  2. 基于Bug的探索 (Bug-based)                                        │
  │     ├─ 宪章: 探索 Bug [X] 的影响范围                                 │
  │     ├─ 示例: "探索 GCVM fault 在不同GPU型号上的复现条件"             │
  │     └─ 适用: Bug定位和影响范围评估                                   │
  │                                                                     │
  │  3. 基于场景的探索 (Scenario-based)                                  │
  │     ├─ 宪章: 模拟用户场景 [X] 的行为                                 │
  │     ├─ 示例: "探索 Steam 游戏启动时切换分辨率的行为"                 │
  │     └─ 适用: 真实使用场景验证                                        │
  │                                                                     │
  │  4. 基于变化的探索 (Variation-based)                                │
  │     ├─ 宪章: 变化参数 [X] 观察系统行为变化                           │
  │     ├─ 示例: "探索多显示器不同分辨率组合下的性能"                    │
  │     └─ 适用: 参数空间探索                                            │
  │                                                                     │
  │  5. 基于压力的探索 (Stress-based)                                    │
  │     ├─ 宪章: 在压力 [X] 下探索系统的稳定性                           │
  │     ├─ 示例: "探索频繁 modeset 下的显示稳定性"                      │
  │     └─ 适用: 极限条件验证                                            │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

### 3. GPU驱动探索性测试启发式方法

#### 3.1 GPU特定启发式

```
AMDGPU驱动探索性测试启发式:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  H1: 边界探索                                                       │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ 原理: GPU硬件和驱动在边界条件附近更容易暴露缺陷                  │  │
  │  │ 示例:                                                            │  │
  │  │   ├─ 最大分辨率 + 最大刷新率                                     │  │
  │  │   ├─ 最小/最大显存分配                                           │  │
  │  │   ├─ 最小/最大时钟频率                                           │  │
  │  │   └─ 最大显示器数量                                              │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  H2: 状态转换探索                                                   │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ 原理: GPU状态转换时的时序窗口是缺陷高发区                        │  │
  │  │ 示例:                                                            │  │
  │  │   ├─ 电源状态切换 (P0 ↔ P3 ↔ P8)                               │  │
  │  │   ├─ 时钟频率切换 (GFXCLK/MCLK变化)                             │  │
  │  │   ├─ modeset 中的显示模式切换                                    │  │
  │  │   └─ 挂起/恢复 (S3/S4)                                          │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  H3: 并发冲突探索                                                   │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  | 原理: GPU驱动的并发访问(多进程/多线程)竞争是典型Bug源            │  │
  │  | 示例:                                                            │  │
  │  |   ├─ 同时运行 GEM + KMS 测试                                     │  │
  │  |   ├─ 一边渲染一边调整电源管理参数                                │  │
  │  |   ├─ 同时多个进程进行 BO 分配/释放                               │  │
  │  |   └─ 热插拔 + 渲染同时进行                                       │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  H4: 硬件变化探索                                                   │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ 原理: 硬件事件(热插拔/断连/超时)触发驱动中的错误处理路径         │  │
  │  │ 示例:                                                            │  │
  │  │   ├─ 反复插拔显示器 (hotplug 循环)                               │  │
  │  │   ├─ 突然断开 DP 线缆 (无需通知)                                 │  │
  │  |   ├─ GPU 温度达到临界点 (thermal throttling)                    │  │
  │  |   └─ PCIe Link Down/Up (Surprise Removal)                       │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  H5: 组合交互探索                                                   │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ 原理: 多个子系统同时活动时的交互可能暴露集成缺陷                  │  │
  │  │ 示例:                                                            │  │
  │  │   ├─ 显示渲染 + 视频编码同时进行                                 │  │
  │  │   ├─ Vulkan 渲染 + ROCm 计算同时进行                            │  │
  │  │   ├─ Wayland + XWayland 混合桌面 + 游戏                         │  │
  │  │   └─ SR-IOV 中多个 VM 同时提交命令                              │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

#### 3.2 通用软件测试启发式

以下通用启发式同样适用于GPU驱动探索性测试，引用自 Bach & Bolton 的探索性测试方法论:

```
通用启发式分类:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  H7: SFDIPO (Structure/Function/Data/Interface/Platform/Operation)  │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ Structure: 测试驱动的代码结构 (模块间调用)                      │  │
  │  │ Function:  测试驱动的功能行为 (参数验证)                       │  │
  │  │ Data:      测试驱动的数据处理 (BO大小/对齐/格式)              │  │
  │  │ Interface: 测试驱动的外部接口 (ioctl/sysfs)                   │  │
  │  │ Platform:  测试不同平台 (x86/ARM, 不同内核版本)              │  │
  │  │ Operation: 测试操作序列 (操作顺序变化)                        │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  |  H8: RCRCRC (Right/Boundary/Configuration/Relationship/Constraints) │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ Right:        结果正确性 (渲染/编码结果验证)                   │  │
  │  │ Boundary:     边界条件 (最大/最小值)                           │  │
  │  │ Configuration:配置变化 (不同GPU型号/驱动参数)                  │  │
  │  │ Relationship: 关系验证 (事件顺序/因果关系)                     │  │
  │  │ Constraints:  约束条件 (资源限制/超时)                        │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

```
电源管理探索性测试脚本:

  #!/bin/bash
  # AMDGPU Power Management Exploratory Test Script
  # 参考: https://docs.kernel.org/gpu/amdgpu/thermal.html
  # 参考: https://gitlab.freedesktop.org/drm/igt-gpu-tools/-/tree/master/tests/amdgpu

  set -e

  GPU_SYSFS="/sys/class/drm/card0/device"
  LOG_FILE="pm_exploratory_$(date +%Y%m%d_%H%M).log"
  BUG_LOG="pm_bugs_$(date +%Y%m%d_%H%M).md"

  log() {
    echo "[$(date +%H:%M:%S)] $*" | tee -a "$LOG_FILE"
  }

  bug() {
    echo "[BUG] $*" | tee -a "$LOG_FILE"
    echo "- **发现**: $*" >> "$BUG_LOG"
    echo "  **时间**: $(date)" >> "$BUG_LOG"
    echo "  **状态**: $(cat $GPU_SYSFS/power_dpm_force_performance_level 2>/dev/null)" >> "$BUG_LOG"
    echo "  **温度**: $(cat $GPU_SYSFS/hwmon/hwmon*/temp1_input 2>/dev/null)" >> "$BUG_LOG"
    echo "" >> "$BUG_LOG"
  }

  monitor_dmesg() {
    dmesg | tail -5 | grep -i "amdgpu\|drm\|error\|fault\|warn" >> "$LOG_FILE" 2>&1
  }

  log "=== AMDGPU PM 探索性测试会话开始 ==="
  log "GPU: $(cat $GPU_SYSFS/vbios_ver 2>/dev/null)"
  log "驱动: $(cat $GPU_SYSFS/driver/amdgpu/version 2>/dev/null)"
  log "温度: $(cat $GPU_SYSFS/hwmon/hwmon*/temp1_input 2>/dev/null)"

  # ---- H2: 状态转换探索 ----
  log "=== H2: 电源状态转换测试 ==="

  for level in "low" "high" "auto" "manual"; do
    log "设置 power_dpm_force_performance_level = $level"
    echo "$level" > "$GPU_SYSFS/power_dpm_force_performance_level" 2>&1 || bug "切换 $level 失败"
    sleep 2
    monitor_dmesg
    log "当前时钟: $(cat $GPU_SYSFS/pp_dpm_sclk 2>/dev/null | head -3)"
  done

  # ---- 快速状态切换 ----
  log "=== 快速状态切换 (20 cycles) ==="
  for i in $(seq 1 20); do
    for level in "low" "auto" "high" "auto"; do
      echo "$level" > "$GPU_SYSFS/power_dpm_force_performance_level" 2>/dev/null || true
    done
  done
  monitor_dmesg
  log "快速切换完成, 检查是否有错误"

  # ---- H5: 组合交互 ----
  log "=== H5: 渲染 + 电源管理交互 ==="

  glmark2 --benchmark ':duration=30' --run-forever &
  GLMARK_PID=$!
  sleep 2

  for i in $(seq 1 10); do
    echo "manual" > "$GPU_SYSFS/power_dpm_force_performance_level" 2>/dev/null
    echo "s 0 1500" > "$GPU_SYSFS/pp_od_clk_voltage" 2>/dev/null || true
    echo "c" > "$GPU_SYSFS/pp_od_clk_voltage" 2>/dev/null || true
    sleep 1
    echo "auto" > "$GPU_SYSFS/power_dpm_force_performance_level" 2>/dev/null
    sleep 2
    monitor_dmesg
  done

  kill $GLMARK_PID 2>/dev/null || true
  wait $GLMARK_PID 2>/dev/null || true

  # ---- 温度监测 ----
  log "=== 温度监测 ==="
  for i in $(seq 1 10); do
    TEMP=$(cat $GPU_SYSFS/hwmon/hwmon*/temp1_input 2>/dev/null)
    SCKL=$(cat $GPU_SYSFS/pp_dpm_sclk 2>/dev/null | grep -oP '\d+Mhz' | tail -1)
    log "温度: ${TEMP}°C, 时钟: ${SCKL}"
    sleep 5
  done

  log "=== 探索性测试会话结束 ==="
  monitor_dmesg
  echo "日志文件: $LOG_FILE"
  echo "Bug报告: $BUG_LOG"
```

#### 6.2 案例二: ROCm 计算与图形交互探索

```
探索性测试发现: ROCm + Vulkan 并发访问时的内存损坏

  场景:
    测试人员正在验证 ROCm 计算和 Vulkan 渲染在同一 GPU 上并发执
    行的稳定性。探索性测试会话使用了组合交互启发式 (H5)。

  测试宪章:
    探索 ROCm HIP 计算与 Vulkan 渲染同时在 Radeon RX 7900 XTX
    上执行时的内存一致性

  执行过程:
    1. 运行 HIP vector_add 示例 (持续计算)
    2. 同时运行 vkcube (Vulkan 渲染)
    3. 两者在不同 Queues 上独立运行
    4. 测试人员观察到 vkcube 画面偶尔出现错误渲染
    5. 进一步增加压力: 在循环中分配/释放 HIP 内存

  发现:
    HIP 计算分配的 VRAM 和 Vulkan 渲染使用的 VRAM 可能重叠,
    导致 Vulkan 渲染结果出现错误。
    这是 GMC (Global Memory Controller) 地址空间管理的问题。

  影响:
    - 错误渲染 (画面 artifacts)
    - 偶尔导致 HIP 计算结果错误
    - 极端情况下 GPU 页面错误 (GCVM fault)

  根因分析:
    ROCm 和 Vulkan 使用不同的内存分配器, 两者在 VRAM 地址分配
    时没有协调机制。当 HIP 分配的缓冲区与 Vulkan 的渲染目标
    使用相同地址范围时, 导致数据损坏。

  社区讨论:
    https://github.com/ROCm/ROCm/issues/1805
    最终通过改进 HSA (Heterogeneous System Architecture) 驱动
    中的内存分区策略解决。
```

#### 6.3 案例三: 电源状态转换中的显示异常

```
探索性测试发现: P-State 转换中的显示闪烁

  场景:
    测试人员在对 RDNA 3 (RX 7600) 进行电源管理探索性测试,
    使用了 H2 (状态转换探索) 和 H5 (组合交互探索) 启发式。

  测试宪章:
    探索 RX 7600 在多种负载组合下的 P-State 转换行为,
    使用 dmesg + 视觉观察 + modetest CRC 校验

  发现:
    ┌─────────────────────────────────────────────────────────────────┐
    │ 当 GFXCLK 从低 P-State (P3, ~500MHz) 快速切换到                │
    │ 高 P-State (P0, ~2700MHz) 时, 在 DP 1.4 5K@60Hz 配置下       │
    │ 观察到约 100ms 的显示闪烁。                                    │
    │                                                                │
    │ dmesg 输出:                                                     │
    │ [ 1234.567] amdgpu: [drm] DMUB 固件通知: 时钟变化开始           │
    │ [ 1234.668] amdgpu: [drm] DMUB 固件通知: 时钟变化完成           │
    │                                                                │
    │ CRC 校验: kms_pipe_crc 在时钟变化窗口内检测到 3 帧 CRC 不一致    │
    └─────────────────────────────────────────────────────────────────┘

  根因:
    DCN (Display Core Next) 在时钟频率切换时, DC 确认未能保留足够
    的帧缓冲带宽, 导致在时钟过渡期间出现短暂的显示异常。

  修复:
    在 amdgpu_dm (Display Manager) 中的时钟变化序列添加了
    VBlank 同步等待, 确保时钟切换只在 VBlank 期间进行。

  后续测试验证:
    1. 自动测试: 在 IGT 中添加 kms_pstate_clock_change 测试
    2. 探索性测试: 在不同分辨率和刷新率组合下验证修复
    3. 回归测试: 确保 RDNA 2 (RX 6000 系列) 未受影响
```

### 9. 扩展思考

1. **探索性测试自动化**: 能否用 AI/ML 辅助探索性测试? 例如, 使用强化学习代理自动探索 GPU 驱动的参数空间, 学习哪些参数组合更容易暴露缺陷。fuzzing 工具如 syzkaller 本质上是一种自动化的探索性测试。

2. **探索性测试团队组织**: 在 AMDGPU 驱动开发中, 哪些阶段最需要经验丰富的探索性测试人员? 应该采用"嵌入式测试人员"模式 (测试人员加入开发团队) 还是 "独立测试团队" 模式? 考虑开源驱动开发的特殊性 (贡献者遍布全球)。

3. **探索性测试的量化管理**: 如何平衡探索性测试的"自由探索"天性与管理层的"可量化"需求? 除了 Bug 发现率, 还有哪些指标可以衡量探索性测试的价值 (如知识获取量、风险覆盖率)?

4. **跨硬件版本探索**: 当你只有 RDNA 3 硬件时, 如何通过探索性测试预测 RDNA 4 上可能的问题? 可以利用硬件模拟器 (如 QEMU + emulated amdgpu) 或基于驱动的代码分析进行预测性探索。

5. **安全探索性测试**: 除了功能正确性, GPU 驱动安全性的探索性测试如何开展? 例如, 探索用户空间程序是否可以通过特定 ioctl 序列导致权限提升或信息泄露。参考已有 CVE (如 CVE-2021-4237 是 amdgpu 中的权限提升漏洞)。
