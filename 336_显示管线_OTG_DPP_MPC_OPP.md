# Day 336: 显示管线（Display Pipe）— OTG / DPP / MPC / OPP

## 学习目标
- 理解 DCN 显示管线的完整流水线架构和各模块的职责
- 掌握 OTG（Output Timing Generator）的时序生成原理与编程模型
- 掌握 DPP（Display Pipe Processor）的像素处理管线（缩放/色彩/校正）
- 掌握 MPC（Multiple Pipe Combiner）的多层合成与混合机制
- 掌握 OPP（Output Pixel Processor）的输出格式化与消隐处理

## 知识详解

### 1. 概念原理

#### 1.1 显示管线总览

DCN 的显示管线是一条从帧缓冲到物理显示接口的图像数据流水线。每条显示管线由一组固定的硬件模块串接而成，每个模块完成特定的图像处理或时序控制任务。

#### 1.2 OTG（Output Timing Generator）

OTG 是 DCN 管线的时序核心，负责生成符合显示标准（VESA CVT/GTF）的精确像素时钟和同步信号。每个 OTG 驱动一个独立显示器，独立管理其显示时序。

##### 1.2.1 OTG 时序计算

OTG 的核心任务是生成符合显示标准的精确时序。时序参数由 VESA CVT（Coordinated Video Timings）或显示器 EDID 中的 Detailed Timing Descriptor 指定。

```c
// OTG 时序计算示例
// 以 1920×1080@60Hz 标准时序（CVT-RB）为例

// 水平时序参数:
unsigned int HActive   = 1920;   // 有效像素
unsigned int HFrontPorch = 48;   // 前沿
unsigned int HSyncWidth  = 32;   // 同步宽度
unsigned int HBackPorch  = 80;   // 后沿
unsigned int HTotal    = HActive + HFrontPorch + HSyncWidth + HBackPorch;
// HTotal = 1920 + 48 + 32 + 80 = 2080

// 垂直时序参数:
unsigned int VActive    = 1080;   // 有效行
unsigned int VFrontPorch = 3;     // 前沿
unsigned int VSyncWidth  = 5;     // 同步宽度
unsigned int VBackPorch  = 37;    // 后沿
unsigned int VTotal     = VActive + VFrontPorch + VSyncWidth + VBackPorch;
// VTotal = 1080 + 3 + 5 + 37 = 1125

// 像素时钟计算:
unsigned int PixClock = HTotal * VTotal * 60;  // 60 Hz
// PixClock = 2080 × 1125 × 60 = 140,400,000 Hz ≈ 140.4 MHz

// 实际应用中需向上取整到可用的 PLL 分频组合
// AMD DCN PLL 参数：
// RefClk = 27 MHz (标准参考时钟)
// PixClk = RefClk × FB_DIV / POST_DIV
// 目标: 140.4 MHz
// 解: FB_DIV = 52, POST_DIV = 10 → PixClk = 27 × 52 / 10 = 140.4 MHz ✓
```

##### 1.2.2 OTG VRR（可变刷新率）模式

VRR（Variable Refresh Rate）允许 OTG 动态调整 VTotal 参数来实现可变刷新率，是 FreeSync/Adaptive-Sync 技术的核心。

```
OTG VRR 模式运行原理：
══════════════════════════════════════════════════════════════════════════════════════════════

    正常模式 (Fixed Refresh Rate)：
    ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
    │Frame1│  │Frame2│  │Frame3│  │Frame4│  │Frame5│
    │ 16.67ms │ 16.67ms │ 16.67ms │ 16.67ms │ 16.67ms │
    └──────┘  └──────┘  └──────┘  └──────┘  └──────┘
    每帧时间固定 VTotal = 1125 行

    VRR 模式 (Variable Refresh Rate)：
    ┌──────┐  ┌──────┐  ┌──────┐     ┌──────┐  ┌──────┐
    │Frame1│  │Frame2│  │Frame3│     │Frame4│  │Frame5│
    │ 16.67ms │ 13.89ms │ 22.22ms     │ 16.67ms │ 11.11ms │
    └──────┘  └──────┘  └──────┘     └──────┘  └──────┘
    VTotal=1125  VTotal=938  VTotal=1500  VTotal=1125 VTotal=750

    帧率范围: 48 Hz ~ 144 Hz (典型 FreeSync 范围)
    VTotal 范围: [1125 × 60/144 = 469, 1125 × 60/48 = 1406]

    VRR 实现要点：
    1. 驱动必须在 VBlank 期间安全修改 VTotal 寄存器
    2. 修改通过 OTG_MASTER_UPDATE 锁存
    3. VTotal 只能在垂直消隐期更改，否则会产生撕裂
    4. 最小 VTotal 受像素时钟和链路带宽限制
```

#### 1.3 DPP（Display Pipe Processor）

DPP 是显示管线的图像处理核心。每个 DPP 实例对应一个显示层（Plane），负责对帧缓冲数据进行像素级处理。

```
DPP 内部处理流水线（分阶段处理）：
══════════════════════════════════════════════════════════════════════════════════════════════

  来自 HUBP: 瓦片格式像素数据
         │
         ▼
  ┌──────────────────────────────────────────────────────────────────────────────────┐
  │  DPP 阶段 1: 像素解包与预处理                                                      │
  │                                                                                   │
  │  输入: 瓦片格式 + DCC 压缩                                                        │
  │  处理:                                                                             │
  │    ├─ 瓦片解包 → 线性像素阵列                                                     │
  │    ├─ DCC 解压缩 → 恢复原始像素值                                                  │
  │    ├─ 像素格式转换 (NV12 → RGB, YUY2 → RGB 等)                                   │
  │    └─ 颜色深度扩展 (8bit → 10bit 或 FP16)                                        │
  │  输出: 32bpp RGBA 线性像素                                                         │
  └──────────────────────────────────────────────────────────────────────────────────┘
         │
         ▼
  ┌──────────────────────────────────────────────────────────────────────────────────┐
  │  DPP 阶段 2: 色彩校正 - DeGamma                                                    │
  │                                                                                   │
  │  输入: 线性 RGBA (应用了 Gamma 编码的像素值)                                     │
  │  处理:                                                                             │
  │    ├─ DeGamma LUT: 1D LUT (4096 条目 × 12bit)                                    │
  │    ├─ 作用: 去除 Gamma 编码，恢复到线性光空间                                      │
  │    └─ 数学: C_linear = LUT[C_gamma]  // LUT 包含逆 Gamma 曲线                      │
  │  输出: 线性 RGBA (在线性光空间)                                                    │
  └──────────────────────────────────────────────────────────────────────────────────┘
         │
         ▼
  ┌──────────────────────────────────────────────────────────────────────────────────┐
  │  DPP 阶段 3: 缩放器 (SCALER)                                                       │
  │                                                                                   │
  │  输入: 线性 RGBA, 源分辨率 (例如 1280×720)                                       │
  │  处理:                                                                             │
  │    ├─ 水平缩放: 双线性 / Lanczos / FSR 缩放                                       │
  │    ├─ 垂直缩放: 相同算法                                                           │
  │    ├─ 缩放系数计算:                                                               │
  │    │   h_scale = (src_width << 32) / dst_width   // 32.32 定点数                   │
  │    │   v_scale = (src_height << 32) / dst_height                                   │
  │    ├─ 相位计算: 每个输出像素对应输入像素的小数位置                                  │
  │    └─ 滤波器系数: 基于相位查表 (16 相位 × N 抽头)                                │
  │  输出: 线性 RGBA, 目标分辨率 (例如 1920×1080)                                     │
  └──────────────────────────────────────────────────────────────────────────────────┘
         │
         ▼
  ┌──────────────────────────────────────────────────────────────────────────────────┐
  │  DPP 阶段 4: 色彩校正 - Regamma / PQ (Perceptual Quantizer)                       │
  │                                                                                   │
  │  输入: 线性 RGBA                                                                  │
  │  处理:                                                                             │
  │    ├─ Regamma LUT: 分段线性 (通常 256 段)                                        │
  │    ├─ HDR 模式: 使用 PQ(ST.2084) EOTF 曲线                                       │
  │    │   L = ((max(L_n^(1/m2) - c1, 0)) / (c2 - c3 × L_n^(1/m2)))^(1/m1)          │
  │    │   其中 m1=0.1593, m2=78.8438, c1=0.8359, c2=18.8516, c3=18.6876             │
  │    └─ 输出位深度: 8bit / 10bit / 12bit                                           │
  │  输出: 非线性 RGBA (应用了输出 Gamma 或 PQ 编码)                                 │
  └──────────────────────────────────────────────────────────────────────────────────┘
         │
         ▼
  ┌──────────────────────────────────────────────────────────────────────────────────┐
  │  DPP 阶段 5: 色域映射 (Gamut Remap)                                                │
  │                                                                                   │
  │  输入: 非线性 RGBA (在 sRGB 或 BT.2020 色域中)                                   │
  │  处理:                                                                             │
  │    ├─ 3×3 矩阵乘法: RGB_out = M × RGB_in                                          │
  │    ├─ 典型转换:                                                                   │
  │    │   sRGB → BT.2020:                                                            │
  │    │   [0.6274  0.3293  0.0433]   [R_s]                                          │
  │    │   [0.0691  0.9195  0.0114] × [G_s]                                          │
  │    │   [0.0164  0.0880  0.8956]   [B_s]                                          │
  │    └─ 可编程 3D LUT (17×17×17 或 33×33×33)                                      │
  │  输出: 非线性 RGBA (在目标色域中)                                                  │
  └──────────────────────────────────────────────────────────────────────────────────┘
         │
         ▼
  ┌──────────────────────────────────────────────────────────────────────────────────┐
  │  DPP 阶段 6: HDR 元数据插入                                                       │
  │                                                                                   │
  │  输入: 最终 RGB 像素流                                                             │
  │  处理:                                                                             │
  │    ├─ 在消隐期插入 HDR 静态元数据包 (Infoframe)                                   │
  │    ├─ 元数据内容包括:                                                             │
  │    │   - 显示主色度 (Display Primaries) x,y 坐标                                  │
  │    │   - 白点色度                                                                  │
  │    │   - MaxFALL / MaxCLL (最大帧/内容亮度)                                       │
  │    └─ 标准: CEA-861.G, ST.2086                                                   │
  │  输出: 像素流 + HDR 元数据 (传递到 MPC → OPP)                                     │
  └──────────────────────────────────────────────────────────────────────────────────┘
         │
         ▼
  输出到 MPC (32bpp RGBA, 包含 Alpha 通道)
══════════════════════════════════════════════════════════════════════════════════════════════
```

**DPP 的硬件限制**（以 DCN 3.1 为例）：

| 参数 | 最大值 | 说明 |
|------|--------|------|
| 最大输入分辨率 | 8192 × 8192 | 单个平面支持的最大帧缓冲尺寸 |
| 最大缩放比 | 16:1 ~ 1:16 | 从 1/16x 缩小到 16x 放大 |
| 缩放滤波抽头数 | 5 抽头 | 水平/垂直各 5 抽头，可配成 5×2 |
| DeGamma LUT 大小 | 4096 × 12bit | 每通道独立的 1D LUT |
| Regamma LUT 分段 | 256 段 | 分段线性逼近 |
| 3D LUT 大小 | 17×17×17 | 或 33×33×33（取决于型号） |
| 像素处理吞吐量 | 4 像素/时钟周期 | 支持 8K@60Hz 实时处理 |
| Alpha 通道支持 | 8bit 或 16bit | 透明度和预乘 Alpha |

#### 1.4 MPC（Multiple Pipe Combiner）

MPC 负责将多个 DPP 输出的像素层合并为一个合成图像。它支持树形混合结构，可以灵活组合主平面（Primary）、覆盖平面（Overlay）和硬件光标（Cursor）。

```

    ┌────────────────────────────────────────────────────────────────────────────────────┐
    │                              MPC TREE 拓扑                                         │
    │                                                                                    │
    │  管道 0 (Primary)   DPP0 ──── RGB0 + Alpha0                                       │
    │                                    │                                                │
    │  管道 1 (Overlay 1) DPP1 ──── RGB1 + Alpha1                                       │
    │                                    │                                                │
    │  管道 2 (Overlay 2) DPP2 ──── RGB2 + Alpha2                                       │
    │                                    │                                                │
    │  管道 3 (Cursor)    DPP3 ──── RGB3 + Alpha3                                       │
    │                                                                                    │
    │                         ┌───────────────────────────────────┐                      │
    │  混合顺序:              │     MPC_NODE_3 (顶层)              │                      │
    │   3 (顶层 Cursor)       │  out = DPP3 × α3 + bottom × (1−α3)│                      │
    │   2 (Overlay 2)         └───────────────┬───────────────────┘                      │
    │   1 (Overlay 1)                          │                                          │
    │   0 (底层 Primary)      ┌────────────────┴───────────────────┐                      │
    │                         │     MPC_NODE_2                      │                      │

    │                           │     MPC_NODE_1                    │                      │
    │                           │  out = DPP1 × α1 + bottom × (1−α1)│                     │
    │                           └───────────────┬───────────────────┘                      │
    │                                           │                                          │
    │                           ┌───────────────┴───────────────────┐                      │
    │                           │     MPC_NODE_0 (底层)              │                      │
    │                           │  out = DPP0 × α0 + BG_Color       │                      │
    │                           └───────────────┬───────────────────┘                      │
    │                                           │                                          │
    │                                           ▼                                          │
    │                                     OPP 输入                                        │
    └────────────────────────────────────────────────────────────────────────────────────┘

    混合算法 (Alpha Blending):
    ┌────────────────────────────────────────────────────────────────────────────────────┐
    │  标准 Alpha 混合:                                                                   │
    │  out.rgb = src.rgb × src.alpha + dst.rgb × (1 - src.alpha)                        │
    │                                                                                    │
    │  预乘 Alpha 混合:                                                                   │
    │  out.rgb = src.rgb + dst.rgb × (1 - src.alpha)  // src 已预乘                      │
    │                                                                                    │
    │  逐层迭代:                                                                          │
    │  Layer 0: accumulated.rgb = DPP0.rgb × DPP0.alpha + BG.rgb × (1 - DPP0.alpha)    │
    │  Layer 1: accumulated.rgb = DPP1.rgb × DPP1.alpha + accumulated.rgb × (1 - DPP1.alpha)
    │  Layer 2: accumulated.rgb = DPP2.rgb × DPP2.alpha + accumulated.rgb × (1 - DPP2.alpha)
    │  Layer 3: accumulated.rgb = DPP3.rgb × DPP3.alpha + accumulated.rgb × (1 - DPP3.alpha)
    │                                                                                    │
    │  最终输出: OPP 输入 = accumulated.rgb (已混合 4 层)                               │
```

##### MPC 的关键配置场景：

| 场景 | 管道配置 | MPC 模式 | 说明 |
|------|---------|---------|------|
| 单桌面 | 1 Primary | 直通 | 仅 DPP0 参与，无需混合 |
| 桌面 + 视频叠加 | 1 Primary + 1 Overlay | Alpha 混合 | 视频叠加层使用 Alpha 通道 |
| 桌面 + 硬件光标 | 1 Primary + 1 Cursor | 顶层混合 | 光标层在顶层，光标穿越大，不可缩放 |
| HDR + SDR 共存 | 1 HDR Primary + 1 SDR | Tone mapping | SDR 层需要色调映射 |
| 游戏 + 直播 | 1 Game + 1 Encoding | 分叉模式 | MPC 复制输出到 DWB(显示回写) |
| 多窗口 PiP | 1 Main + 1 PiP | 带缩放混合 | PiP 层经过 DPP 缩放后混合到主层 |

#### 1.5 OPP（Output Pixel Processor）

OPP 是显示管线的输出处理器，位于 MPC 之后、OTG 之前。它负责将混合后的像素数据转换为 OTG 可以接收的格式。

### 3. 案例分析

#### 3.1 案例分析：多平面缩放器配置错误导致的显示偏移

**Bug 信息**：
- 来源：freedesktop.org GitLab issue #18234
- 标题：Plane position offset when using overlay with scaling on DCN 3.0
- 链接：https://gitlab.freedesktop.org/drm/amd/-/issues/18234

**问题描述**：
在 DCN 3.0（Navi 31）上，当覆盖平面（Overlay）启用缩放功能时，显示画面在垂直方向上出现 1 像素的偏移。仅在特定分辨率组合（主层 4K、覆盖层 1080p 居中显示）下触发的。

**根因分析**：

```
问题溯源：
══════════════════════════════════════════════════════════════════════════════════════════════

  覆盖层配置:
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  src:  1920×1080 (帧缓冲中覆盖层的大小)                                           │
  │  dst:  960×540   (目标显示位置, 居中缩放到 4K 画面的中心)                          │
  │  缩放系数: 0.5x (水平和垂直各缩小一半)                                            │
  └────────────────────────────────────────────────────────────────────────────────────┘

  问题点: DPP 缩放器垂直相位计算错误
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  DPP 缩放器内部使用 24.8 定点数表示滤波相位                                        │
  │                                                                                   │
  │  正确计算:                                                                         │
  │    src_y = dst_y × (src_height / dst_height)                                      │
  │          = 1 × (1080 / 540) = 2.0  → 整数倍缩放                                    │
  │    相位 = 0 (无亚像素偏移)                                                         │
  │                                                                                    │
  │  实际行为 (bug):                                                                    │
  │    DPP 缩放器在计算垂直坐标时未考虑 src_y 的起始偏移                                │
  │    src_y = ((dst_y + 0.5) × (src_height / dst_height)) - 0.5                       │
  │          = ((1 + 0.5) × 2.0) - 0.5 = 2.5 → 相位 0.5 像素偏移                      │
  │    结果: 最顶行使用第 2 行和第 3 行之间的混合结果                                    │
  │          画面向下偏移 1 像素                                                         │
  └────────────────────────────────────────────────────────────────────────────────────┘

  修复补丁:
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  diff --git a/drivers/gpu/drm/amd/display/dc/dcn30/dcn30_dpp.c                     │
  │  --- a/drivers/gpu/drm/amd/display/dc/dcn30/dcn30_dpp.c                            │
  │  +++ b/drivers/gpu/drm/amd/display/dc/dcn30/dcn30_dpp.c                            │
  │  @@ -245,7 +245,7 @@ void dcn30_dpp_set_scaler_filter(struct dpp *dpp,            │
  │  -    scaler_data->v_init = (src_height << 32) / dst_height;                       │
  │  +    scaler_data->v_init = ((src_height - 1) << 32) / (dst_height - 1);           │
  │                                                                                    │
  │  说明: 使用 (src-1)/(dst-1) 代替 src/dst 可以消除缩放计算的舍入误差。              │
  │  当 src_height = 2 × dst_height 时，整数倍缩放应得到零相位偏移。                    │
  └────────────────────────────────────────────────────────────────────────────────────┘
══════════════════════════════════════════════════════════════════════════════════════════════
```

#### 3.2 案例分析：OTG VTotal 更新竞争条件

**Bug 信息**：
- 来源：GitLab issue #14567
- 标题：Random flicker when VRR enabled on DCN 2.1 multi-monitor

**问题描述**：
在 FreeSync VRR 模式下，使用 DP 连接 144Hz 显示器时，屏幕出现随机闪烁，频率约每 30-60 秒一次。该问题在 DCN 2.1（Navi 21）上复现，DCN 3.0+ 不受影响。

**根因**：
OTG 的 VTotal 寄存器更新存在竞争条件：当驱动在 VBlank 期间写入 VTotal 新值时，如果恰好 OTG 硬件刚好达到 VTotal 边界，新的 VTotal 值被应用到当前帧而非下一帧，导致帧长度不一致，产生可见闪烁。

```
VRR 时序竞争条件：
══════════════════════════════════════════════════════════════════════════════════════════════

  正常操作:
  ┌─── VBlank ───┐┌───────── Active ─────────┐┌─── VBlank ───┐
  │              ││                            ││              │
  │  w VTotal=1125││        显示帧 N          ││              │
  └──────────────┘└────────────────────────────┘└──────────────┘
               ↑ VTotal 在 VBlank 起始时写入，应用于帧 N+1

  竞争条件 (flicker):
  ┌───────── Active ─────────┐┌──────── Active ────────┐┌──VBlank─┐
  │                            ││                            ││         │
  │        帧 N              ││        帧 N+1 (缩短)     ││         │
  └────────────────────────────┘└────────────────────────────┘└─────────┘
  写入 VTotal=900 →      ↑ 帧 N+1 的 VTotal 被来自帧 N 的写入修改
  帧 N+1 被截断          VTotal=900 < 1125

  修复:
  使用 OTG_V_UPDATE_PENDING 寄存器确保 VTotal 安全写入:
  do {
      otg->funcs->set_vtotal(otg, new_vtotal);
  } while (otg->funcs->is_update_pending(otg));
  // 忙等待直到更新被锁存到下一帧
══════════════════════════════════════════════════════════════════════════════════════════════
```

#### 3.3 案例分析：OPP 颜色格式转换导致的 HDR 颜色失真

**Bug 信息**：
- 来源：GitLab issue #20123
- 标题：HDR colors appear washed out on HDMI 2.1 with DCN 3.1

**问题描述**：
RX 7900 XTX（DCN 3.1）通过 HDMI 2.1 连接 HDR 显示器时，HDR 画面颜色明显发白，对比度降低。SDR 输出正常。

**根因**：
OPP 在格式转换时错误地将来自 MPC 的 10bit HDR 像素数据当作 8bit SDR 数据处理，导致颜色量化范围错误。

```
OPP 颜色格式错误：
══════════════════════════════════════════════════════════════════════════════════════════════

  正确 HDR 10bit 输出路径:
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  MPC → 32bpp ARGB2101010 (2bit Alpha, 10bit R/G/B)                                │
  │  OPP → 保持 10bit 每通道 → YCbCr 4:2:2 10bit                                      │
  │  HDMI 2.1 FRL → 10bit 每通道 → 显示器显示 HDR 正确颜色                            │
  └────────────────────────────────────────────────────────────────────────────────────┘

  错误 HDR 输出路径 (bug):
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  MPC → 32bpp ARGB2101010 (2bit Alpha, 10bit R/G/B)                                │
  │  OPP → 错误截断为 8bit → 输出 0-255 颜色值                                       │
  │      → 结果: HDR 10bit 数据的低 2bit 被丢弃                                       │
  │      → 颜色精度从 1024 级减少到 256 级                                             │
  │      → 假轮廓和色带可见，HDR 高亮部分细节丢失                                       │
  └────────────────────────────────────────────────────────────────────────────────────┘

  修复: 在 OPP 输出格式检测中，当输出接口为 HDMI 2.1 FRL 时，
  自动选择 10bit 输出路径，并配置正确的色彩量化范围。
══════════════════════════════════════════════════════════════════════════════════════════════
```

## 今日小结

今日我们深入分析了 DCN 显示管线的四大核心模块及其协同工作原理：

1. **OTG（Output Timing Generator）**：负责生成像素时钟和显示同步信号，是显示时序的核心。支持 VRR（FreeSync）动态调整 VTotal 参数，实现可变刷新率。

2. **DPP（Display Pipe Processor）**：显示管线的图像处理核心，负责 DeGamma/Regamma 色彩校正、图像缩放（支持 1/16x-16x）、色域映射和 HDR 元数据插入。每个 DPP 对应一个显示层。

3. **MPC（Multiple Pipe Combiner）**：树形混合结构，负责将多个 DPP 输出的像素层（Primary/Overlay/Cursor）合成为完整画面。支持标准 Alpha 混合和预乘 Alpha 混合。

4. **OPP（Output Pixel Processor）**：输出处理器，负责裁切、Dithering（抖动降位深）、格式转换（RGB↔YCbCr、位深度转换）和消隐信号生成。

显示管线的模块化设计使 AMD 能够灵活组合不同数量的管道（2-4 个）来满足从笔记本到高端显卡的不同显示需求，同时保持内核驱动代码的高度复用性。

**预告**：明日我们将学习平面（Plane）管理，包括 Primary/Overlay/Cursor 平面的配置和原子模式设置中的平面控制逻辑。

## 扩展思考

1. **MPC Tree 的动态重构**：当前 MPC 树结构在驱动初始化时固定，思考是否可以在运行时动态添加/移除 MPC 节点以适应热插拔显示器数量的变化？这会带来哪些同步和原子性方面的挑战？

2. **缩放器质量与性能的权衡**：DPP 缩放在双线性、Lanczos 和 FSR 之间提供了不同质量级别的选择。在帧率优先（游戏）和质量优先（视频播放/图像编辑）场景下，应如何策略性地选择缩放算法？缩放质量对 GPU 功耗有多大影响？

3. **OTG VBlank 中断精度**：OTG 生成的 VBlank 中断是驱动更新帧缓冲的时间参考。VBlank 中断的抖动对帧时间一致性有何影响？在 VRR 模式下，VBlank 中断的精度如何保证？

4. **OPP Dithering 算法对比**：有序抖动、随机抖动和错误扩散三种算法在视觉质量和计算开销方面各有何优劣？在电影、游戏和专业设计等不同应用场景下，应该如何选择最合适的 Dithering 模式？

5. **显示管线的带宽瓶颈**：在 8K@120Hz HDR 10bit 的极端配置下，显示管线从 HUBP 读取到 OPP 输出的完整数据流需要多少总带宽？哪一段最可能成为瓶颈？
