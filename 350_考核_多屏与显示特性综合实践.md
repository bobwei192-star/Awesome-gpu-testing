#### 1.4 MST 带宽计算参考

DisplayPort MST 的带宽管理是理解多屏配置的核心。以下为 MST 带宽计算的方法和示例：

```
MST 带宽管理核心概念：
┌─────────────────────────────────────────────────────────────────┐
│  DP 链路总带宽 = Lane Count × Link Rate × 8/10 编码开销         │
│                                                                  │
│  常见组合:                                                       │
│  HBR2 (5.4 Gbps) × 4 lanes = 21.6 Gbps → 有效 17.28 Gbps       │
│  HBR3 (8.1 Gbps) × 4 lanes = 32.4 Gbps → 有效 25.92 Gbps       │
│  UHBR10 (10 Gbps) × 4 lanes = 40 Gbps → 无 8b/10b (128b/132b)  │
│                                                                  │
│  每个显示器所需的 PBN (Payload Bandwidth Number):                │
│  PBN = pixel_clock × bpp × 54 / (64 × 8 × 1000)                │
│                                                                  │
│  示例 - 4K@60 10bpc HDR:                                        │
│  pixel_clock = 533.25 MHz                                       │
│  bpp = 30 (10bpc × 3)                                          │
│  PBN = 533.25 × 30 × 54 / (64 × 8 × 1000) = 168.8              │
│                                                                  │
│  Total Payload Bandwidth = 25.92 Gbps (HBR3 4L)                │
│  Max PBN available = 25.92 × 54 / 64 × 1000 = 2187            │
│                                                                  │
│  三个 4K@60 显示器各需 ~169 PBN, 共需 ~507 PBN                  │
│  远低于 2187 PBN 的总限制                                        │
└─────────────────────────────────────────────────────────────────┘
```

#### 1.6 FreeSync VRR 寄存器级编程参考

FreeSync 的驱动实现核心是动态编程 CRTC 的 V_TOTAL 寄存器。以下为 AMDGPU DCN 中的相关寄存器编程流程：

```
AMDGPU FreeSync VRR 寄存器编程流程：

  1. 用户空间设置 vrr_enabled 属性
     ↓
  2. DM 层调用 amdgpu_dm_connector_atomic_set_property()
     设置 connector->state->vrr_enabled = true
     ↓
  3. DC 层在 commit 时检查 vrr_enabled
     ↓
  4. DCN 硬件寄存器编程:
     ┌───────────────────────────────────────┐
     │ OTG_V_TOTAL (CRTC Vertical Total)     │
     │   - 最小值: 对应最高刷新率            │
     │   - 最大值: 对应最低刷新率            │
     │   - 在 VBlank 期间动态调整            │
     ├───────────────────────────────────────┤
     │ OTG_V_TOTAL_MIN_SEL                    │
     │   - 0: 使用固定 V_TOTAL               │
     │   - 1: VRR 模式，可变 V_TOTAL         │
     ├───────────────────────────────────────┤
     │ OTG_V_TOTAL_MAX                       │
     │   - 设置 V_TOTAL 的上限               │
     ├───────────────────────────────────────┤
     │ OTG_DRR_V_TOTAL                       │
     │   - 在 DRR (Dynamic Refresh Rate) 模式下│
     │     由硬件自动调整                     │
     └───────────────────────────────────────┘

  VRR 刷新率计算公式:
    refresh_rate = pixel_clock / (htotal × vtotal)
    V_TOTAL 变化范围:
      min_VTOTAL = pixel_clock / (htotal × max_refresh_rate)
      max_VTOTAL = pixel_clock / (htotal × min_refresh_rate)

  示例 (4K@60Hz, VRR range 48-60Hz):
    pixel_clock = 533.25 MHz
    htotal = 4000
    native_VTOTAL = 2222 (60Hz)
    min_VTOTAL = 2222 (60Hz, 最大值时 VTOTAL 最小)
    max_VTOTAL = 2222 × 60/48 = 2778 (48Hz)
    
  关键内核源码路径:
  drivers/gpu/drm/amd/display/modules/freesync/freesync.c
    - mod_freesync_build_variable_info()
    - mod_freesync_get_min_max()
  drivers/gpu/drm/amd/display/dc/dcn20/dcn20_hwseq.c
    - dcn20_set_drr()
```

#### 1.7 HDR 色彩体积与 PQ 曲线参考

HDR 的核心是 SMPTE ST2084 (PQ) 感知量化曲线和 BT.2020 宽色域。以下为 HDR 色彩管理的技术参考：

```
PQ (Perceptual Quantizer) 曲线:
  ST2084 定义的 EOTF (Electro-Optical Transfer Function):
  
  L = ((max((N^(1/m2) - c1), 0)) / (c2 - c3 × N^(1/m2)))^(1/m1)
  
  其中:
  N = 归一化的数字编码值 (0-1)
  L = 亮度 (cd/m^2)
  m1 = 2610/16384 = 0.1593...
  m2 = 2523/4096 × 128 = 78.843...
  c1 = 3424/4096 = 0.8359...
  c2 = 2413/4096 × 32 = 18.8516...
  c3 = 2392/4096 × 32 = 18.6875...

  PQ 曲线的亮度编码范围: 0.0001 到 10000 cd/m^2

BT.2020 色域 vs BT.709 色域 (CIE 1931 xy):
┌───────────────────────────────────────────────────────┐
│                     BT.2020                           │
│     绿 (0.170, 0.797) ───────────────── 红 (0.708, 0.292)│
│                          \                  /         │
│                           \    BT.709      /          │
│                            \  绿 (0.300,   /          │
│                             \   0.600)    /           │
│                              \          /            │
│                               \        /             │
│                                \      /              │
│                                 \    /               │
│                                  \  /                │
│                            白 (0.3127, 0.3290)       │
│                                  /  \                │
│                   蓝 (0.150, 0.060)   \              │
│                   BT.709 蓝 (0.150,   \             │
│                   0.060) 重合            \           │
│                             蓝 (0.131, 0.046)        │
│                             (BT.2020)                │
└───────────────────────────────────────────────────────┘

HDR Static Metadata Type1 (SMPTE ST2086) 结构:
struct hdr_static_metadata {
    uint16_t display_primaries[3][2];  // 红/绿/蓝 x,y (各 16-bit)
    uint16_t white_point[2];           // 白点 x,y
    uint16_t max_display_mastering_luminance;  // 最大亮度 (0.0001 cd/m^2)
    uint16_t min_display_mastering_luminance;  // 最小亮度
    uint16_t max_content_light_level;          // MaxCLL
    uint16_t max_frame_average_light_level;    // MaxFALL
};

DCN 硬件中的 HDR 处理管线:
  1. Input:    RGB 线性光值 → Gamut Remap Matrix (BT.2020 ↔ BT.709)
  2. Transfer: 1D LUT (PQ/HLG/SRGB 曲线) 或可编程分段线性
  3. Mix:      3×3 色彩矩阵 (亮度和色度调整)
  4. Output:   3D LUT (可选, 用于高级色彩分级)
  5. Dither:   Temporal dithering 减少 banding (10bpc → 8bpc)
```

```bash
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <fcntl.h>
#include <sys/mman.h>
#include <xf86drm.h>
#include <xf86drmMode.h>

int main(int argc, char **argv)
{
    int fd;
    drmVBlank vbl;
    int crtc_id = atoi(argv[1]);
    int num_samples = 100;
    int64_t prev_seq, cur_seq;
    double refresh_rate;

    fd = drmOpen("amdgpu", NULL);
    if (fd < 0) {
        printf("Failed to open amdgpu DRM\n");
        return 1;
    }

    printf("Collecting %d VBlank samples for CRTC %d...\n",
           num_samples, crtc_id);

        vbl.request.sequence = 1;
        vbl.request.signal = 0;
        vbl.request.crtc = crtc_id;

            printf("VBlank wait failed at sample %d\n", i);

                       i, (long)seq_diff,
                       (long long)vbl.reply.tval_usec);
            }
        }
        prev_seq = cur_seq;
        usleep(1000);  /* 1ms delay between samples */
    }

    drmClose(fd);
    return 0;
```

```bash
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <fcntl.h>
#include <xf86drm.h>
#include <xf86drmMode.h>

int main(int argc, char **argv)
{
    int fd;
    drmModeRes *res;
    drmModeAtomicReq *req;
    uint32_t crtc_id, conn_id;
    uint32_t vrr_enabled_prop_id = 0;
    int i;

    fd = drmOpen("amdgpu", NULL);
    res = drmModeGetResources(fd);

    /* 查找第一个连接器的 VRR 属性 */
    for (i = 0; i < res->count_connectors; i++) {
        drmModeConnector *conn = drmModeGetConnector(fd, res->connectors[i]);
        drmModeObjectProperties *props =
            drmModeObjectGetProperties(fd, conn->connector_id,
                                        DRM_MODE_OBJECT_CONNECTOR);
        for (int j = 0; j < props->count_props; j++) {
            drmModePropertyRes *prop =
                drmModeGetProperty(fd, props->props[j]);
            if (prop && strcmp(prop->name, "vrr_enabled") == 0) {

                conn_id = conn->connector_id;

                       conn_id, vrr_enabled_prop_id);
            }
            drmModeFreeProperty(prop);
        }
        drmModeFreeObjectProperties(props);
        drmModeFreeConnector(conn);
        if (vrr_enabled_prop_id) break;
    }

    /* 通过 atomic 设置 vrr_enabled */
    req = drmModeAtomicAlloc();
    drmModeAtomicAddProperty(req, conn_id, DRM_MODE_OBJECT_CONNECTOR,
                             vrr_enabled_prop_id, 1);

    /* TEST_ONLY 提交以检查 VRR 是否被接收 */
    int ret = drmModeAtomicCommit(fd, req,
                                   DRM_MODE_ATOMIC_TEST_ONLY |
                                   DRM_MODE_ATOMIC_ALLOW_MODESET,
                                   NULL);
    if (ret == 0) {
        printf("VRR TEST_ONLY commit succeeded\n");
    } else {
        printf("VRR TEST_ONLY commit failed: %s\n", strerror(-ret));
    }

    drmModeAtomicFree(req);
    drmModeFreeResources(res);
    drmClose(fd);
    return 0;
}
```

```bash

$ cat > set_hdr.c << 'EOF'
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <fcntl.h>
#include <unistd.h>
#include <xf86drm.h>
#include <xf86drmMode.h>

int main(int argc, char **argv)
{
    int fd;
    drmModeConnector *conn;
    drmModeObjectProperties *props;
    drmModePropertyRes *prop;
    int conn_id, i;
    uint32_t hdr_prop_id = 0;
    struct hdr_output_metadata hdr_meta;

    if (argc < 2) {
        printf("Usage: %s <connector_id>\n", argv[0]);
        return 1;
    }

    fd = drmOpen("amdgpu", NULL);
    conn_id = atoi(argv[1]);

    /* 获取连接器属性 */
    props = drmModeObjectGetProperties(fd, conn_id,
                                        DRM_MODE_OBJECT_CONNECTOR);
    for (i = 0; i < props->count_props; i++) {
        prop = drmModeGetProperty(fd, props->props[i]);
        if (prop && strcmp(prop->name, "HDR_OUTPUT_METADATA") == 0) {
            hdr_prop_id = prop->prop_id;
            printf("Found HDR_OUTPUT_METADATA property, id=%d\n",
                   hdr_prop_id);
        }
        if (prop && strcmp(prop->name, "Colorspace") == 0) {
            printf("Found Colorspace property, id=%d, val=%lld\n",
                   prop->prop_id, (long long)props->prop_values[i]);
        }
        drmModeFreeProperty(prop);
    }

    /* 构造 SMPTE ST2086 HDR 元数据 */
    memset(&hdr_meta, 0, sizeof(hdr_meta));
    hdr_meta.metadata_type = 0;  /* Static Metadata Type 1 */
    /* 设置 Display Primaries (x,y 各 16-bit) */
    hdr_meta.metadata.mdm.static_metadata_type1.display_primaries[0][0] = 34000;  /* Red x */
    hdr_meta.metadata.mdm.static_metadata_type1.display_primaries[0][1] = 16000;  /* Red y */
    hdr_meta.metadata.mdm.static_metadata_type1.display_primaries[1][0] = 13250;  /* Green x */
    hdr_meta.metadata.mdm.static_metadata_type1.display_primaries[1][1] = 34500;  /* Green y */
    hdr_meta.metadata.mdm.static_metadata_type1.display_primaries[2][0] = 7500;   /* Blue x */
    hdr_meta.metadata.mdm.static_metadata_type1.display_primaries[2][1] = 3000;   /* Blue y */
    /* 设置 White Point */
    hdr_meta.metadata.mdm.static_metadata_type1.white_point[0] = 31270;  /* White x */
    hdr_meta.metadata.mdm.static_metadata_type1.white_point[1] = 32900;  /* White y */
    /* 设置亮度范围 (cd/m^2 * 10000) */
    hdr_meta.metadata.mdm.static_metadata_type1.max_display_mastering_luminance = 6000000;  /* 600 cd/m^2 */
    hdr_meta.metadata.mdm.static_metadata_type1.min_display_mastering_luminance = 50;       /* 0.005 cd/m^2 */
    /* 设置 Maximum Content Light Level (MCLL) */
    hdr_meta.metadata.mdm.static_metadata_type1.max_content_light_level = 10000000;
    /* 设置 Maximum Frame-Average Light Level (MFALL) */
    hdr_meta.metadata.mdm.static_metadata_type1.max_frame_average_light_level = 4000000;

    /* 使用 atomic 属性设置 HDR 元数据 */
    drmModeAtomicReq *req = drmModeAtomicAlloc();
    drmModeAtomicAddProperty(req, conn_id, DRM_MODE_OBJECT_CONNECTOR,
                             hdr_prop_id, (uint64_t)&hdr_meta);
    drmModeAtomicCommit(fd, req, DRM_MODE_ATOMIC_ALLOW_MODESET, NULL);
    drmModeAtomicFree(req);

    printf("HDR metadata set successfully\n");

    drmModeFreeObjectProperties(props);
    drmClose(fd);
    return 0;
}
EOF

$ gcc -o set_hdr set_hdr.c -ldrm
$ ./set_hdr <connector_id>
```

**Bug 标题**：AMDGPU MST 多屏扩展时，第三个显示器间歇性黑屏

**Bug 报告摘要**：

```
环境：
- GPU: AMD Navi 21 (RX 6800 XT)
- 显示器: 3× Dell U2723QE (4K@60Hz) 通过 DP MST 菊花链连接
- 驱动: Linux 6.5 (amdgpu DC enabled)
- 桌面环境: GNOME 45 (Wayland)

现象：
1. 系统启动后三个显示器都正常点亮
2. 登录进入 GNOME 后约 5-10 分钟，第三个显示器（菊花链末端）黑屏
3. 将第三个显示器重新插拔后恢复，但几分钟后再次黑屏
4. 前两个显示器始终正常工作
5. 切换到 X11 会话后问题不再复现

内核日志：
[  245.678901] [drm:dc_link_detect [amdgpu]] *ERROR* MST link bandwidth insufficient
[  245.679012] [drm:dm_helpers_dp_mst_send_allocate_payload [amdgpu]] *ERROR*
    Failed to allocate MST payload for connector DP-3
[  510.123456] [drm:dc_link_detect [amdgpu]] *ERROR* MST payload table full
[  510.123567] [drm:dm_helpers_dp_mst_send_allocate_payload [amdgpu]] *ERROR*
    Failed to allocate MST payload: no available payload ID

相关寄存器快照（读取 DPCD）：

DPCD Address 0x00201 (MST Status):      0x13  (MST enabled, ACT sent)
DPCD Address 0x00202 (MST Control):     0x02  (UP request enabled)
DPCD Address 0x00280 (Payload Table Size): 0x08  (max 8 payload IDs)
```

#### 4.3 分析参考

以下是针对本案例的详细分析，供自我评估使用。

```
问题 1 分析：
黑屏的直接原因是 MST Payload ID 耗尽。DPCD Address 0x00280 显示
Payload Table Size 为 8（最大支持 8 个 payload ID）。对于 3×4K@60
的 MST 菊花链架构：

  从源端（GPU）到第一个显示器：1 个 payload（流 1）
  从第一个显示器（MST branch）到第二个显示器：1 个 payload（流 2）
  从第二个显示器到第三个显示器：1 个 payload（流 3）

在 Wayland（GNOME）下，合成器可能会为每个显示器创建额外的
"scanout" payload 用于桌面合成，导致 payload ID 快速耗尽。

问题 2 分析：
X11 下问题不复现的原因是：

  1. X11 的合成器（如 compton/picom）通常不使用 atomic modeset
     进行增量更新，而是使用 legacy modeset
  2. X11 不会为每个显示器维护独立的 scanout buffer，减少了
     payload 分配请求
  3. X11 不频繁触发 MST payload 的 allocate/deallocate 周期

问题 3 修复方案：

  方案 A（驱动修复）：优化 MST payload 管理，在分配前先回收
  过期/不再使用的 payload ID。参考 amdgpu_dm_mst.c 中的
  dm_dp_mst_get_vcpi_slots() 实现，增加 payload 复用逻辑。

  方案 B（Workaround）：降低第三个显示器的分辨率（如降到
  2560×1440），减少带宽占用，从而减少所需的 payload ID 数量。

  方案 C（内核参数）：尝试修改 MST 的带宽分配策略，增加
  MST 的 slot 数量限制或修改 VCPI 分配算法。

  实际修复参考（Linux 6.6+）：
  commit 8e7e4c8f7c5e ("drm/amd/display: Fix MST payload reuse logic")
  该补丁修复了 MST payload ID 不能正确回收的问题。
```

## 扩展思考

1. **EDID 安全**：EDID 覆盖注入可以在测试中模拟各种显示器能力，但这种能力是否可能被恶意利用？例如，通过注入精心构造的 EDID 触发驱动中的缓冲区溢出漏洞。请查阅 CVE 数据库中与 EDID 相关的安全漏洞。

2. **多屏性能开销**：现代桌面环境越来越多地使用独立合成器（如 GNOME Mutter、KDE KWin）。在 3×4K@60 或 6×4K@60 等多屏配置下，合成器的 GPU 负载和非连续内存带宽会成为瓶颈。思考如何测量和优化多屏场景下的合成性能。

3. **HDR 端到端校准**：HDR 内容的色彩准确度不仅取决于驱动和显示器，还取决于应用程序是否正确传递了 PQ（Perceptual Quantizer）或 HLG（Hybrid Log-Gamma）信号。探索如何建立从 GPU 输出到色度计采集的完整校准流程。
