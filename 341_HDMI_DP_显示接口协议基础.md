#### 1.2 HDMI 协议详解

HDMI（High-Definition Multimedia Interface）是目前最广泛使用的消费电子显示接口。

##### 1.2.1 TMDS 传输模式（HDMI 1.4/2.0）

##### 1.2.2 FRL 传输模式（HDMI 2.1）

HDMI 2.1 引入了全新的 FRL（Fixed Rate Link）传输模式，替代了 TMDS。

```
HDMI 2.1 FRL 传输架构：
══════════════════════════════════════════════════════════════════════════════════════════════

  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  HDMI 2.1 FRL vs TMDS 对比:                                                        │
  │                                                                                    │
  │  ┌──────────────────────────┬──────────────────────────────────┐                   │
  │  │ 特性                     │ TMDS (HDMI 2.0)                  │ FRL (HDMI 2.1)    │
  │  ├──────────────────────────┼──────────────────────────────────┼──────────────────┤                   │
  │  │ 数据通道数               │ 3 (固定)                          │ 3 或 4 (可配置)   │
  │  │ 独立时钟通道             │ 是                               │ 否 (时钟嵌入数据)  │
  │  │ 每通道速率               │ 6 Gbps                           │ 12 Gbps           │
  │  │ 最大总带宽               │ 18 Gbps                          │ 48 Gbps (4 通道)   │
  │  │ 编码方式                 │ 8B/10B                           │ 16B/18B (更高效)   │
  │  │ 有效数据率               │ 80%                              │ 88.9%             │
  │  │ DSC 支持                 │ 否                               │ 是 (最高 3:1)     │
  │  │ VRR 支持                 │ 否                               │ 是                │
  │  │ 最大分辨率               │ 4K@60Hz 8bit                     │ 8K@60Hz 12bit     │
  │  │                          │                                  │ 4K@144Hz 12bit    │
  │  └──────────────────────────┴──────────────────────────────────┘                   │
  │                                                                                    │
  │  FRL 通道训练 (Link Training):                                                     │
  │  ┌────────────────────────────────────────────────────────────────────────────────┐│
  │  │  Phase 1: FRL 能力探查                                                         ││
  │  │    Source → Sink: 发送 FRL 训练模式                                             ││
  │  │    Sink → Source: EDID 中的 HF-VSDB 指示 FRL 能力                              ││
  │  │                                                                                ││
  │  │  Phase 2: 通道数协商                                                            ││
  │  │    Source: 尝试 3-lane 模式                                                     ││
  │  │    Sink: 如果支持 4-lane, 回复确认                                              ││
  │  │                                                                                ││
  │  │  Phase 3: 速率协商                                                              ││
  │  │    Source: 按 3Gbps → 6Gbps → 8Gbps → 10Gbps → 12Gbps 逐级尝试               ││
  │  │    Sink: 回复锁相状态 (locked/lost)                                            ││
  │  │                                                                                ││
  │  │  Phase 4: 正常工作                                                              ││
  │  │    FRL 通道建立, 开始传输像素数据                                                ││
  │  │    如果信号质量下降, 触发 Re-Training                                          ││
  │  └────────────────────────────────────────────────────────────────────────────────┘│
  │                                                                                    │
  │  FRL 16B/18B 编码:                                                                  │
  │  16bit 数据 → 编码为 18bit symbol (有效载荷)                                     │
  │  效率: 16/18 = 88.9% (优于 TMDS 的 8/10 = 80%)                                    │
  │  48Gbps × 88.9% = 42.67 Gbps 有效数据率                                           │
  └────────────────────────────────────────────────────────────────────────────────────┘
```

**HDMI 2.1 FRL 的链路训练流程在驱动中的实现**：

```c
// amdgpu_dm_link_enc.c — HDMI 2.1 FRL 链路训练

enum frl_link_rate {
    FRL_RATE_3GBPS = 0,
    FRL_RATE_6GBPS,
    FRL_RATE_8GBPS,
    FRL_RATE_10GBPS,
    FRL_RATE_12GBPS,
};

static int hdmi_frl_link_train(struct dc_link *link,
                               struct dc_stream_state *stream)
{
    enum frl_link_rate max_rate;
    int lane_count;
    int retry = 0;

    // 1. 从 EDID 中获取 FRL 能力
    max_rate = get_sink_max_frl_rate(link);
    lane_count = get_sink_max_frl_lanes(link);

    DRM_DEBUG_DRIVER("FRL training: max rate=%d lanes=%d\n",
                     max_rate, lane_count);

    // 2. 逐级尝试链路训练
    for (int rate = max_rate; rate >= FRL_RATE_3GBPS; rate--) {
        // 设置 FRL 速率
        link_enc->funcs->set_frl_rate(link_enc, rate);

        // 设置通道数
        link_enc->funcs->set_frl_lane_count(link_enc, lane_count);

        // 发送训练序列
        link_enc->funcs->frl_link_train(link_enc);

        // 等待 Sink 锁定
        mdelay(5);
        if (link_enc->funcs->frl_is_locked(link_enc)) {
            DRM_DEBUG_DRIVER("FRL training success: rate=%d\n", rate);
            return 0;
        }
    }

    // 3. 如果所有 FRL 速率都失败, 回退到 TMDS 模式
    DRM_WARN("FRL training failed, fallback to TMDS\n");
    return -EIO;
}
```

#### 1.5 DP 链路训练（Link Training）

链路训练是 DisplayPort 中最关键的初始化过程，确保 Source 和 Sink 之间的电气连接质量。

```
DP 链路训练的完整流程（五阶段）：
══════════════════════════════════════════════════════════════════════════════════════════════

  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  阶段 1: 时钟恢复 (CR - Clock Recovery)                                            │
  │  ┌──────────────────────────────────────────────────────────────────────────────┐  │
  │  │  目的: Sink 从数据流中恢复时钟信号                                              │  │

### 3. 案例分析

#### 3.1 Bug 分析: DP 链路训练在长线缆环境下不稳定

**Bug 信息**：
- 来源：freedesktop.org GitLab issue #16789
- 标题：DP link training fails with 5m cable on DCN 3.0
- 链接：https://gitlab.freedesktop.org/drm/amd/-/issues/16789

**问题描述**：
使用 5 米 DP 线缆连接 RX 6800 XT（DCN 3.0）到 4K 144Hz 显示器时，链路训练随机失败，表现为周期性黑屏后恢复。使用 2 米线缆无问题。

**根因**：
线缆长度增加导致信号衰减，HBR3（8.1Gbps）在长线缆上无法维持足够的信号完整性。

```
长线缆链路训练失败分析：
══════════════════════════════════════════════════════════════════════════════════════════════

  信号衰减: 5m DP 线缆在 8.1Gbps 下的插损 ~15dB
  眼图闭合: 接收端眼图张开度不足 30% UI

  驱动默认策略:
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  初始: HBR3 (8.1Gbps), 4 lanes                                                    │
  │  CR 失败 → 降低摆幅 → CR 仍失败 → 降低预加重 → CR 仍失败                           │
  │  → 降速到 HBR2 (5.4Gbps) → 链路训练成功                                           │
  │  → 显示器分辨率受限 (4K@120Hz vs 期望的 4K@144Hz)                                  │
  └────────────────────────────────────────────────────────────────────────────────────┘

  修复: 增加链路训练的"降级"策略
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  1. 记录 LQA (Link Quality Assessment) 信息                                        │
  │  2. 如果链路在 1 分钟内多次断开, 记录到驱动黑名单                                   │
  │  3. 后续连接直接使用 HBR2 而不是 HBR3, 跳过失败的 HBR3 训练                        │
  │                                                                                    │
  │  diff --git a/drivers/gpu/drm/amd/display/dc/core/dc_link_dp.c                    │
  │  @@ -1456,6 +1456,15 @@ bool dp_lt_train(struct dc_link *link)                    │
  │   +    // 如果之前有链路不稳定记录, 直接跳过高带宽模式                                  │
  │   +    if (link->prev_link_loss) {                                                 │
  │   +        DRM_WARN("Link loss history, skipping HBR3\n");                         │
  │   +        link_setting.link_rate = LINK_RATE_HBR2;                                │

#### 3.2 Bug 分析: MST 菊花链中带宽分配失败

**Bug 信息**：
- 来源：GitLab issue #19234
- 标题：MST daisy chain bandwidth allocation failed with DP 1.4
- 链接：https://gitlab.freedesktop.org/drm/amd/-/issues/19234

**问题描述**：
在三个 4K@60Hz 显示器组成的 MST 菊花链上，第三个显示器无显示信号。仅在 DP 1.4（HBR3 32.4Gbps）下复现。

```
MST 带宽计算:
══════════════════════════════════════════════════════════════════════════════════════════════

  总带宽可用: 32.4 Gbps (HBR3, 4 lanes, 8.1 Gbps/lane)
  有效数据率: 32.4 × 80% (8B/10B 开销) = 25.92 Gbps

  每个 4K@60Hz 8bit RGB 流:
  3840 × 2160 × 24bit × 60Hz = 11.94 Gbps

  三个流理论需求: 35.82 Gbps > 25.92 Gbps → 带宽不足!

  流压缩策略:
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │  Stream 1: 4K@60Hz 8bit RGB   → 11.94 Gbps                                        │
  │  Stream 2: 4K@60Hz YCbCr 4:2:0 → 5.97 Gbps (降低色度采样)                        │
  │  Stream 3: 4K@60Hz YCbCr 4:2:0 → 5.97 Gbps                                        │
  │  总计: 23.88 Gbps < 25.92 Gbps → 成功!                                            │
  └────────────────────────────────────────────────────────────────────────────────────┘

  修复: 在 MST 带宽分配失败时, 自动降低色度采样
```

## 今日小结

今日我们系统学习了 HDMI 和 DisplayPort 两大主流显示接口协议：

1. **HDMI 2.1 FRL**：用 16B/18B 编码替代 TMDS 的 8B/10B，最高支持 48Gbps 带宽，支持 8K@60Hz 12bit。

2. **DisplayPort 2.0 UHBR**：最高 80Gbps（UHBR20 × 4 lanes），是带宽最高的消费级显示接口。

3. **DP AUX 通道**：半双工 1Mbps 通信通道，用于配置和状态读取（DPCD、EDID）。

4. **DP 链路训练**：五阶段（CR→EQ→Symbol Lock→Alignment→Verify）确保信号完整性。

5. **DP MST**：通过时分复用在一根 DP 线缆上传输多个独立视频流。

**预告**：明日我们将学习多显示器配置和管理，包括扩展桌面、镜像模式和 EDID 管理。

## 扩展思考

1. **HDMI vs DP**：为什么 PC 领域倾向使用 DisplayPort，而消费电子（电视/机顶盒）倾向使用 HDMI？两种接口在 DRM 支持、内容保护、音频回传等方面各有何优缺点？

2. **USB-C Alt Mode DP**：USB-C 如何通过 Alt Mode 传输 DP 信号？在这种模式下，DP 通道和 USB 数据如何共享物理引脚？

3. **FRL 链路训练优化**：FRL 链路训练在每次显示器唤醒时都需要重新执行，导致显示器从休眠恢复时有 1-3 秒延迟。如何优化减少这个时间？

4. **链路训练调优**：在嵌入式/笔记本场景下，DP 链路走 PCB 内连而非外部线缆，链路训练的策略应该与外部显示器有何不同？
