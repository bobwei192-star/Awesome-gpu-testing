# 测试用例设计方法：场景法与状态转换图

## 1. 学习目标

- 理解场景法的核心思想和应用场景
- 掌握状态转换图的建模方法
- 学会将场景法和状态转换图应用于GPU驱动测试
- 能够结合等价类划分、边界值分析和决策表综合设计测试用例
- 理解场景法在复现复杂GPU Bug中的关键作用

## 2. 场景法理论基础

### 2.1 场景法的定义与价值

场景法（Scenario Testing）是一种基于用户使用场景设计测试用例的方法，关注的是用户在实际使用过程中的操作流程和系统响应。与基于规格说明的结构化方法不同，场景法强调真实世界的使用模式和业务流。

```
场景法与其他方法的关系：

┌─────────────────────────────────────────────────────────────┐
│  等价类划分（WHAT）                                          │
│    ↓                                                         │
│  边界值分析（WHERE）                                          │
│    ↓                                                         │
│  决策表/因果图（IF-THEN）                                     │
│    ↓                                                         │
│  场景法（HOW） ← 我们在这里                                   │
│    ↓                                                         │
│  状态转换图（WHEN）                                           │
│    ↓                                                         │
│  实际执行测试                                                │
└─────────────────────────────────────────────────────────────┘
```

场景法的核心价值：

```
1. 发现集成问题：单一接口测试无法发现的组件间交互问题
2. 覆盖真实使用模式：用户实际操作的流程路径
3. 揭示时序依赖：操作顺序对系统行为的影响
4. 验证端到端功能：从用户输入到系统输出的完整链路
5. 复现复杂缺陷：许多GPU Bug只在特定场景序列下触发
```

### 2.2 基本流与备选流

场景法的核心概念是基本流（Basic Flow）和备选流（Alternate Flow）。

```
基本流（Basic Flow）：
  系统的最主要、最常用的功能路径，即"快乐路径"。
  例如：打开应用 → 创建BO → 使用BO → 销毁BO → 关闭应用

备选流（Alternate Flow）：
  基本流的变体，包括可选路径、错误路径和异常路径。
  例如：
  - 备选流1: 创建BO时内存不足 → 返回错误
  - 备选流2: 使用BO时发生GPU Hang → 重置
  - 备选流3: 销毁BO时还有进程在使用 → 引用计数延迟释放

场景法测试设计步骤：

┌─────────────────────────────────────────────────────────────┐
│  步骤1: 分析需求，识别基本流                                  │
│  步骤2: 识别备选流（异常、超时、边界、中断、并发）              │
│  步骤3: 构建场景矩阵（基本流 + 备选流的组合）                   │
│  步骤4: 为每个场景设计测试用例                                 │
│  步骤5: 场景覆盖评估                                           │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 场景矩阵构建

场景矩阵是系统化场景设计的关键工具。

```
场景矩阵结构：

┌─────────────────────────────────────────────────────────────────┐
│  场景ID  │  路径描述              │  基本流 │  备选流组合       │
├─────────────────────────────────────────────────────────────────┤
│  S1     │  完美路径               │  全程   │  无               │
│  S2     │  异常路径               │  部分   │  A1(创建BO失败)   │
│  S3     │  异常路径               │  部分   │  A2(使用中挂起)   │
│  S4     │  异常路径               │  部分   │  A3(销毁延迟)     │
│  S5     │  复杂路径               │  部分   │  A1 + A3          │
│  S6     │  循环路径               │  部分   │  A1 → A2 → A3    │
│  S7     │  回退路径               │  部分   │  基本流中途回退   │
│  S8     │  边界路径               │  部分   │  资源边界条件     │
└─────────────────────────────────────────────────────────────────┘

场景的三种基本类型：

类型A: 快乐路径场景
  定义：用户按预期方式使用功能，系统正常响应
  覆盖：基本流从头到尾执行，不触发任何错误
  风险：依赖环境完全正常，无法发现异常处理问题

类型B: 异常处理场景
  定义：在基本流中插入备选流（错误/异常处理）
  覆盖：系统在非理想条件下的行为
  风险：需要知道所有可能的异常点

类型C: 复杂交互场景
  定义：多个备选流组合，或备选流之间的交互
  覆盖：多个错误同时发生或交错发生
  风险：组合爆炸，需要合理筛选
```

## 3. 状态转换图理论基础

### 3.1 状态转换图的核心概念

状态转换图（State Transition Diagram）用图形化的方式描述系统在不同状态之间如何迁移。

```
基本元素：

┌──────────────┐
│   状态       │  用圆角矩形表示系统的稳定状态
│  State      │  例如：IDLE, ACTIVE, SUSPENDED, ERROR
└──────────────┘

──────────────►  转换箭头表示从一个状态到另一个状态
    event      箭头上的标签表示触发转换的事件

┌──────────────┐
│   起始状态    │  用实心圆点表示系统的初始状态
└──────────────┘

┌──────────────┐
│   终止状态    │  用带圆圈的实心圆点表示系统结束
└──────────────┘

[guard]/action  转换上的守卫条件和动作
                例如：ref==0/free_bo

状态转换表（State Transition Table）：

┌──────────────┬────────────┬──────────────┬──────────────────┐
│  当前状态    │  事件       │  守卫条件     │  下一状态/动作    │
├──────────────┼────────────┼──────────────┼──────────────────┤
│  IDLE        │  CREATE    │  资源充足     │  ACTIVE          │
│  IDLE        │  CREATE    │  资源不足     │  ERROR           │
│  ACTIVE      │  TIMEOUT   │  -            │  IDLE            │
│  ACTIVE      │  ERROR     │  -            │  RECOVERY        │
│  RECOVERY    │  RESET_OK  │  -            │  IDLE            │
│  RECOVERY    │  RESET_FAIL│  -            │  FATAL_ERROR     │
└──────────────┴────────────┴──────────────┴──────────────────┘
```

### 3.2 状态转换图的测试覆盖准则

```
覆盖准则从弱到强：

准则1: 状态覆盖（State Coverage）
  每个状态至少被访问一次
  目标状态数: N
  测试用例数: N

准则2: 转换覆盖（Transition Coverage）
  每个可能的转换至少被执行一次
  目标转换数: M (≤ N²)
  测试用例数: M

准则3: 路径覆盖（Path Coverage）
  每个可能的路径序列至少被执行一次
  目标路径数: 指数级增长
  测试用例数: 通常选择代表性路径

准则4: 循环覆盖（Loop Coverage）
  循环0次、1次、多次
  适用于状态可自循环的场景
  测试用例数: 3

推荐策略：转换覆盖 + 代表性路径覆盖
          转换覆盖保证所有迁移被测试
          路径覆盖发现序列相关缺陷
```

## 4. GPU驱动中的场景法应用

### 4.1 AMDGPU BO管理场景分析

```
场景：BO完整生命周期管理

基本流（BF）:
  1. 打开DRM设备
  2. 创建BO（gem_create）
  3. 映射BO到用户空间（gem_mmap）
  4. 设置BO域（gem_set_domain）
  5. 使用BO（CPU/GPU操作）
  6. 取消映射（munmap）
  7. 关闭BO（gem_close）
  8. 关闭DRM设备

备选流识别：

  AF1: 创建BO失败
    触发: 请求大小超过VRAM容量
    预期: 返回错误码，资源未分配

  AF2: 映射失败
    触发: BO还未完全创建（时机竞争）
    预期: 返回映射失败或延迟映射

  AF3: 使用BO时GPU Hang
    触发: BO中的命令缓冲区包含非法指令
    预期: GPU恢复机制触发，BO重置

  AF4: 关闭BO时仍有引用
    触发: 多个进程共享同一个BO
    预期: 引用计数减少，不释放BO

  AF5: 关闭DRM设备前未关闭BO
    触发: 进程异常退出
    预期: 内核自动清理BO资源

场景矩阵：

┌────────────┬────────────────────────────────┬────────────────────┐
│ 场景ID     │ 路径                           │ 预期结果           │
├────────────┼────────────────────────────────┼────────────────────┤
│ BO-S1      │ BF全程（快乐路径）              │ BO正确创建/使用/销毁│
│ BO-S2      │ BF + AF1（创建失败）           │ 优雅报错，无泄漏    │
│ BO-S3      │ BF + AF2（映射失败）           │ 可恢复的错误处理    │
│ BO-S4      │ BF + AF3（GPF挂起）            │ GPU恢复，BO重置     │
│ BO-S5      │ BF + AF4（多进程共享）          │ 引用计数正确        │
│ BO-S6      │ BF + AF5（异常退出）            │ 内核清理           │
│ BO-S7      │ BF + AF1 + AF3（组合）          │ 完整错误恢复链路    │
│ BO-S8      │ BF循环100次（压力场景）          │ 无泄漏/无碎片化     │
└────────────┴────────────────────────────────┴────────────────────┘
```

### 4.2 AMDGPU执行调度场景分析

```
场景：命令提交与执行

基本流（BF）:
  1. 创建命令缓冲区（BO）
  2. 填充命令数据
  3. 提交执行（gem_execbuf）
  4. 等待执行完成（gem_wait）
  5. 读取结果
  6. 销毁命令缓冲区

备选流：

  AF1: 提交到繁忙队列
    触发: 所有硬件队列已满
    预期: 返回-EBUSY或阻塞等待

  AF2: 执行超时
    触发: 命令执行超过阈值（通常2秒）
    预期: GPU重置，返回错误

  AF3: 提交非法命令
    触发: 命令缓冲区包含非法操作码
    预期: GPU页面错误，恢复机制触发

  AF4: 依赖fence未到达
    触发: 提交的命令依赖另一个未完成的命令
    预期: 硬件自动等待fence

场景矩阵：

┌────────────┬────────────────────────────────┬────────────────────┐
│ 场景ID     │ 路径                           │ 预期结果           │
├────────────┼────────────────────────────────┼────────────────────┤
│ EXEC-S1    │ BF全程                          │ 正常完成           │
│ EXEC-S2    │ BF + AF1（队列满）              │ 阻塞或返回EBUSY    │
│ EXEC-S3    │ BF + AF2（超时）                │ GPU重置            │
│ EXEC-S4    │ BF + AF3（非法命令）            │ GPU恢复           │
│ EXEC-S5    │ BF + AF4（fence依赖）           │ 正确串行化         │
│ EXEC-S6    │ BF + AF2 + AF3（命令错误→超时） │ 复合错误恢复       │
│ EXEC-S7    │ BF重复1000次（压力）            │ 无泄漏/无性能下降   │
└────────────┴────────────────────────────────┴────────────────────┘
```

### 5.3 BO引用计数状态转换

```
BO（Buffer Object）引用计数状态机：

状态定义：
  ACTIVE: 引用计数 > 0，BO正常使用
  PENDING_FREE: 引用计数 = 0，但GPU还在使用
  FREE: 完全释放，BO回到空闲列表

额外状态：
  EXPORTED: BO被导出为dma-buf，有外部引用
  PINNED: BO被锁定在物理内存中

                    ┌────────────────┐
                    │   CREATED      │
                    │   ref=1        │
                    └───────┬────────┘
                            │ get()
                    ┌───────▼────────┐
          get() ──►│    ACTIVE      │◄─────── get()
          put() ──►│    ref>0       │───────► put()
                    └───────┬────────┘
                            │ put() → ref=0
                    ┌───────▼────────┐
           export  │  PENDING_FREE  │◄─────── release_dmabuf
          ────────►│  ref=0         │
                    │  but GPU busy  │───────► gpu_idle
                    └───────┬────────┘
                            │ GPU idle
                    ┌───────▼────────┐
                    │     FREE       │
                    └────────────────┘

关键转换测试：

转换1: ACTIVE → PENDING_FREE → FREE
  场景：用户关闭BO + GPU尚未完成使用
  测试：关闭fd → 检查ref=0 → 等待fence → 确认BO释放

转换2: ACTIVE → EXPORTED → PENDING_FREE
  场景：BO被导出为dma-buf，用户关闭本地引用
  测试：创建BO → 导出dma-buf → 关闭BO fd → 外部仍可访问

转换3: ACTIVE → PINNED → ACTIVE
  场景：BO被锁定用于DMA操作
  测试：创建BO → pin → unpin → 确认正常

转换4: 循环创建/销毁
  场景：反复创建和销毁BO
  测试：循环1000次 → 确认空闲列表无泄露
```

### 6.2 使用IGT验证GPU上下文状态转换

```c
/*
 * IGT测试：GPU上下文状态转换验证
 * 对应状态转换表中的关键转换路径
 */

#include "igt.h"

#define CONTEXT_CREATE_FLAGS_USE_HWS 0x1

typedef struct {
    int drm_fd;
    uint32_t ctx_id;
    uint32_t bo_handle;
    uint64_t bo_size;

/*
 * 转换测试1: INIT → READY → ACTIVE → COMPLETED
 * 完整的上下文生命周期：创建→提交→完成
 */
static void
test_context_lifecycle(ctx_test_data_t *data)
{
    struct drm_amdgpu_ctx_in ctx_in = {0};
    struct drm_amdgpu_ctx_out ctx_out = {0};
    int ret;

    igt_info("测试: INIT → READY → ACTIVE → COMPLETED\n");

    /* INIT → READY: 创建上下文 */
    ctx_in.op = AMDGPU_CTX_OP_ALLOC;
    ctx_in.flags = 0;
    ret = drmCommandWriteRead(data->drm_fd, DRM_AMDGPU_CTX,
                               &ctx_in, sizeof(ctx_in));
    igt_assert_eq(ret, 0);
    data->ctx_id = ctx_out.alloc.id;
    igt_info("  上下文创建成功: ctx_id=%u\n", data->ctx_id);

    /* READY → ACTIVE: 提交执行 */
    struct drm_amdgpu_gem_create_in gem_in = {0};
    struct drm_amdgpu_gem_create_out gem_out = {0};

    gem_in.in.bo_size = 4096;
    gem_in.in.alignment = 4096;
    gem_in.in.domains = AMDGPU_GEM_DOMAIN_GTT;
    gem_in.in.domain_flags = 0;

    ret = drmCommandWriteRead(data->drm_fd, DRM_AMDGPU_GEM_CREATE,
                               &gem_in, sizeof(gem_in));
    igt_assert_eq(ret, 0);
    data->bo_handle = gem_out.out.handle;

    /* 提交到GPU */
    struct drm_amdgpu_gem_execbuf_in execbuf_in = {0};
    execbuf_in.in.ctx_id = data->ctx_id;
    execbuf_in.in.bo_handles = (uint64_t)&data->bo_handle;
    execbuf_in.in.num_bos = 1;

    ret = drmCommandWriteRead(data->drm_fd, DRM_AMDGPU_GEM_EXECBUF,
                               &execbuf_in, sizeof(execbuf_in));
    igt_assert_eq(ret, 0);
    igt_info("  提交执行成功\n");

    /* ACTIVE → COMPLETED: 等待完成 */
    struct drm_amdgpu_gem_wait_in wait_in = {0};
    struct drm_amdgpu_gem_wait_out wait_out = {0};
    wait_in.in.bo_handle = data->bo_handle;
    wait_in.in.timeout_ns = 5000000000ULL; /* 5秒超时 */

    ret = drmCommandWriteRead(data->drm_fd, DRM_AMDGPU_GEM_WAIT,
                               &wait_in, sizeof(wait_in));
    igt_assert_eq(ret, 0);
    igt_info("  GPU完成执行\n");

    /* 清理 */
    gem_close(data->drm_fd, data->bo_handle);
    amdgpu_ctx_free(data->ctx_id);
}

/*
 * 转换测试2: ACTIVE → PREEMPTED → ACTIVE
 * 提交高优任务抢占当前任务
 */
static void
test_context_preempt(ctx_test_data_t *data)
{
    struct drm_amdgpu_ctx_in ctx_low = {0};
    struct drm_amdgpu_ctx_in ctx_high = {0};
    struct drm_amdgpu_ctx_out ctx_low_out = {0};
    struct drm_amdgpu_ctx_out ctx_high_out = {0};
    int ret;

    igt_info("测试: ACTIVE → PREEMPTED → ACTIVE (抢占)\n");

    /* 创建低优先级上下文 */
    ctx_low.op = AMDGPU_CTX_OP_ALLOC;
    ctx_low.flags = 0;  /* 默认低优先级 */
    ret = drmCommandWriteRead(data->drm_fd, DRM_AMDGPU_CTX,
                               &ctx_low, sizeof(ctx_low));
    igt_assert_eq(ret, 0);
    igt_info("  低优先级上下文创建: %u\n", ctx_low_out.alloc.id);

    /* 创建高优先级上下文 */
    ctx_high.op = AMDGPU_CTX_OP_ALLOC;
    ctx_high.flags = AMDGPU_CTX_PRIORITY_HIGH;
    ret = drmCommandWriteRead(data->drm_fd, DRM_AMDGPU_CTX,
                               &ctx_high, sizeof(ctx_high));
    igt_assert_eq(ret, 0);

    igt_info("  高优先级上下文创建: %u\n", ctx_high_out.alloc.id);

    /* 创建BO并提交到低优上下文 */
    data->bo_handle = gem_create(data->drm_fd, 4096);
    igt_assert(data->bo_handle);

    struct drm_amdgpu_gem_execbuf_in exec_low = {0};
    exec_low.in.ctx_id = ctx_low_out.alloc.id;
    exec_low.in.bo_handles = (uint64_t)&data->bo_handle;
    exec_low.in.num_bos = 1;

    ret = drmCommandWriteRead(data->drm_fd, DRM_AMDGPU_GEM_EXECBUF,
                               &exec_low, sizeof(exec_low));
    igt_assert_eq(ret, 0);

    /* 在低优任务执行中，提交高优任务 */
    struct drm_amdgpu_gem_execbuf_in exec_high = {0};
    exec_high.in.ctx_id = ctx_high_out.alloc.id;
    exec_high.in.bo_handles = (uint64_t)&data->bo_handle;
    exec_high.in.num_bos = 1;

    ret = drmCommandWriteRead(data->drm_fd, DRM_AMDGPU_GEM_EXECBUF,
                               &exec_high, sizeof(exec_high));

    igt_info("  高优任务提交成功（应触发抢占）\n");
    igt_assert_eq(ret, 0);

    /* 等待所有完成 */
    gem_wait(data->drm_fd, data->bo_handle, 5000);

    /* 清理 */
    gem_close(data->drm_fd, data->bo_handle);
    amdgpu_ctx_free(ctx_low_out.alloc.id);
    amdgpu_ctx_free(ctx_high_out.alloc.id);
}
```

```python
#!/usr/bin/env python3
"""
场景法驱动的GPU驱动测试框架
支持场景矩阵定义、自动执行和结果验证
"""

import subprocess
import time
import json
import logging
from enum import Enum
from typing import List, Dict, Callable, Optional


class ScenarioResult(Enum):
    PASS = "PASS"
    FAIL = "FAIL"
    SKIP = "SKIP"
    BLOCKED = "BLOCKED"


class ScenarioStep:
    """场景步骤定义"""
    
    def __init__(self, name: str, action: Callable,
                 expected: str, timeout: int = 30):
        self.name = name
        self.action = action
        self.expected = expected
        self.timeout = timeout
        self.result = None
        self.error = None


class Scenario:
    """场景定义"""
    
    def __init__(self, scenario_id: str, description: str,
                 basic_flow: List[ScenarioStep],
                 alternate_flows: Optional[List[ScenarioStep]] = None):
        self.id = scenario_id
        self.description = description
        self.basic_flow = basic_flow
        self.alternate_flows = alternate_flows or []
        self.steps = []
        self.result = ScenarioResult.SKIP
    
    def execute(self):
        """执行场景"""
        logging.info(f"执行场景: {self.id} - {self.description}")
        self.steps = []
        
        try:
            # 执行基本流
            for step in self.basic_flow:
                self._execute_step(step)
            
            # 执行备选流
            for step in self.alternate_flows:
                self._execute_step(step)
            
            self.result = ScenarioResult.PASS
            
        except AssertionError as e:
            self.result = ScenarioResult.FAIL
            logging.error(f"场景 {self.id} 失败: {e}")
        except Exception as e:
            self.result = ScenarioResult.BLOCKED
            logging.error(f"场景 {self.id} 阻塞: {e}")
        
        return self.result
    
    def _execute_step(self, step: ScenarioStep):
        """执行单个步骤"""
        logging.info(f"  步骤: {step.name}")
        step.action()
        self.steps.append(step)
    
    def report(self) -> Dict:
        """生成场景报告"""
        return {
            "id": self.id,
            "description": self.description,
            "result": self.result.value,
            "steps": [{"name": s.name, "expected": s.expected}
                     for s in self.steps]
        }


class GPUScenarioRunner:
    """GPU场景执行器"""
    
    def __init__(self, drm_device: str = "/dev/dri/card0"):
        self.drm_device = drm_device
        self.scenarios: List[Scenario] = []
        self.results: List[Dict] = []
    
    def add_scenario(self, scenario: Scenario):
        """添加场景"""
        self.scenarios.append(scenario)
    
    def run_all(self):
        """执行所有场景"""
        for scenario in self.scenarios:
            result = scenario.execute()
            self.results.append(scenario.report())
            logging.info(f"  结果: {result.value}")
    
    def generate_report(self, output_file: str):
        """生成报告"""
        report = {
            "total": len(self.scenarios),
            "passed": sum(1 for r in self.results
                         if r["result"] == "PASS"),
            "failed": sum(1 for r in self.results
                         if r["result"] == "FAIL"),
            "scenarios": self.results
        }
        
        with open(output_file, 'w') as f:
            json.dump(report, f, indent=2)
        
        print(f"\n场景测试报告: {output_file}")
        print(f"总计: {report['total']}, "
              f"通过: {report['passed']}, "
              f"失败: {report['failed']}")
```

### 6.4 状态转换自动化测试框架

```python
#!/usr/bin/env python3
"""
状态转换图驱动的GPU驱动测试框架
支持状态模型定义、转换路径生成和自动执行
"""
from enum import Enum, auto
from typing import Dict, List, Tuple, Set, Optional, Callable
from dataclasses import dataclass, field
import json
import logging
import subprocess
import time


class StateTransitionError(Exception):
    """状态转换错误"""
    pass


@dataclass
class Transition:
    """状态转换定义"""
    name: str
    source: str
    target: str
    action: Callable
    guard: Optional[Callable] = None
    expected_output: Optional[str] = None


@dataclass
class StateModel:
    """状态机模型"""
    name: str
    states: Set[str]
    initial_state: str
    transitions: List[Transition] = field(default_factory=list)
    current_state: str = ""
    
    def __post_init__(self):
        self.current_state = self.initial_state
    
    def add_transition(self, t: Transition):
        """添加转换"""
        assert t.source in self.states, f"源状态 {t.source} 不在状态集中"
        assert t.target in self.states, f"目标状态 {t.target} 不在状态集中"
        self.transitions.append(t)
    
    def get_valid_transitions(self, state: Optional[str] = None) -> List[Transition]:
        """获取从给定状态出发的有效转换"""
        s = state or self.current_state
        valid = []
        for t in self.transitions:
            if t.source == s:
                if t.guard is None or t.guard():
                    valid.append(t)
        return valid
    
    def apply_transition(self, transition: Transition) -> str:
        """应用转换并返回新状态"""
        if transition.source != self.current_state:
            raise StateTransitionError(
                f"无法从 {self.current_state} 应用 {transition.name}"
                f" (需要源状态: {transition.source})"
            )
        if transition.guard and not transition.guard():
            raise StateTransitionError(
                f"转换 {transition.name} 的守卫条件不满足"
            )
        transition.action()
        self.current_state = transition.target
        logging.info(f"  状态转换: {transition.source} → {transition.target}")
        return self.current_state
    
    def reset(self):
        """重置到初始状态"""
        self.current_state = self.initial_state
        logging.info(f"  重置状态: {self.current_state}")


class StateTransitionTester:
    """状态转换测试器"""
    
    def __init__(self, model: StateModel):
        self.model = model
        self.coverage: Dict[str, bool] = {}
        self.paths_tested: List[List[str]] = []
    
    def test_state_coverage(self) -> float:
        """状态覆盖测试：访问每个状态至少一次"""
        visited: Set[str] = set()
        self.model.reset()
        visited.add(self.model.current_state)
        
        # 贪心算法：尽可能访问未访问的状态
        max_iterations = 100
        iteration = 0
        
        while len(visited) < len(self.model.states) and iteration < max_iterations:
            iteration += 1
            valid_transitions = self.model.get_valid_transitions()
            
            # 优先选择能到达未访问状态的转换
            best_transition = None
            for t in valid_transitions:
                if t.target not in visited:
                    best_transition = t
                    break
            
            if best_transition is None and valid_transitions:
                best_transition = valid_transitions[0]
            
            if best_transition is None:
                break
            
            self.model.apply_transition(best_transition)
            visited.add(self.model.current_state)
        
        coverage = len(visited) / len(self.model.states)
        logging.info(f"状态覆盖: {len(visited)}/{len(self.model.states)} = {coverage:.1%}")
        return coverage
    
    def test_transition_coverage(self) -> float:
        """转换覆盖测试：每条转换至少执行一次"""
        executed: Set[str] = set()
        total = len(self.model.transitions)
        
        for t in self.model.transitions:
            # 尝试执行每条转换
            self.model.reset()
            try:
                # 找到到达源状态的路径
                path = self._find_path(self.model.initial_state, t.source)
                if path is None:
                    logging.warning(f"  无法到达状态 {t.source}")
                    continue
                
                # 沿路径到达源状态
                for step in path:
                    self.model.apply_transition(step)
                
                # 执行目标转换
                self.model.apply_transition(t)
                executed.add(t.name)
                logging.info(f"  转换覆盖: {t.name}")
                
            except (StateTransitionError, AssertionError) as e:
                logging.warning(f"  转换 {t.name} 执行失败: {e}")
        
        coverage = len(executed) / total if total > 0 else 1.0
        logging.info(f"转换覆盖: {len(executed)}/{total} = {coverage:.1%}")
        return coverage
    
    def _find_path(self, source: str, target: str,
                   max_depth: int = 10) -> Optional[List[Transition]]:
        """BFS查找两个状态间的路径"""
        if source == target:
            return []
        
        visited: Set[str] = {source}
        queue: List[Tuple[str, List[Transition]]] = [(source, [])]
        
        while queue:
            current, path = queue.pop(0)
            
            if len(path) >= max_depth:
                continue
            
            for t in self.model.get_valid_transitions(current):
                if t.target not in visited:
                    new_path = path + [t]
                    if t.target == target:
                        return new_path
                    visited.add(t.target)
                    queue.append((t.target, new_path))
        
        return None
    
    def test_n_path_coverage(self, n: int = 2) -> float:
        """N路径覆盖测试"""
        if n < 1:
            return 0.0
        
        all_paths: Set[str] = set()
        total_possible = len(self.model.states) ** n
        
        self.model.reset()
        
        # 执行随机路径
        for _ in range(min(50, total_possible * 2)):
            self.model.reset()
            path_str = self.model.current_state
            
            for _ in range(n):
                valid = self.model.get_valid_transitions()
                if not valid:
                    break
                import random
                t = random.choice(valid)
                self.model.apply_transition(t)
                path_str += f"→{self.model.current_state}"
            
            all_paths.add(path_str)
        
        coverage = len(all_paths) / total_possible if total_possible > 0 else 1.0
        logging.info(f"N路径覆盖(n={n}): {len(all_paths)}/{total_possible} = {coverage:.1%}")
        return coverage
    
    def generate_test_report(self) -> Dict:
        """生成状态转换测试报告"""
        state_cov = self.test_state_coverage()
        transition_cov = self.test_transition_coverage()
        
        report = {
            "model_name": self.model.name,
            "states": list(self.model.states),
            "transitions_count": len(self.model.transitions),
            "state_coverage": f"{state_cov:.1%}",
            "transition_coverage": f"{transition_cov:.1%}",
            "paths_tested": len(self.paths_tested),
            "status": "PASS" if (state_cov >= 0.8 and transition_cov >= 0.8) else "FAIL"
        }
        
        return report


# 示例：使用状态转换测试框架测试GPU电源管理
```

### 7.1 案例：IGT测试gem_basic缺陷

参考实际AMDGPU Bug（freedesktop.org Bugzilla #106942）：

```
Bug报告摘要：
┌─────────────────────────────────────────────────────────────┐
│ Bug ID: 106942                                               │
│ 标题: amdgpu: gem_close followed by gem_mmap causes         │
│       use-after-free                                         │
│ 影响版本: Linux 4.16-4.19                                    │
│ 复现步骤:                                                    │
│   1. gem_create(size=4096)                                    │
│   2. gem_mmap(handle) → addr                                 │
│   3. gem_close(handle)                                       │
│   4. access addr[0] = 0x42  ← use-after-free                 │
│ 实际行为: 步骤4写入已释放的内存                                │
│ 期望行为: 步骤4应返回SIGSEGV或mmap自动失效                     │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 场景重建

```
从Bug报告重建测试场景：

基本流（正常使用）:
  1. gem_create → 获得handle
  2. gem_mmap → 获得CPU访问地址
  3. 正常使用内存
  4. gem_unmap
  5. gem_close

Bug场景（问题发现）:
  1. gem_create → handle
  2. gem_mmap → addr
  3. gem_close（先关闭handle）
  4. 继续访问addr ← use-after-free!

这个场景违反了正常的"先unmap再close"顺序
→ 是一个"非法操作顺序"场景
```

### 7.3 衍生测试场景

```
从这一个Bug，可以衍生出多个测试场景：

场景1: close → mmap（非法）
  步骤: create → close → mmap(handle)
  预期: mmap失败（handle已无效）

场景2: close → read/write（非法）
  步骤: create → mmap → close → write(addr)
  预期: SIGSEGV或写入失败

场景3: double close（边界）
  步骤: create → close → close
  预期: 第二次close返回错误

场景4: close → export → reimport（恢复）
  步骤: create → export(dma-buf) → close → import(dma-buf)
  预期: 通过dma-buf fd仍可访问

场景5: 并发close和mmap
  步骤: 线程A: close; 线程B: mmap(同一handle)
  预期: 线程B的mmap要么成功（线程A的close还未生效）
        要么失败（close先执行）

场景6: close → close的不同fd
  步骤: 进程A: create → export → 进程B: import →
        进程A: close → 进程B: close
  预期: 引用计数正确，第二次close才释放
```

## 8. 场景法与状态转换图的局限性与补充

### 8.1 方法局限性

```
场景法的局限性：

1. 场景选择主观性
   选择哪些场景依赖测试人员的经验和判断
   可能遗漏边缘场景
   解决方案：结合等价类划分系统化识别场景

2. 场景组合爆炸
   基本流 + N个备选流的组合数 = 2^N
   N=10时，共有1024个场景
   解决方案：风险优先级排序，选择高价值场景

3. 无法保证接口级覆盖
   场景覆盖 ≠ 代码覆盖
   某些接口可能在所有场景中都被间接测试
   但某些分支可能永远无法通过场景到达
   解决方案：结合代码覆盖率分析

状态转换图的局限性：

1. 状态空间爆炸
   状态数N，转换数O(N²)
   复杂系统的状态模型可能非常庞大
   解决方案：层次化状态机，隐藏内部状态

2. 并发状态难以建模
   多个独立的状态机同时运行
   组合后的状态数 = 各个状态数的乘积
   解决方案：关注核心状态机，使用独立分析

3. 时间连续行为无法描述
   状态转换图是离散的，无法描述连续变化
   例如：频率的渐变过程
   解决方案：结合模拟和分析工具
```

## 10. 今日小结

今天的主题是场景法和状态转换图，这是从"用户视角"和"系统行为视角"设计测试用例的重要方法。

**核心收获**：

1. **场景法三要素**：基本流（快乐路径）、备选流（异常/可选路径）、场景矩阵（组合覆盖）。场景法擅长发现集成问题和时序依赖缺陷，弥补了结构化方法的不足。

2. **状态转换图三要素**：状态（系统稳定态）、事件（触发迁移的输入）、转换（带守卫条件和动作的迁移）。状态转换图擅长发现非法状态转换和缺失的状态处理。

3. **GPU驱动应用**：从BO管理、执行调度、KMS显示到电源管理，场景法和状态转换图覆盖了GPU驱动的核心功能区域。每个功能域都可以构建完整的场景矩阵和状态转换模型。

4. **IGT实践**：通过IGT框架，可以将场景法和状态转换图直接编码为可执行的测试用例。热插拔场景、上下文状态转换、GPU恢复路径等都可以用代码精确描述。

5. **方法集成**：场景法和状态转换图不是孤立的，需要与等价类划分、边界值分析和决策表协同工作，形成完整的测试设计方法体系。

**关键洞察**：场景法的最大价值在于发现"开发者没想到的"操作顺序。GPU驱动中的许多严重Bug（如use-after-free、double-free、竞态条件）都是在非标准操作序列下触发的。状态转换图则帮助系统化地发现这些"非法"但可能的操作路径。
