## 参考答案

### 第一部分：选择题

1. **B** - sparse 专门检查内核地址空间属性
2. **B** - smatch 的优势是数据流和状态机分析
3. **B** - CONFIG_GCOV_KERNEL 启用内核覆盖率支持
4. **C** - debugfs 暴露 gcov 运行时数据
5. **B** - 分支覆盖反映错误处理路径覆盖
6. **B** - Coccinelle 用于语义补丁
7. **A** - 代码风格是通用审查关注点，不是测试视角特有
8. **C** - .info 文件是 lcov 的聚合数据格式
9. **B** - genhtml 生成 HTML 报告
10. **B** - 锁的配对是并发安全的核心

### 第二部分：判断题

11. **错误**。100% 覆盖率只表示代码被执行过，不表示执行结果正确。覆盖率不能验证逻辑正确性。

12. **正确**。sparse 的核心功能就是检查 __user/__iomem/__force 等地址空间属性的正确使用。

13. **错误**。覆盖率数据存储在内存中的 debugfs，系统重启后会丢失。需要定期收集到持久存储。

14. **正确**。smatch 的交叉函数分析可以跟踪跨函数调用的资源分配和释放，检测内存泄露。

15. **正确**。静态分析不依赖编译产物，但通常在编译配置后进行，可以快速反馈代码问题。

### 第三部分：简答题

**16. 静态分析工具区别：**

| 工具 | 主要功能 | 适用场景 |
|------|---------|---------|
| sparse | 类型检查、地址空间分析 | 内核地址空间属性、接口边界 |
| smatch | 数据流分析、状态机 | 内存泄露、空指针、锁配对 |
| Coccinelle | 语义补丁、模式检测 | API 迁移、代码模式统一 |

**17. 内核覆盖率收集挑战：**

- 需要重新编译内核启用 CONFIG_GCOV_KERNEL
- 覆盖率数据通过 debugfs 暴露，需要 root 权限
- 模块动态加载需要分别收集
- 系统崩溃时数据丢失
- 性能开销影响测试结果

**18. 测试视角审查维度：**

- 边界条件处理（零值、极大值、空指针）
- 错误处理路径（资源释放、返回值检查）
- 并发安全性（锁配对、原子操作）
- 可测试性设计（接口抽象、mock 支持）

**19. lcov 生成报告步骤：**

```bash
# 1. 收集覆盖率数据
lcov --capture --directory /sys/kernel/debug/gcov \
     --output-file coverage.info

# 2. 提取 AMDGPU 代码
lcov --extract coverage.info "*/drivers/gpu/drm/amd/*" \
     --output-file amdgpu_coverage.info

# 3. 生成 HTML 报告
genhtml amdgpu_coverage.info --output-directory html_report \
     --branch-coverage --legend
```

**20. 覆盖率与成本平衡：**

- 核心模块（调度、内存管理）要求更高覆盖率（>80%）
- 调试功能、错误处理路径可以适当降低
- 新代码应要求 100% 覆盖
- 遗留代码逐步提升，不强制一次性达标

### 第四部分：综合实践题

**综合实践题一参考答案：**

```bash
#!/bin/bash
# static_analysis_ci.sh

set -e

REPORT_FILE="static_analysis_report.md"
ERRORS=0
WARNINGS=0

echo "# 静态分析报告" > "$REPORT_FILE"
echo "时间: $(date)" >> "$REPORT_FILE"
echo "" >> "$REPORT_FILE"

# 1. checkpatch
echo "## 1. checkpatch" >> "$REPORT_FILE"
checkpatch_output=$(mktemp)
./scripts/checkpatch.pl --strict --show-types -g HEAD > "$checkpatch_output" 2>&1 || true

checkpatch_errors=$(grep -c "ERROR:" "$checkpatch_output" || true)
checkpatch_warnings=$(grep -c "WARNING:" "$checkpatch_output" || true)

echo "- 错误: $checkpatch_errors" >> "$REPORT_FILE"
echo "- 警告: $checkpatch_warnings" >> "$REPORT_FILE"

if [ "$checkpatch_errors" -gt 0 ]; then
    ERRORS=$((ERRORS + checkpatch_errors))
    echo "**checkpatch 发现错误，阻断合并**" >> "$REPORT_FILE"
fi

# 2. sparse
echo "" >> "$REPORT_FILE"
echo "## 2. sparse" >> "$REPORT_FILE"
sparse_output=$(mktemp)
make C=1 CHECK="sparse" drivers/gpu/drm/amd/amdgpu/ > "$sparse_output" 2>&1 || true

sparse_warnings=$(grep -c "warning:" "$sparse_output" || true)
echo "- 警告: $sparse_warnings" >> "$REPORT_FILE"

if [ "$sparse_warnings" -gt 10 ]; then
    WARNINGS=$((WARNINGS + sparse_warnings))
    echo "**sparse 警告超过阈值**" >> "$REPORT_FILE"
fi

# 3. smatch
echo "" >> "$REPORT_FILE"
echo "## 3. smatch" >> "$REPORT_FILE"
smatch_output=$(mktemp)
smatch --project=kernel drivers/gpu/drm/amd/amdgpu/*.c > "$smatch_output" 2>&1 || true

smatch_issues=$(grep -c "warn:" "$smatch_output" || true)
echo "- 问题: $smatch_issues" >> "$REPORT_FILE"

# 总结
echo "" >> "$REPORT_FILE"
echo "## 总结" >> "$REPORT_FILE"
echo "- 总错误: $ERRORS" >> "$REPORT_FILE"
echo "- 总警告: $WARNINGS" >> "$REPORT_FILE"

if [ "$ERRORS" -gt 0 ]; then
    echo "**结果: FAILED**" >> "$REPORT_FILE"
    exit 1
else
    echo "**结果: PASSED**" >> "$REPORT_FILE"
fi
```

失败策略：
- 阻断合并：checkpatch ERROR、sparse 地址空间错误、smatch 内存安全
- 允许失败：checkpatch WARNING（风格问题）、sparse 一般警告

**综合实践题二参考答案：**

测试盲区分析：
- amdgpu_dpm.c (45%)：电源管理状态切换难以测试
- amdgpu_ras.c (40%)：错误注入需要特殊硬件支持

测试改进计划：
1. amdgpu_dpm.c：增加 sysfs 接口测试、温度阈值边界测试
2. amdgpu_ras.c：增加 debugfs 错误注入测试、模拟 ECC 错误
3. amdgpu_ttm.c：增加 BO 迁移失败测试、内存压力测试

CI 门禁：
- 行覆盖 < 50%：阻断合并
- 分支覆盖下降 > 5%：警告
- 新代码必须 > 80% 覆盖
