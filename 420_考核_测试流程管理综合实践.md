## 参考答案

### 第一部分：选择题

1. **B** - 编译与静态分析是 CI 流水线的第一阶段，用于快速发现代码问题
2. **B** - GitLab CI 使用 `tags:` 关键字匹配 Runner 标签
3. **B** - 组合爆炸指多维度组合导致指数级增长的测试组合数
4. **B** - 正交数组 + 风险驱动选择是科学的矩阵精简方法
5. **B** - 环境快照用于保存和恢复测试环境状态
6. **B** - GPU 驱动测试需要真实硬件，不能完全容器化
7. **B** - Mesa 使用 LAVA 调度硬件测试
8. **C** - 冒烟测试应在 5-15 分钟内完成
9. **C** - JUnit XML 是 CI 系统广泛支持的测试结果格式
10. **B** - Ubuntu 22.04 LTS 是 ROCm 的主要测试平台

### 第二部分：判断题

11. **错误**。GPU 驱动测试需要真实硬件执行，虽然部分编译和静态分析可以在容器中完成，但功能测试、性能测试必须在真实 GPU 上运行。

12. **正确**。正交数组是一种组合测试方法，通过精心设计确保每对参数值至少出现一次，从而大幅减少测试组合数。

13. **错误**。压力测试执行时间长（通常 24-72 小时），不适合每次提交触发，应在每日构建或发布前执行。

14. **错误**。环境快照可以保存系统配置、modprobe 配置、X11 配置等，但内核版本和驱动版本通常需要手动恢复或通过 PXE 重新部署。

15. **正确**。`artifacts` 关键字用于定义作业的输出产物，可以在后续作业中使用或下载。

### 第三部分：简答题

**16. GPU 驱动 CI 流水线典型阶段：**

| 阶段 | 内容 | 质量门禁 |
|------|------|---------|
| 编译检查 | 内核编译、静态分析 | 编译零错误、静态分析无新增警告 |
| 冒烟测试 | 驱动加载、设备检测、基本渲染 | 100% 通过 |
| 功能回归 | IGT 核心测试、KMS 测试 | >95% 通过 |
| 性能测试 | 基准测试、性能对比 | 退化 <5% |
| 压力测试 | 长时间运行、内存压力 | 无 hang/崩溃 |

**17. 硬件 Runner 池：**

硬件 Runner 池是一组配备真实 GPU 的物理测试机，通过 CI 系统统一调度。GPU 驱动测试需要真实硬件执行，无法完全虚拟化。Runner 池需要设备调度系统管理设备的借用、归还、健康检查和状态清理。

**18. 测试矩阵精简策略：**

- **正交数组**：适用于参数组合多但预算有限的情况，确保每对参数至少出现一次
- **风险驱动选择**：优先覆盖高风险组合，如新硬件+新软件
- **等价类划分**：将相似硬件/软件归为等价类，每类选一个代表

**19. 环境快照关键信息：**

- 操作系统版本和配置
- 内核版本和配置
- 已安装的驱动和库版本
- modprobe 和 X11 配置
- 系统环境变量

恢复流程：恢复快照配置 → 验证内核版本 → 验证驱动版本 → 运行健康检查 → 标记可用

**20. 处理 flaky 测试：**

- 引入自动重试机制（最多 3 次）
- 对 flaky 测试进行标记和隔离
- 分析 flaky 根因（时序、温度、并发）
- 建立 flaky 测试数据库跟踪稳定性趋势
- 设置单独的 flaky 测试作业，不阻断主流水线

### 第四部分：综合实践题

**综合实践题一参考答案：**

```yaml
# .gitlab-ci.yml
stages:
  - build
  - analyze
  - smoke
  - regression

variables:
  KERNEL_CONFIG: "amd_gpu_defconfig"

build_kernel:
  stage: build
  image: kernel-build-env:latest
  script:
    - make $KERNEL_CONFIG
    - make -j$(nproc)
  artifacts:
    paths:
      - arch/x86/boot/bzImage
      - drivers/gpu/drm/amd/*.ko
    expire_in: 1 day
  tags:
    - docker

static_analysis:
  stage: analyze
  image: kernel-build-env:latest
  script:
    - make C=1 CHECK="sparse" drivers/gpu/drm/amd/
    - scripts/checkpatch.pl --strict -g HEAD
  tags:
    - docker
  allow_failure: true

smoke_test:
  stage: smoke
  needs: [build_kernel]
  tags:
    - gpu-amd
    - baremetal
  script:
    - sudo insmod drivers/gpu/drm/amd/amdgpu.ko
    - test -c /dev/dri/card0
    - kmscube --drm-device=/dev/dri/card0
  timeout: 15 minutes

igt_regression:
  stage: regression
  needs: [smoke_test]
  tags:
    - gpu-amd
    - baremetal
  script:
    - ./igt-runner --test-list quick-tests.txt --output-format junit
  artifacts:
    reports:
      junit: results.xml
  timeout: 2 hours
```

硬件矩阵：
- RX 7900 XTX (RDNA3, 最新架构)
- RX 6950 XT (RDNA2, 主流架构)
- MI250X (CDNA2, 计算卡)

**综合实践题二参考答案：**

测试矩阵（使用正交数组精简）：

| 组合 | GPU | 操作系统 | 测试重点 |
|------|-----|---------|---------|
| 1 | RX 7900 XTX | Ubuntu 24.04 | 最新硬件+最新系统 |
| 2 | RX 7900 XTX | Fedora 40 | 最新硬件+替代系统 |
| 3 | RX 6950 XT | Ubuntu 22.04 | 主流硬件+稳定系统 |
| 4 | RX 6750 XT | Fedora 39 | 中端硬件+替代系统 |
| 5 | MI250X | Ubuntu 24.04 | 计算卡+最新系统 |
| 6 | MI100 | Ubuntu 22.04 | 旧计算卡+稳定系统 |
| 7 | Ryzen APU | Fedora 40 | 集成显卡+最新系统 |

精简理由：
- 每对 GPU-OS 组合至少出现一次
- 优先覆盖最新和主流组合
- 计算卡和集成显卡有特殊测试需求

健康检查脚本：

```bash
#!/bin/bash
# check_lab_health.sh

MACHINES=("lab-01" "lab-02" "lab-03" "lab-04" "lab-05" "lab-06")

for host in "${MACHINES[@]}"; do
    echo "=== $host ==="
    ssh "$host" '
        echo "GPU: $(lspci -nn | grep 1002: | head -1)"
        if [ -f /sys/class/drm/card0/device/hwmon/hwmon0/temp1_input ]; then
            temp=$(cat /sys/class/drm/card0/device/hwmon/hwmon0/temp1_input)
            echo "温度: $((temp/1000))°C"
        fi
        echo "驱动: $(lsmod | grep amdgpu | head -1)"
        echo "状态: $(systemctl is-active graphical.target)"
    '
done
```
