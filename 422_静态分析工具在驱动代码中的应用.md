# 静态分析工具在驱动代码中的应用

## 学习目标

- 理解静态分析在 GPU 驱动代码质量保证中的作用
- 掌握 sparse、smatch、Coccinelle 等内核静态分析工具的使用方法
- 学会解读静态分析报告并优先处理高价值警告
- 了解静态分析与动态测试的互补关系
- 能够将静态分析集成到 CI 流水线中

## 知识详解

### 一、概念原理

#### 1.1 静态分析在 GPU 驱动测试中的定位

静态分析（Static Analysis）是在不执行代码的情况下检查代码错误的技术。在 GPU 驱动测试中，它与动态测试形成互补：

```
┌─────────────────────────────────────────────────────────────────────┐
│              静态分析与动态测试的互补关系                            │
│                                                                     │
│   静态分析                          动态测试                         │
│   ─────────────                     ─────────────                   │
│                                                                     │
│   ● 不执行代码                      ● 需要运行代码                   │
│   ● 发现语法和逻辑错误              ● 发现运行时行为问题             │
│   ● 覆盖所有代码路径                ● 覆盖实际执行路径               │
│   ● 快速反馈（秒/分钟级）           ● 慢速反馈（分钟/小时级）       │
│   ● 可能产生误报                    ● 结果准确但覆盖有限           │
│                                                                     │
│   适用场景：                        适用场景：                       │
│   ├── 空指针解引用                  ├── 性能回归                   │
│   ├── 内存泄露（部分）              ├── 并发竞态                   │
│   ├── 类型不匹配                    ├── 硬件交互问题               │
│   ├── 未初始化变量                  ├── 功耗异常                   │
│   └── 死锁潜在风险                  └── 渲染错误                   │
│                                                                     │
│   GPU 驱动中的价值：                                                │
│   ┌─────────────────────────────────────────────────────────┐      │
│   │  内核代码路径复杂，静态分析可覆盖错误处理等难测试路径    │      │
│   │  硬件交互代码难以 mock，静态分析提供不依赖硬件的检查    │      │
│   │  驱动代码权限高（内核态），静态分析可提前发现安全问题   │      │
│   └─────────────────────────────────────────────────────────┘      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 二、实践操作

#### 2.1 sparse 工具使用

sparse 是 Linus Torvalds 编写的内核专用类型检查工具：

```bash
#!/bin/bash
# sparse_analysis.sh - sparse 静态分析脚本
# 参考：内核文档 Documentation/dev-tools/sparse.rst

set -e

KERNEL_DIR="${1:-.}"
cd "$KERNEL_DIR"

echo "=== sparse 静态分析 ==="
echo "分析目录: $KERNEL_DIR"
echo ""

# 1. 基本类型检查
echo "## 1. 基本类型检查 (C=1)"
echo "```"
make C=1 CHECK="sparse" drivers/gpu/drm/amd/amdgpu/ 2>&1 | \
    grep -E "warning:|error:" | head -50 || true
echo "```"
echo ""

# 2. 详细检查（包含指针别名分析）
echo "## 2. 详细检查 (C=2)"
echo "```"
make C=2 CHECK="sparse" drivers/gpu/drm/amd/amdgpu/amdgpu_device.c 2>&1 | \
    grep -E "warning:|error:" | head -30 || true
echo "```"
echo ""

# 3. 检查 __user/__iomem/__force 属性使用
echo "## 3. 地址空间属性检查"
echo "```"
# 检查缺少 __iomem 标记的 MMIO 访问
sparse -D__KERNEL__ -DCONFIG_AMDGPU \
    -Iinclude -Iarch/x86/include \
    drivers/gpu/drm/amd/amdgpu/amdgpu_regs.c 2>&1 | \
    grep "warning:" | head -20 || true
echo "```"
```

sparse 核心检查项示例：

```c
// sparse 检查示例：地址空间不匹配

// ❌ 问题：用户空间指针未标记 __user
int amdgpu_ioctl(struct drm_device *dev, void *data, struct drm_file *filp) {
    struct amdgpu_bo *bo;
    void *user_ptr = data;  // 应该标记为 __user
    
    // sparse 警告：将 __user 指针赋值给普通指针
    bo = user_ptr;  // WARNING: incompatible types
}

// ✅ 正确：使用 __user 标记
int amdgpu_ioctl(struct drm_device *dev, void __user *data,
                 struct drm_file *filp) {
    struct amdgpu_bo *bo;
    void __user *user_ptr = data;
    
    // 使用 copy_from_user/copy_to_user 访问用户空间数据
    if (copy_from_user(&bo, user_ptr, sizeof(bo)))
        return -EFAULT;
}

// ❌ 问题：MMIO 地址未标记 __iomem
void amdgpu_write_reg(struct amdgpu_device *adev, u32 reg, u32 val) {
    u32 *reg_ptr = adev->rmmio + reg;  // 应该标记为 __iomem
    *reg_ptr = val;  // sparse 警告
}

// ✅ 正确：使用 __iomem 标记
void amdgpu_write_reg(struct amdgpu_device *adev, u32 reg, u32 val) {
    void __iomem *reg_ptr = adev->rmmio + reg;
    iowrite32(val, reg_ptr);  // 使用专用 MMIO 访问函数
}
```

#### 2.2 smatch 工具使用

smatch 是内核社区开发的高级静态分析工具，支持数据流和状态机分析：

```bash
#!/bin/bash
# smatch_analysis.sh - smatch 静态分析脚本
# 参考：https://repo.or.cz/w/smatch.git

set -e

KERNEL_DIR="${1:-.}"
cd "$KERNEL_DIR"

echo "=== smatch 静态分析 ==="
echo ""

# 1. 生成 smatch 数据库
echo "## 1. 生成 smatch 数据库"
echo "```"
make CHECK="smatch --project=kernel" -j$(nproc) \
    drivers/gpu/drm/amd/amdgpu/ 2>&1 | tail -20
echo "```"
echo ""

# 2. 检查内存泄露
echo "## 2. 内存泄露检查"
echo "```"
smatch --project=kernel -p=memory \
    drivers/gpu/drm/amd/amdgpu/*.c 2>&1 | \
    grep -E "memory leak|unfreed" | head -20 || true
echo "```"
echo ""

# 3. 检查空指针解引用
echo "## 3. 空指针检查"
echo "```"
smatch --project=kernel -p=null \
    drivers/gpu/drm/amd/amdgpu/*.c 2>&1 | \
    grep -E "null dereference|dereference of NULL" | head -20 || true
echo "```"
echo ""

# 4. 检查锁配对
echo "## 4. 锁配对检查"
echo "```"
smatch --project=kernel -p=locking \
    drivers/gpu/drm/amd/amdgpu/*.c 2>&1 | \
    grep -E "unlocked|double lock|missing unlock" | head -20 || true
echo "```"
```

smatch 检查示例：

```c
// smatch 检查示例：返回值未检查

// ❌ 问题：返回值未检查
void amdgpu_init_power(struct amdgpu_device *adev) {
    amdgpu_dpm_init(adev);  // smatch: 返回值被忽略
    amdgpu_hwmon_init(adev);  // smatch: 返回值被忽略
}

// ✅ 正确：检查返回值
int amdgpu_init_power(struct amdgpu_device *adev) {
    int r;
    
    r = amdgpu_dpm_init(adev);
    if (r) {
        DRM_ERROR("DPM init failed: %d\n", r);
        return r;
    }
    
    r = amdgpu_hwmon_init(adev);
    if (r) {
        DRM_ERROR("hwmon init failed: %d\n", r);
        amdgpu_dpm_fini(adev);
        return r;
    }
    
    return 0;
}

// smatch 检查示例：锁未释放

// ❌ 问题：错误路径未解锁
void amdgpu_process_irq(struct amdgpu_device *adev) {
    spin_lock(&adev->irq_lock);
    
    if (!adev->irq.enabled) {
        return;  // smatch: 锁 'adev->irq_lock' 未释放
    }
    
    // 处理中断...
    
    spin_unlock(&adev->irq_lock);
}

// ✅ 正确：确保所有路径都解锁
void amdgpu_process_irq(struct amdgpu_device *adev) {
    spin_lock(&adev->irq_lock);
    
    if (!adev->irq.enabled) {
        spin_unlock(&adev->irq_lock);  // 释放锁
        return;
    }
    
    // 处理中断...
    
    spin_unlock(&adev->irq_lock);
}
```

#### 2.3 Coccinelle 语义补丁

Coccinelle 用于编写"语义补丁"，自动检测和修改代码模式：

```bash
#!/bin/bash
# coccinelle_check.sh - Coccinelle 语义补丁检查
# 参考：https://coccinelle.gitlabpages.inria.fr/website/

set -e

KERNEL_DIR="${1:-.}"
cd "$KERNEL_DIR"

echo "=== Coccinelle 语义补丁检查 ==="
echo ""

# 1. 检查 kzalloc 返回值未检查
echo "## 1. kzalloc 返回值检查"
cat > /tmp/check_kzalloc.cocci << 'EOF'
@r@
expression E;
position p;
@@

E = kzalloc@p(...);
... when != E == NULL
    when != IS_ERR(E)
    when != !E

@script:python@
p << r.p;
@@

print("kzalloc 返回值未检查: %s" % (p[0].file))
EOF

spatch --sp-file /tmp/check_kzalloc.cocci \
    drivers/gpu/drm/amd/amdgpu/ 2>/dev/null | head -20 || true
echo ""

# 2. 检查 devm_ 函数返回值未检查
echo "## 2. devm_ 函数返回值检查"
cat > /tmp/check_devm.cocci << 'EOF'
@r@
expression E;
position p;
@@

E = devm_kzalloc@p(...);
... when != E == NULL
    when != !E

@script:python@
p << r.p;
@@

print("devm_kzalloc 返回值未检查: %s" % (p[0].file))
EOF

spatch --sp-file /tmp/check_devm.cocci \
    drivers/gpu/drm/amd/amdgpu/ 2>/dev/null | head -20 || true
```

### 三、案例分析

#### 3.1 真实案例：通过 sparse 发现 AMDGPU uAPI 问题

**背景**：审查 amdgpu ioctl 接口时发现用户空间指针未正确标记。

```c
// 原始代码
struct drm_amdgpu_info *args = data;

// sparse 报告：
// warning: incorrect type in assignment (different address spaces)
//    expected struct drm_amdgpu_info *args
//    got void [noderef] <asn:1>*data

// 修复：添加 __user 标记
struct drm_amdgpu_info __user *args = data;

// 后续访问必须使用 copy_from_user/copy_to_user
if (copy_from_user(&args_copy, args, sizeof(args_copy)))
    return -EFAULT;
```

**测试启示**：
- 此问题难以通过动态测试发现（可能在某些架构上工作正常）
- sparse 在编译阶段即可捕获此类跨地址空间访问问题
- 应在 CI 中强制要求 sparse 零警告

#### 3.2 真实案例：通过 smatch 发现内存泄露

**背景**：amdgpu_job_free 中未释放 fence 引用。

```c
// smatch 报告：
// drivers/gpu/drm/amd/amdgpu/amdgpu_job.c:123
// amdgpu_job_free() warn: 'fence' not released on lines: 123

// 修复前
void amdgpu_job_free(struct drm_sched_job *sched_job) {
    struct amdgpu_job *job = to_amdgpu_job(sched_job);
    
    if (job->fence)  // 检查但可能未释放
        return;
    
    dma_fence_put(job->fence);  // 释放 fence
    kfree(job);
}

// 修复后
void amdgpu_job_free(struct drm_sched_job *sched_job) {
    struct amdgpu_job *job = to_amdgpu_job(sched_job);
    
    if (job->fence) {
        dma_fence_put(job->fence);  // 确保释放
        job->fence = NULL;
    }
    
    kfree(job);
}
```

## 今日小结

- 静态分析与动态测试互补，可在不执行代码的情况下发现内存安全、类型错误等问题
- sparse 专注于内核地址空间检查（__user/__iomem/__force）
- smatch 提供数据流和状态机分析，可发现内存泄露、空指针、锁配对问题
- Coccinelle 通过语义补丁实现代码模式检测和自动重构
- 静态分析工具应集成到 CI 流水线，作为代码合并的门禁条件
