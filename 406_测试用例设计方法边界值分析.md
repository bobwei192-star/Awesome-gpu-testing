# Day 406: 测试用例设计方法——边界值分析

## 1. 学习目标

- 理解边界值分析（Boundary Value Analysis, BVA）的理论基础和数学原理
- 掌握边界值分析的三种变体：标准BVA、健壮BVA和最坏情况BVA

- 理解等价类划分与边界值分析的协同使用策略

## 2. 边界值分析理论基础

### 2.1 核心概念

边界值分析是基于大量实践经验总结出的测试设计方法——软件缺陷往往集中在输入域的边界附近而非内部区域。这一发现源于程序实现中的常见错误模式：

```
常见边界缺陷类型：
─────────────────────────────────────────────────
1. 边界比较运算符错误：  <  和  <=  混淆
   示例：使用 `if (value < 100)` 代替 `if (value <= 100)`
   
2. 偏移量计算错误：  size - 1  忘记减一
   示例：访问 `array[size]` 而不是 `array[size - 1]`

3. 循环边界错误：  for(i = 0; i < n; i++)  写成  for(i = 0; i <= n; i++)
   示例：多循环一次导致缓冲区溢出

4. 数据类型溢出：  int 范围外的值
   示例：`int count = 2147483647; count++` 导致回绕

5. 空值/零值处理错误：除零、空指针
   示例：`int avg = sum / count` 当 `count == 0` 时崩溃
```

### 2.2 数学定义

对于一个取值范围为 `[min, max]` 的数值型变量，边界值分析定义的测试点为：

```
标准边界值（Standard BVA）：
────────────────────────────────
  下边界：min
  上边界：max
  上边界邻值：max + 1 (超出范围)
  下边界邻值：min - 1 (超出范围)
  内部代表值：(min + max) / 2 附近的正常值

可视化表示（范围 [10, 100]）：
  
  无效区          有效区              无效区
  ──────┬────────┬─────────────────────┬──────────────
        │        │                     │
        │        │                     │
       9        10         ...        100      101
   (min-1)   (min)                    (max)   (max+1)
  
  需要测试的5个标准值：10, 11, 55, 99, 100  （单位测试）
  或 7个值：9, 10, 11, 55, 99, 100, 101  （健壮测试）
```

### 2.3 三种变体详细对比

#### 标准BVA（Normal BVA）

假设变量 `x` 在范围 `[L, R]` 内：

```
测试用例 = { L, L+ϵ, R-ϵ, R, (L+R)/2 }
其中 ϵ 为最小步长/精度
```

**适用场景**：输入高度可靠，无需考虑异常值测试

```
     L     L+ϵ         (L+R)/2        R-ϵ    R
     │      │             │             │     │
     ▼      ▼             ▼             ▼     ▼
  ───────────────────────────────────────────────────▶  变量取值
```

#### 健壮BVA（Robust BVA）

```
测试用例 = { L-ϵ, L, L+ϵ, R-ϵ, R, R+ϵ, (L+R)/2 }
增加超出边界的测试值
```

**适用场景**：输入可能因外部因素超出预期范围（如传感器噪声、用户误操作）

```
  L-ϵ     L     L+ϵ     (L+R)/2    R-ϵ     R     R+ϵ
   │      │      │         │         │      │      │
   ▼      ▼      ▼         ▼         ▼      ▼      ▼
──┼──────┼──────┼─────────┼─────────┼──────┼──────┼──▶  变量取值
  异常区  正常区                             异常区
```

#### 最坏情况BVA（Worst-Case BVA）

对于n个变量，标准BVA生成 `(4n+1)` 个测试用例，最坏情况BVA生成 `5^n` 个测试用例（考虑所有组合）：

```
n=1: 5个测试用例（标准BVA同样为5个）
n=2: 25个测试用例（标准BVA为9个）
n=3: 125个测试用例（标准BVA为13个）
n=4: 625个测试用例（标准BVA为17个）

适用场景：需要高可靠性的安全关键系统
代价：测试用例数量呈指数增长
```

### 2.4 多维边界值分析

当输入由多个变量组成时，需要考虑多维边界：

```
二维边界示例：分辨率设置 (width × height)

height
  ▲
  │    (640,4320)      (7680,4320)
  │        ┌─────────────────────┐
  │        │                     │
  │        │    有效区域         │
  │        │                     │
  │        └─────────────────────┘
  │    (640,480)       (7680,480)
  └──────────────────────────────────────▶ width

边界线：width ∈ {640, 7680}, height ∈ {480, 4320}
角点：4个角点 + 4条边上的其他边界点
```

  写指针 (w)

边界测试：
  r == w       : ring为空（初始状态）
  w + 1 == r   : ring满（单元素空洞避免混淆）
  w + 1023 == r: ring几乎满
  w回绕到0     : 写指针跨越ring末尾
  r == 0且w==1023: 读指针重置边界
```

```

1. 固件更新改变底层阈值

典型碎片化测试代码（Python）：
```python
def test_vram_fragmentation_boundary(drm_fd, vram_size):
    """
    测试VRAM碎片化后的分配边界
    """
    handles = []
    # 阶段1：分配大量中等大小BO
    alloc_size = vram_size // 100  # 每块1%
    for i in range(50):
        handle = gem_create(drm_fd, alloc_size)
        handles.append(handle)
    
    # 阶段2：释放间隔的块（制造碎片）
    for i in range(1, 50, 2):
        gem_close(drm_fd, handles[i])
        handles[i] = None
    
    # 阶段3：测试分配边界
    # 尝试分配接近总VRAM的大块
    large_size = vram_size // 2
    try:
        handle = gem_create(drm_fd, large_size)
        # 如果成功，说明分配器支持碎片合并
        print(f"FRAG-BOUNDARY: large alloc {large_size} SUCCESS")
        gem_close(drm_fd, handle)
    except OSError as e:
        # 如果失败，说明碎片化影响了上界
        print(f"FRAG-BOUNDARY: large alloc {large_size} FAILED ({e})")
    
    # 清理
    for h in handles:
        if h is not None:
            gem_close(drm_fd, h)
```

### 8.2 Python边界值生成示例

```python
#!/usr/bin/env python3
"""
GPU驱动参数边界值生成器
"""
import itertools
from typing import List, Tuple, Any

class BVAGenerator:
    def __init__(self, min_val: float, max_val: float, step: float = 1):
        self.min_val = min_val
        self.max_val = max_val
        self.step = step
    
    def standard_bva(self) -> List[float]:
        """标准BVA: 5个值"""
        mid = (self.min_val + self.max_val) / 2
        return [
            self.min_val,
            self.min_val + self.step,
            mid,
            self.max_val - self.step,
            self.max_val
        ]
    
    def robust_bva(self) -> List[float]:
        """健壮BVA: 7个值"""
        return [
            self.min_val - self.step,
            self.min_val,
            self.min_val + self.step,
            (self.min_val + self.max_val) / 2,
            self.max_val - self.step,
            self.max_val,
            self.max_val + self.step
        ]
    
    def worst_case_bva(self, other_vars: List['BVAGenerator']) -> List[Tuple]:
        """最坏情况BVA: 所有变量边界的笛卡尔积"""
        my_bounds = self.standard_bva()
        all_bounds = [my_bounds] + [v.standard_bva() for v in other_vars]
        return list(itertools.product(*all_bounds))

# 示例使用
freq = BVAGenerator(500, 3000, 1)
volt = BVAGenerator(500, 1200, 1)

standard = freq.standard_bva()
print(f"GFXCLK 标准BVA: {standard}")

robust = freq.robust_bva()
print(f"GFXCLK 健壮BVA: {robust}")

# 频率和电压的最坏情况BVA
worst = freq.worst_case_bva([volt])
print(f"频率 x 电压 最坏情况BVA: {len(worst)} 个测试用例")
for f, v in worst[:5]:
    print(f"  freq={f}MHz, volt={v}mV")
```

### 8.5 BVA结果分析仪表盘

使用Python生成可视化的边界值测试趋势分析：

```python
#!/usr/bin/env python3
"""
BVA测试结果分析器 —— 检测边界值漂移
"""
import json
import sys
from datetime import datetime
from collections import defaultdict

class BVATrendAnalyzer:
    def __init__(self, history_file: str):
        with open(history_file) as f:
            self.history = json.load(f)
    
    def detect_drift(self, param: str, boundary: str, window: int = 10):
        """
        检测边界值漂移
        param: 参数名
        boundary: 边界类型 (min/max)
        window: 滑动窗口大小
        """
        runs = [r for r in self.history 
                if r['param'] == param and r['boundary'] == boundary]
        runs = sorted(runs, key=lambda r: r['timestamp'])[-window:]
        
        if len(runs) < 3:
            return {"drift_detected": False, "reason": "数据点不足"}
        
        # 计算均值和标准差
        values = [r['actual_value'] for r in runs]
        mean = sum(values) / len(values)
        std = (sum((v - mean) ** 2 for v in values) / len(values)) ** 0.5
        
        # 检测趋势
        recent = values[-3:]
        recent_mean = sum(recent) / len(recent)
        
        drift_ratio = abs(recent_mean - mean) / mean if mean != 0 else 0
        
        return {
            "drift_detected": drift_ratio > 0.05,  # 超过5%视为漂移
            "drift_ratio": f"{drift_ratio*100:.1f}%",
            "historical_mean": mean,
            "recent_mean": recent_mean,
            "std": std,
            "data_points": len(runs)
        }
    
    def generate_report(self):
        report = {
            "generated_at": datetime.now().isoformat(),
            "summary": {"total_params": 0, "drifted_params": 0},
            "params": []
        }
        
        params = set((r['param'], r['boundary']) for r in self.history)
        report['summary']['total_params'] = len(params)
        
        for param, boundary in params:
            drift = self.detect_drift(param, boundary)
            if drift['drift_detected']:
                report['summary']['drifted_params'] += 1
                report['params'].append({
                    "param": param,
                    "boundary": boundary,
                    "drift_ratio": drift['drift_ratio'],
                    "historical_mean": drift['historical_mean'],
                    "recent_mean": drift['recent_mean']
                })
        
        return report

# 使用示例
analyzer = BVATrendAnalyzer("bva_history.json")
report = analyzer.generate_report()
print(json.dumps(report, indent=2))

# 输出漂移预警
if report['summary']['drifted_params'] > 0:
    print(f"\n⚠ 检测到 {report['summary']['drifted_params']} 个边界值漂移!")
    for p in report['params']:
        print(f"  参数 {p['param']} ({p['boundary']}): "
              f"漂移 {p['drift_ratio']}, "
              f"历史均值 {p['historical_mean']:.1f} → 近期均值 {p['recent_mean']:.1f}")
```

 * IGT边界值测试示例
 * 测试gem_create的BO大小边界
 */
#include "igt.h"

#define MIN_BO_SIZE 4096      /* 最小BO大小：1页 */
#define MAX_BO_SIZE (4ULL * 1024 * 1024 * 1024)  /* 最大BO大小：4GB */

static void test_bo_boundary(int fd)
{
    uint64_t sizes[] = {
        /* 标准BVA测试值 */
        MIN_BO_SIZE,           /* 最小值 */
        MIN_BO_SIZE + 1,       /* 最小值+1（非页对齐） */
        MIN_BO_SIZE * 2,       /* 2页 */
        (MIN_BO_SIZE + MAX_BO_SIZE) / 2,  /* 中间值 */
        MAX_BO_SIZE - 4096,    /* 最大值-1页 */
        MAX_BO_SIZE,           /* 最大值 */
        /* 健壮BVA增加的测试值 */
        MIN_BO_SIZE - 1,       /* 最小值-1（无效） */
        MAX_BO_SIZE + 4096,    /* 最大值+1（无效） */
    };
    
    for (int i = 0; i < ARRAY_SIZE(sizes); i++) {
        uint32_t handle;
        int ret;
        
        ret = gem_create(fd, sizes[i], &handle);
        
        if (sizes[i] < MIN_BO_SIZE || sizes[i] > MAX_BO_SIZE) {
            /* 无效值应该失败 */
            igt_assert_eq(ret, -EINVAL);
            igt_info("BVA-PASS: size=%lu correctly rejected (errno=%d)\n",
                     sizes[i], errno);
        } else {
            /* 有效值应该成功 */
            igt_assert_eq(ret, 0);
            gem_close(fd, handle);
            igt_info("BVA-PASS: size=%lu allocated successfully\n", sizes[i]);
        }
    }
}

igt_main

        fd = drm_open_driver(DRIVER_AMDGPU);
        igt_require(fd >= 0);
    }
    
    igt_subtest("bo-boundary-value-analysis")
        test_bo_boundary(fd);

        close(fd);

## 10. 边界值分析的局限性与应对

### 10.1 局限性

```
1. 一维边界局限性
   └── 仅关注单一参数的边界，忽略参数间交互
   └── 应对：结合决策表和多维BVA

2. 内部边界盲区
   └── 只关注输入域边界，不覆盖代码内部边界
   └── 应对：结合白盒测试和覆盖率分析

3. 非数值参数适应性差
   └── 对于字符串、枚举等类型需要特殊处理
   └── 应对：使用枚举边界和字符串长度边界

4. 状态依赖边界遗漏
   └── 不考虑系统状态对边界值的影响
   └── 应对：结合状态转换图测试

5. 组合爆炸
   └── 多参数最坏情况BVA导致测试用例数量爆炸
   └── 应对：使用优先级和风险排序
```

### 10.2 优化策略

```
测试用例数量优化策略：

1. 单缺陷假设
   └── 假设缺陷由单个边界触发，而非多个边界组合
   └── 减少：5^n → 4n+1

2. 边界值优先级排序
   ┌──────────┬─────────────────────────────────┐
   │ 优先级    │ 条件                            │
   ├──────────┼─────────────────────────────────┤
   │ P0       │ 直接导致崩溃/数据损坏的边界      │
   │ P1       │ 导致功能异常的边界              │
   │ P2       │ 性能降级相关的边界              │
   │ P3       │ 边缘/罕见场景的边界             │
   └──────────┴─────────────────────────────────┘

3. 自适应边界采样
   └── 根据历史缺陷分布调整测试密度
   └── 高缺陷密度区域增加测试点

4. 等价类内BVA
   └── 仅在等价类边界附近进行BVA
   └── 等价类内部使用代表值测试

5. 变粒度BVA
   └── 关键参数使用最坏情况BVA
   └── 次要参数使用标准BVA
   └── 非关键参数仅做边界确认
```

## 11. 真实Bug案例分析：边界值触发的GPU驱动缺陷

### 11.1 freedesktop.org Bugzilla #109241 —— GEM_CREAT边界值缺陷

该Bug报告了在intel集成显卡上，gem_create的size参数边界处理缺陷，类似的逻辑问题同样存在于AMDGPU驱动中。

**缺陷描述**：
```
Bug标题: gem_create with size = 0 does not return -EINVAL
Bug链接: https://bugs.freedesktop.org/show_bug.cgi?id=109241
影响驱动: i915 (同类问题也影响amdgpu)
严重程度: 中等
```

**边界值分析**：
```
gem_create size参数规范：
  ┌────────────────────────────────────────────────────┐
  │ 有效范围: [4096, max_bo_size]                      │
  │ 返回值:                                            │
  │   成功: 返回0并通过handle参数返回句柄               │
  │   无效参数: 返回-EINVAL                            │
  │   内存不足: 返回-ENOMEM                            │
  └────────────────────────────────────────────────────┘

缺陷触发条件：
  size = 0 时的代码路径分析：

  amdgpu_gem_create_ioctl()
    ├── 检查 size == 0 → 是
    │   └── 预期行为: 返回 -EINVAL
    │   └── 实际行为: 继续执行(未检查)
    │       └── ttm_bo_init() 接收 size=0
    │           └── ttm_tt_create() 分配0字节 → 成功
    │           └── drm_vma_node 初始化 size=0
    │               └── 后续mmap访问导致page fault
  
    └── 修复方案:
        └── 在入口处添加 size == 0 检查
```

**边界测试缺失**：
```
回归测试中未覆盖的边界值：
  测试集       | 覆盖的size值
  ─────────────┼────────────────────
  等价类测试    | 4096, 2MB, 1GB    ← 只测试了有效值
  边界值测试    | 缺失!             ← 需要增加 size=0, size=4095
  ─────────────┼────────────────────
  正确BVA覆盖    | 0, 4095, 4096,    ← size=0是最关键的边界
  应包含        | 4097, max-1, max
  
教训: 如果当时有完整的BVA测试，size=0的边界缺陷可以提前发现
```

### 11.2 freedesktop.org Bugzilla #101409 —— 分辨率切换中的边界crash

**缺陷描述**：
```
Bug标题: [Regression] Switching display mode causes NULL pointer dereference
Bug链接: https://bugs.freedesktop.org/show_bug.cgi?id=101409
影响驱动: amdgpu (DC display core)
严重程度: 高 (内核崩溃)
```

**边界分析**：
```
在分辨率切换过程中，某个内部数据结构在边界条件下未被正确初始化：

display_mode结构体的timing参数边界：
┌────────────────┬──────────┬──────────┬────────────────────┐
│ timing参数      │ 最小值    │ 最大值    │ 缺陷相关            │
├────────────────┼──────────┼──────────┼────────────────────┤
│ htotal         │ 1        │ 65535    │ ≤ min临界值         │
│ vtotal         │ 1        │ 65535    │ ≤ min临界值         │
│ hdisplay       │ 1        │ 16384    │ 可能导致零除法       │
│ vdisplay       │ 1        │ 16384    │ 可能导致零除法       │
│ pixel_clock    │ 1        │ 600000    │ 极低时除零          │
└────────────────┴──────────┴──────────┴────────────────────┘

问题根因：
  在某个边界分辨率设置下，timing的hdisplay被设置为0
  → dc_validate_stream() 未检查 hdisplay == 0
  → 后续计算: width = stream->timing.hdisplay
  → 用于除法运算:  pitch = width / 4  (除零!)
  → 内核崩溃 (divide error)

边界值分析回顾：
  有效的分辨率先验知识要求 width > 0 且 height > 0
  BVA测试应包含：width = {0, 1, 2, 16383, 16384, 16385}
  缺陷在 width=0 的边界处触发
```

### 11.3 边界值驱动的测试改进建议

基于上述真实Bug案例，总结边界值测试的改进方向：

```
核心改进措施：

1. 入口参数检查完整性
   对所有ioctl入口参数进行完整边界检查：
   ┌──────────────────┬────────────────────────────────────┐
   │ 参数类型          │ 必须检查的边界                      │
   ├──────────────────┼────────────────────────────────────┤
   │ 大小/长度         │ 0, min-1, min, max, max+1         │
   │ 索引/计数器       │ -1, 0, max, max+1                 │
   │ 指针              │ NULL, 有效地址, 非法地址            │
   │ 枚举/标志         │ 0, 最大值, 中间值, 非法值           │
   │ 位域              │ 全0, 全1, 单一bit, 非法组合        │
   └──────────────────┴────────────────────────────────────┘

2. 零值边界专项测试
   ┌────────────────────────────────────────────────────┐
   │ 驱动中常见的零值边界缺陷：                          │
   │  - size == 0 (分配零字节)                          │
   │  - count == 0 (提交零个命令)                       │
   │  - timeout == 0 (等待零时间)                       │
   │  - mode->hdisplay == 0 (零宽度显示)                │
   │  - fence_seqno == 0 (零序号fence)                  │
   │ 每个零值边界都需要专项测试用例                       │
   └────────────────────────────────────────────────────┘

3. 负值/超大值测试
   ┌────────────────────────────────────────────────────┐
   │ 内核空间与用户空间的边界差异：                       │
   │  - 用户空间传入的size可能是(size_t)-1 (即SIZE_MAX)  │
   │  - 有符号参数可能传入负数                           │
   │  - 驱动应检查这些跨空间边界的值                      │
   │ 测试原则：永远不要信任用户空间传入的边界值            │
   └────────────────────────────────────────────────────┘
```

## 13. 今日小结

边界值分析是测试用例设计中最基础、最高效的方法之一。在实践中，我们掌握了：

1. **理论核心**：边界值分析基于"缺陷集中在边界附近"的实践经验，通过系统性地测试输入域的边界值来发现缺陷。

2. **三种变体**：标准BVA（5个值）、健壮BVA（7个值）和最坏情况BVA（多维组合），适用于不同的测试深度需求。

3. **GPU驱动应用**：在频率设置、BO分配、分辨率配置、超时参数、PCIe BAR、fence序号和命令缓冲区等GPU驱动特有的场景中，边界值分析能有效地发现参数处理缺陷。

4. **方法集成**：边界值分析与等价类划分是天生的搭档——等价类划分确定测试范围，边界值分析细化边界测试点。

5. **自动化支持**：通过Shell脚本、Python生成器和CI/CD流水线，可以实现边界测试的全自动化执行和趋势分析。

6. **真实Bug验证**：通过freedesktop.org上的实际Bug案例证实，完善的BVA测试可以有效预防GPU驱动中的边界缺陷。

**关键洞察**：在GPU驱动测试中，边界值分析的价值尤为突出，因为驱动代码中包含了大量与硬件规格相关的边界条件（频率限制、大小限制、超时阈值等）。硬件规格的细微变化都可能导致边界值的偏移，因此边界测试需要在每个硬件迭代中重新执行。同时，GPU驱动的并发特性使得边界条件不仅仅是静态的数值范围，还需要考虑并发访问时的资源争用边界。real-world bug案例表明，最简单的边界（如零值、负值、空值）往往是缺陷最集中的区域，BVA测试应优先覆盖这些基础边界。

## 14. 扩展思考

1. 思考边界值分析在GPU电源管理序列（P-State切换）中的应用——这里的边界不仅是数值边界，还包括时序边界。

2. 考虑如何在AMDGPU驱动的CI/CD流程中自动化边界值测试的生成和执行，包括在GitLab CI中集成前文所述的Shell自动化脚本。

3. 探索边界值分析与故障注入测试的结合：在边界条件下注入典型故障（如内存分配失败、超时），验证驱动的容错能力。

4. 对比分析边界值分析在用户空间驱动组件测试和内核空间驱动测试中的不同应用策略——内核空间的边界缺陷通常导致系统级崩溃，其测试优先级应更高。

5. 研究如何使用KUnit或kselftest框架编写内核级别的边界值单元测试，将BVA测试左移到开发阶段。
