### 第二部分：简答题（20分）

**11. ATIF vs ATCS（5分）**
- ATIF（AMD Technology Interface）负责显示相关的 ACPI 事件处理（背光控制、显示切换、GPU 状态查询）（2分）
- ATCS（AMD Technology Control Services）负责系统级别的电源控制（AC/DC 切换、散热模式、PCIe 链路管理）（2分）
- 两者在初始化时被 `amdgpu_atif_handler()` 和 `amdgpu_atcs_handler()` 分别处理（1分）

**12. S3 睡眠时 AMDGPU 的关键步骤（5分）**
1. `amdgpu_device_suspend()` 被调用，暂停渲染队列和调度器（1分）
2. GPU 寄存器状态保存到内存（1分）
3. 调用 `amdgpu_acpi_suspend()` 处理 ACPI 相关挂起逻辑（1分）
4. GPU 进入 D3hot 状态（如支持 Runtime PM）（1分）
5. 系统进入 S3 状态，CPU 关闭（1分）

**13. S0ix 活跃状态阻塞（5分）**
- 定义：某些设备或驱动阻止系统进入最深度的 S0ix 子状态，导致功耗无法降低到目标水平（2分）
- 常见原因：
  - GPU 处于性能状态而非低功耗状态（1分）
  - PCIe 设备未正确进入 L1 链路状态（1分）
  - USB 设备唤醒设置不当（0.5分）
  - 内核定时器过于频繁（0.5分）

**14. 背光调试排查顺序（5分）**
1. 检查 `ls /sys/class/backlight/` 查看可用的背光接口（1分）
2. 检查 `dmesg | grep -E "backlight|amdgpu.*bl"` 查看驱动日志（1分）
3. 检查内核参数 `acpi_backlight=video|native|vendor` 是否正确（1分）
4. 通过 `cat /sys/class/backlight/*/brightness` 和 `max_brightness` 检查当前亮度值（1分）
5. 尝试手动写入亮度值验证硬件功能（1分）

**评分标准：** 每个要点按分值给分，表述不完整酌情扣分。

### 第三部分：实践操作题（30分）

**15. ACPI 事件监控脚本（15分）**

```bash
#!/bin/bash
# acpi_monitor_analyze.sh - ACPI 事件监控分析脚本

# 配置
MONITOR_DURATION=${1:-60}
OUTPUT_DIR="/tmp/acpi_monitor_$$"
EVENT_LOG="$OUTPUT_DIR/events.log"
REPORT_FILE="$OUTPUT_DIR/report.txt"

mkdir -p "$OUTPUT_DIR"
echo "ACPI Monitor Started at $(date)"
echo "Duration: ${MONITOR_DURATION}s"
echo "Output: $OUTPUT_DIR"

# 启动 acpi_listen 监控
timeout $MONITOR_DURATION sudo acpi_listen > "$EVENT_LOG" 2>&1 &
MONITOR_PID=$!

# 同时收集系统状态
(
    for i in $(seq 1 $MONITOR_DURATION); do
        echo "=== TIMESTAMP: $(date +%s%N) ==="
        cat /sys/power/mem_sleep 2>/dev/null
        cat /sys/class/drm/card0/device/power_state 2>/dev/null
        sleep 1
    done
) > "$OUTPUT_DIR/state.log" 2>&1 &
STATE_PID=$!

wait $MONITOR_PID 2>/dev/null
kill $STATE_PID 2>/dev/null

# 分析事件
TOTAL_EVENTS=$(wc -l < "$EVENT_LOG")
AMDGPU_EVENTS=$(grep -ci "amd\|ATIF\|ATCS\|amdgpu" "$EVENT_LOG" 2>/dev/null)

echo "==========================================" > "$REPORT_FILE"
echo "ACPI Event Analysis Report" >> "$REPORT_FILE"
echo "==========================================" >> "$REPORT_FILE"
echo "Monitor Duration: ${MONITOR_DURATION}s" >> "$REPORT_FILE"
echo "Total Events: $TOTAL_EVENTS" >> "$REPORT_FILE"
echo "AMDGPU Related: $AMDGPU_EVENTS" >> "$REPORT_FILE"
echo "" >> "$REPORT_FILE"

# 事件类型统计
echo "--- Event Type Statistics ---" >> "$REPORT_FILE"
if [ -f "$EVENT_LOG" ]; then
    sort "$EVENT_LOG" | uniq -c | sort -rn >> "$REPORT_FILE"
fi

echo "" >> "$REPORT_FILE"
echo "--- Power State Timeline ---" >> "$REPORT_FILE"
cat "$OUTPUT_DIR/state.log" >> "$REPORT_FILE"

echo "Report generated: $REPORT_FILE"
cat "$REPORT_FILE"
```

- 能识别 AMDGPU 相关事件（3分）

**16. 系统挂起/恢复测试脚本（15分）**

```bash
#!/bin/bash
# suspend_stress_test.sh - 系统挂起/恢复压力测试

ITERATIONS=${1:-5}
SUSPEND_MODE=${2:-"mem"}  # mem=S3, freeze=S0ix
LOG_DIR="/var/log/suspend_test_$(date +%Y%m%d_%H%M%S)"
FAIL_COUNT=0

mkdir -p "$LOG_DIR"
echo "Suspend/Resume Stress Test"
echo "Mode: $SUSPEND_MODE"
echo "Iterations: $ITERATIONS"
echo "Log: $LOG_DIR"

for i in $(seq 1 $ITERATIONS); do
    echo "--- Iteration $i/$ITERATIONS ---"
    
    # 预检查
    dmesg -c > /dev/null 2>&1  # 清空内核日志
    
    # GPU 状态检查
    GPU_STATE_BEFORE=$(cat /sys/class/drm/card0/device/power_state 2>/dev/null)
    echo "GPU state before: $GPU_STATE_BEFORE"
    
    # 执行挂起
    echo "Suspending..."
    sudo rtcwake -m $SUSPEND_MODE -s 30 -f
    
    # 等待唤醒
    sleep 5
    
    # 收集恢复后日志
    dmesg > "$LOG_DIR/iteration_${i}_dmesg.log"
    
    # GPU 状态检查
    GPU_STATE_AFTER=$(cat /sys/class/drm/card0/device/power_state 2>/dev/null)
    echo "GPU state after: $GPU_STATE_AFTER"
    
    # 检测 GPU 功能
    if [ "$GPU_STATE_AFTER" != "D0" ]; then
        echo "FAIL: GPU not in D0 state after resume"
        ((FAIL_COUNT++))
        continue
    fi
    
    # 检查 framebuffer
    if command -v glxinfo &> /dev/null; then
        glxinfo | grep -i "renderer" > "$LOG_DIR/iteration_${i}_gpu.txt"
    fi
    
    # 检查性能状态
    cat /sys/class/drm/card0/device/power_dpm_force_performance_level 2>/dev/null
    
    echo "Iteration $i passed"
done

# 生成报告
echo "=========================================="
echo "Test Summary"
echo "=========================================="
echo "Total Iterations: $ITERATIONS"
echo "Passed: $((ITERATIONS - FAIL_COUNT))"
echo "Failed: $FAIL_COUNT"

if [ $FAIL_COUNT -gt 0 ]; then
    echo "Failed iterations:"
    grep -l "FAIL" $LOG_DIR/*.log 2>/dev/null
fi
```

**评分标准：**
- 支持 S3 和 S0ix 两种模式（3分）
- 循环执行并记录日志（3分）
- GPU 状态检测完整性（3分）
- 恢复后功能验证（3分）
- 输出汇总报告（3分）

### 第四部分：案例分析（20分）

**17. S0ix 功耗异常分析**

(1) **睡眠模式识别（4分）**
- 当前使用 `[deep]` 即 S3 模式（2分）
- 应使用 `s2idle` 模式才能实现 S0ix（2分）
- 需要通过 `echo s2idle | sudo tee /sys/power/mem_sleep` 切换

(2) **S0ix 驻留时间分析（4分）**
- 8.5 秒异常低（2分）
- 正常 S0ix 驻留时间应在数小时级别（2分）
- 表明系统实际未真正进入 S0ix 状态

(3) **GPU 电源策略分析（4分）**
- GPU 当前处于 `performance` 状态（2分）
- 这会阻止 S0ix 进入深度睡眠，因为 GPU 保持高频率运行（2分）
- 应在睡眠前切换为 `auto` 或 `low` 状态

(4) **排查方法（4分）**
- 使用 `cat /sys/kernel/debug/pmc_core/lpm_enable_fail` 查看阻止 S0ix 的设备（1分）
- 使用 powertop 的 `--debug` 选项分析 C 状态阻塞（1分）
- 检查 `dmesg | grep -i "s0ix\|s2idle\|LPM"` 内核日志（1分）
- 使用 `turbostat` 监控实际 C 状态驻留时间（1分）

(5) **修复方案（4分）**
1. 切换到 s2idle 模式（1分）
2. 设置 GPU 电源策略为 auto（1分）
3. 更新内核到支持 Phoenix 平台 S0ix 的最新版本（1分）
4. 检查 BIOS 设置，确保 Modern Standby 已启用（1分）
