### 1. 概念原理

#### 1.1 电源管理架构总览

AMD GPU 电源管理是一个复杂的分层系统，涉及硬件电路、固件、内核驱动与用户空间工具。其核心目标是在满足性能需求的同时，最小化功耗与温度，并确保硬件安全。以下为基于 RDNA 3 架构的完整电源管理子系统架构图：

**各层详细说明**：

**用户空间层**：
- **rocm-smi**：ROCm 系统管理接口，提供 GPU 状态监控与简单控制。
- **amdgpu_top**：实时显示 GPU 利用率、频率、温度等。
- **powercap**：Linux 功耗限制框架的用户空间工具。
- **自定义脚本**：用户可编写脚本通过 sysfs/hwmon 接口进行自动化控制。

**内核空间层**：
- **amdgpu_pm.c**：电源管理框架，负责 sysfs/hwmon 接口导出、热管理、风扇控制、ACPI 事件处理等上层逻辑。
- **amdgpu_dpm.c**：动态电源管理引擎，负责频率/电压曲线管理、性能状态切换、SMU 通信等底层硬件控制。
- **amdgpu_acpi.c**：处理 ACPI（高级配置与电源接口）事件，如系统挂起/恢复、GPU 热插拔等。
- **其他模块**：如 thermal 子系统集成、RAS（可靠性、可用性、可服务性）支持、调试接口等。

**SMU 固件接口层**：
- **PSP（Platform Security Processor）**：安全处理器，负责与 SMU 的安全通信。
- **Mailbox 机制**：驱动通过 PSP 向 SMU 发送命令（如设置频率、电压），SMU 返回执行结果。
- **SMN（System Management Network）寄存器**：驱动可直接读取的硬件寄存器，用于获取实时状态（如当前频率、电压、温度）。

**硬件层**：
- **VRM（电压调节模块）**：将 VID 代码转换为实际电压，为 GPU 核心、显存等供电。
- **PLL（锁相环）**：生成高频时钟信号，分频后供给各个时钟域。
- **ADC（模数转换器）**：测量功耗、电压、电流等模拟信号，转换为数字值供 SMU 使用。
- **温度传感器**：分布在 GPU 芯片关键位置，监测热点温度。
- **SMU 微控制器**：GPU 内部专用 MCU，执行电源管理算法，直接控制 VRM、PLL 等硬件。

| 术语 | 全称 | 说明 | 典型值/范围 | 相关源码文件 |
|------|------|------|------------|-------------|
| **DPM** | Dynamic Power Management | 动态电源管理。根据 GPU 负载、温度、功耗限制等因素，自动调整频率与电压，以在性能与能效间取得平衡。 | 状态：performance/balanced/battery | `amdgpu_dpm.c`, `amdgpu_pm.c` |
| **P-State** | Performance State | 性能状态。一组预设的频率/电压组合，通常 P0 为最高性能（高频高电压），P1、P2... 为逐步降低的性能档位。 | P0~P7（取决于 GPU） | `amdgpu_dpm.c` |
| **SMU** | System Management Unit | 系统管理单元。GPU 内部负责电源、热管理、时钟控制的微控制器，执行固化算法，响应驱动命令。 | 固件版本：xx.xx.xx | `smu_v13_0.c`, `smu_v11_0.c` 等 |
| **PPT** | Package Power Limit | 封装功耗限制。GPU 允许的最大持续功耗（单位：瓦）。超过此限制将触发降频。 | 150~400 W（取决于 GPU TDP） | `amdgpu_dpm.c`, `smu_*` |
| **TDP** | Thermal Design Power | 热设计功耗。散热系统需要 dissipate 的最大热量，通常作为 PPT 的默认值。 | 同 PPT | - |
| **V/F Curve** | Voltage/Frequency Curve | 电压‑频率曲线。描述每个频率点所需的最低稳定电压，通常为非线性关系（高频段电压急剧上升）。 | 见 1.4 节 | `amdgpu_dpm.c`, SMU 固件 |
| **VID** | Voltage Identifier | 电压标识符。数字代码（如 0x00~0x1F），对应特定电压值，用于 SMU 与 VRM 间的通信。 | 0x00=750 mV, 0x1F=1300 mV | `smu_*` |
| **BACO** | Bus Active, Clock Off | 总线活动、时钟关闭。一种低功耗状态，保持 PCIe 总线活动（可响应配置访问）但关闭 GPU 内部大部分时钟。 | 进入/退出延迟：~10 ms | `amdgpu_dpm.c`, SMU 固件 |
| **BAMACO** | Bus Active, Memory Clock Off | 总线活动、显存时钟关闭。类似 BACO，同时关闭显存时钟以进一步节能。 | 比 BACO 更省电 | `amdgpu_dpm.c`, SMU 固件 |
| **ACPI** | Advanced Configuration and Power Interface | 高级配置与电源接口。系统级电源管理标准，用于处理挂起/恢复、热插拔、设备电源状态等。 | 状态：D0（全功率）~D3（关闭） | `amdgpu_acpi.c` |

| **PowerPlay** | AMD PowerPlay Technology | AMD 电源管理技术的总称，涵盖 DPM、电压调节、时钟门控等。 | - | `amdgpu_dpm.c`, `smu_*` |
| **OverDrive** | AMD OverDrive | 允许用户手动调整频率、电压、风扇曲线等参数的超频/降频界面。 | 通过 sysfs `pp_od_clk_voltage` 访问 | `amdgpu_pm.c` |

#### 1.3 电源管理数据流详细分析

以下为典型场景（用户通过 sysfs 将 DPM 状态改为 "performance"）的完整数据流：

```
1. 用户写入 sysfs：
   echo "performance" > /sys/class/drm/card0/device/power_dpm_state

2. sysfs 属性回调：
   amdgpu_set_power_dpm_state() [amdgpu_pm.c]
      ↓
   mutex_lock(&adev->pm.mutex);
   adev->pm.dpm.state = target_state;
   amdgpu_dpm_set_power_state(adev);  // 调用 DPM 引擎

3. DPM 引擎处理：
   amdgpu_dpm_set_power_state() [amdgpu_dpm.c]
      ↓
   amdgpu_dpm_change_power_state_locked(adev)
      ↓
   // 根据目标状态选择性能档位
   new_ps = amdgpu_dpm_select_performance_state(adev, target_state);
      ↓
   // 应用新状态
   amdgpu_dpm_apply_state(adev, new_ps)
      ↓
   amdgpu_dpm_apply_power_state(adev, new_ps)

4. SMU 通信：
   amdgpu_dpm_apply_power_state() [amdgpu_dpm.c]
      ↓
   // 通过 SMU 接口发送频率/电压请求
   smu_set_performance_level(adev->smu, level);
      ↓
   smu_send_smc_msg(adev->smu, msg_id, ¶ms);  // 实际与 PSP/SMU 通信

5. PSP/SMU 执行：
   PSP 接收驱动命令，验证后转发给 SMU 固件
      ↓
   SMU 解析命令，根据当前温度、功耗、V/F 曲线计算实际频率/电压
      ↓
   SMU 通过内部总线配置 VRM（发送 VID）、PLL（设置分频器）
      ↓
   SMU 等待硬件稳定后，通过 SMN 寄存器更新状态

6. 驱动读取确认：
   驱动定期轮询或等待 SMU 中断，确认状态已切换
      ↓
   更新内部状态，sysfs 属性反映新值（如 pp_dpm_sclk 显示新频率）

7. 用户验证：
   cat /sys/class/drm/card0/device/power_dpm_state
   // 输出 "performance"
```

#### 1.4 电压‑频率曲线（V/F Curve）深度解析

V/F 曲线是电源管理的核心，它决定了每个频率点所需的最低稳定电压。曲线形状受工艺、温度、芯片体质（硅质量）影响。下图展示了一条典型 V/F 曲线及其关键区域：

```
图例：
- 绿色虚线：理想 V/F 曲线（线性）
- 红色实线：实际 V/F 曲线（非线性）
- 蓝色区域：安全裕度（实际电压比最低要求高 ΔV）
- 点 A：最低电压点（Vmin, Fmin）
- 点 B：能效最优点（dP/dF 最小）
- 点 C：频率饱和点（超过此点电压急剧上升）
```

**曲线特性**：
- **亚阈值区域**（<800 mV）：电压极低，晶体管工作在亚阈值区，频率极低，功耗极小。
- **线性区域**（800‑1100 mV）：电压与频率近似线性关系，能效较好。
- **饱和区域**（>1100 mV）：为达到高频需要大幅提高电压，功耗呈二次方增长，能效急剧下降。
- **温度影响**：温度升高时，同一频率所需电压增加（曲线整体上移）。
- **工艺影响**：先进工艺（如 5nm）可在更低电压下达到相同频率。

**V/F 曲线获取方法**：
1. **固件内置**：SMU 固件包含经过芯片验证的 V/F 曲线。
2. **驱动读取**：通过 `pp_od_clk_voltage` 可查看部分离散点。
3. **实验测量**：通过反复压力测试确定各频率下的最低稳定电压。

**状态转换条件**：
- **负载变化**：GPU 利用率持续高于阈值 → 进入更高性能状态；利用率低于阈值 → 进入更低性能状态。
- **温度触发**：温度超过 Tjmax → 降频（thermal throttle）；温度低于恢复阈值 → 恢复频率。
- **功耗限制**：瞬时功耗超过 PPT → 降频（power throttle）。
- **用户干预**：通过 sysfs 手动切换 DPM 状态。
- **系统事件**：ACPI 挂起/恢复、显示器关闭/打开等。

**状态切换延迟**：
- **P-State 切换**：~10 µs‑1 ms（取决于硬件）
- **BACO 进入/退出**：~1‑10 ms
- **D3（冷启动）**：~100 ms‑1 s

**与调度器的交互**：
- 调度器根据任务优先级与期限分配计算资源。
- 电源管理根据调度器提供的负载预测（workload profile）提前调整频率。
- 高频可降低任务延迟，但增加功耗；低频可节能，但可能延长任务完成时间。

**与内存管理的交互**：
- 显存频率（MCLK）影响内存带宽与延迟。
- 内存密集型任务需要高 MCLK，此时可适当降低 SCLK 以保持总功耗在限制内。
- 显存电压（VDDCI）与 MCLK 关联，调整 MCLK 时自动调整 VDDCI。

**与温度监控的交互**：
- 温度传感器提供实时热点温度。
- 电源管理根据温度调整风扇转速、降低频率（thermal throttling）。
- 温度过高可能触发紧急关机（thermal shutdown）。


**SMU 消息传递协议**：

驱动与 SMU 之间的通信通过 MMIO (Memory-Mapped I/O) Mailbox 寄存器进行：

```c
// SMU Mailbox 寄存器定义（典型 RDNA 3）
#define mmMP0_SMN_C2PMSG_00      0x0280  // 驱动→SMU 消息 ID
#define mmMP0_SMN_C2PMSG_01      0x0281  // 驱动→SMU 参数
#define mmMP0_SMN_C2PMSG_02      0x0282  // 驱动→SMU 参数
#define mmMP0_SMN_C2PMSG_30      0x029E  // SMU→驱动 响应
#define mmMP0_SMN_C2PMSG_31      0x029F  // SMU→驱动 参数
#define mmMP0_SMN_C2PMSG_32      0x02A0  // SMU→驱动 参数

// 消息发送流程（驱动侧）：
// 1. 等待 SMU 空闲（轮询 C2PMSG_30 的 busy 位）
// 2. 写入参数到 C2PMSG_01/02
// 3. 写入消息 ID 到 C2PMSG_00（触发 SMU 中断）
// 4. 等待 SMU 处理完成（轮询 C2PMSG_30，或等待中断）
// 5. 读取响应状态与返回数据

// 内核实现示例（smu_v13_0.c）：
int smu_send_smc_msg(struct smu_context *smu, uint32_t msg_id, uint32_t *params)
{
    int ret;
    uint32_t timeout = 5000;  // 5ms 超时
    
    // 1. 检查 SMU 是否忙
    ret = read_smu_register(smu, mmMP0_SMN_C2PMSG_30, &busy);
    if (busy & SMU_BUSY_MASK)
        return -EBUSY;
    
    // 2. 写入参数
    if (params)
        write_smu_register(smu, mmMP0_SMN_C2PMSG_01, params[0]);
    
    // 3. 发送消息（写入消息 ID 触发）
    write_smu_register(smu, mmMP0_SMN_C2PMSG_00, (uint32_t)msg_id);
    
    // 4. 等待完成
    do {
        udelay(1);
        read_smu_register(smu, mmMP0_SMN_C2PMSG_30, &busy);
    } while ((busy & SMU_BUSY_MASK) && --timeout);
    
    if (timeout == 0)
        return -ETIMEDOUT;
    
    // 5. 读取响应
    if (busy & SMU_ERROR_MASK)
        return -EIO;
    
    // 6. 读取返回数据
    if (params)
        read_smu_register(smu, mmMP0_SMN_C2PMSG_01, &params[0]);
    
    return 0;
}
```

**SMU 消息分类**：

| 消息类别 | 消息 ID 范围 | 示例 | 描述 |
|---------|------------|------|------|
| 控制类 | 0x00-0x1F | SMU_MSG_SetGfxFreq, SMU_MSG_SetVdd | 设置频率、电压 |
| 查询类 | 0x20-0x3F | SMU_MSG_GetGfxFreq, SMU_MSG_GetTemp | 读取状态 |
| 配置类 | 0x40-0x5F | SMU_MSG_SetPPT, SMU_MSG_SetFanCurve | 设置配置参数 |
| 电源类 | 0x60-0x7F | SMU_MSG_EnterBACO, SMU_MSG_ExitBACO | 电源状态切换 |
| 调试类 | 0x80-0x9F | SMU_MSG_DumpRegs, SMU_MSG_RunTest | 调试与测试 |

#### 3.4 电源管理关键算法详解

**3.4.1 DPM 状态选择算法**

DPM 引擎根据多种输入决定目标 P-State：

```
输入：
- GPU 利用率 (0-100%)
- 当前温度
- 当前功耗
- 用户设置 (performance/balanced/battery)
- 任务类型 (3D/VIDEO/COMPUTE)

算法流程：
1. 计算负载因子 L = f(utilization, task_type)
2. 计算温度因子 T = g(current_temp, tj_max)
3. 计算功耗因子 P = h(current_power, ppt_limit)
4. 计算综合分数 S = w1*L + w2*(1-T) + w3*(1-P)
   (w1, w2, w3 为权重系数)
5. 选择目标状态:
   if S > 0.8 → P0-P1 (最高性能)
   if S > 0.5 → P2-P4 (平衡)
   if S > 0.2 → P5-P6 (省电)
   else → P7 (最低功耗)
6. 应用防抖(debounce): 连续 N 次采样结果一致才切换
7. 执行状态切换
```

**3.4.2 PPT (Package Power Limit) 管理算法**

PPT 算法实时监控 GPU 功耗，当超过限制时主动降频：

```
PPT 算法伪代码（SMU 固件实现）：

每次控制周期（约 1ms）:
  1. 读取当前功耗（通过 SMU 内部 ADC）
  2. 计算移动平均功耗 P_avg = α * P_avg + (1-α) * P_instant
  3. 计算功耗余量 P_margin = PPT_limit - P_avg
  
  if P_margin > 0:
    // 功耗在限制内，允许升频
    freq_target = min(freq_max, freq_current + Δf_up)
  else:
    // 功耗超过限制，必须降频
    // 降频幅度与超出量成正比
    delta = abs(P_margin) / PPT_limit * max_freq_step
    freq_target = max(freq_min, freq_current - delta)
  
  4. 根据 V/F 曲线查找对应电压
  5. 应用新频率/电压

// 双时间尺度 PPT 实现
// fast PPT: 短时间（10ms）内功耗超过限制，快速响应
// power_limit_fast = PPT_limit * 1.1
// slow PPT: 长时间（1s）平均功耗限制
// power_limit_slow = PPT_limit

if P_avg_fast > power_limit_fast:
    // 紧急降频（每次降 2 个 P-State）
    freq_target = freq_current - 2 * state_step
elif P_avg_slow > power_limit_slow:
    // 渐进降频
    freq_target = freq_current - state_step
else:
    // 正常调节
```

**3.4.3 时钟门控 (Clock Gating) 策略**

时钟门控是减少动态功耗的关键技术，在 RDNA 3 中分为多个层级：

```
层级 0: CG (Coarse-Grained Clock Gating) - 粗粒度门控
  - 引擎级别（如整个 GFX 引擎空闲时关闭时钟）
  - 进入延迟: ~1μs
  - 退出延迟: ~5μs

层级 1: FG (Fine-Grained Clock Gating) - 细粒度门控
  - 功能单元级别（如着色器中的 ALU）
  - 进入延迟: ~100ns
  - 退出延迟: ~200ns

层级 2: GLS (Global Clock Slowdown) - 全局时钟减速
  - 低负载时降低全局时钟频率
  - 与 DPM 协同工作

层级 3: Power Gating - 电源门控
  - 完全切断空闲单元的供电
  - 进入延迟: ~10μs
  - 退出延迟: ~50μs

控制策略：
- SMU 根据引擎活动计数器决定门控状态
- 活动计数器 >= 阈值 → 开启时钟
- 活动计数器 = 0 持续 N 个周期 → 关闭时钟
- 不同引擎（GFX/MM/SDMA/VCN）独立门控
```

#### 4.1 电源管理性能分析脚本

```python
#!/usr/bin/env python3
"""
GPU 电源管理性能分析工具
测量不同 DPM 状态下的性能、功耗与温度
"""
import os
import time
import subprocess
import statistics

class GPUPowerAnalyzer:
    def __init__(self, card=0):
        self.card = card
        self.dev_path = f"/sys/class/drm/card{card}/device"
        self.hwmon = self._find_hwmon()
        
    def _find_hwmon(self):
        base = f"{self.dev_path}/hwmon"
        if not os.path.exists(base):
            return None
        for entry in os.listdir(base):
            if entry.startswith('hwmon'):
                return f"{base}/{entry}"
        return None
    
    def _read_sysfs(self, path):
        try:
            with open(path, 'r') as f:
                return f.read().strip()
        except:
            return None
    
    def set_perf_level(self, level):
        path = f"{self.dev_path}/power_dpm_force_performance_level"
        try:
            with open(path, 'w') as f:
                f.write(level)
            return True
        except:
            return False
    
    def measure_gpu_state(self):
        """测量当前 GPU 状态"""
        state = {}
        state['perf_level'] = self._read_sysfs(
            f"{self.dev_path}/power_dpm_force_performance_level")
        state['sclk'] = self._read_sysfs(f"{self.dev_path}/pp_dpm_sclk")
        state['mclk'] = self._read_sysfs(f"{self.dev_path}/pp_dpm_mclk")
        
        if self.hwmon:
            temp = self._read_sysfs(f"{self.hwmon}/temp1_input")
            state['temp'] = int(temp) / 1000 if temp else None
            power = self._read_sysfs(f"{self.hwmon}/power1_average")
            state['power'] = int(power) / 1000000 if power else None
            fan = self._read_sysfs(f"{self.hwmon}/fan1_input")
            state['fan'] = fan
        
        return state
    
    def run_stress_test(self, duration=30):
        """运行压力测试并记录数据"""
        print(f"运行 {duration} 秒压力测试...")
        records = []
        start = time.time()
        while time.time() - start < duration:
            records.append(self.measure_gpu_state())
            time.sleep(1)
        return records
    
    def benchmark_all_levels(self):
        """测试所有性能级别"""
        levels = ['low', 'auto', 'high', 'manual']
        results = {}
        
        for level in levels:
            if not self.set_perf_level(level):
                print(f"无法设置 {level} 级别")
                continue
            time.sleep(2)  # 等待稳定
            
            print(f"测试性能级别: {level}")
            records = self.run_stress_test(30)
            
            # 计算统计
            temps = [r['temp'] for r in records if r['temp']]
            powers = [r['power'] for r in records if r['power']]
            
            results[level] = {
                'avg_temp': statistics.mean(temps) if temps else 0,
                'max_temp': max(temps) if temps else 0,
                'avg_power': statistics.mean(powers) if powers else 0,
                'max_power': max(powers) if powers else 0,
                'records': records
            }
            
            print(f"  平均温度: {results[level]['avg_temp']:.1f}°C")
            print(f"  平均功耗: {results[level]['avg_power']:.2f}W")
        
        # 恢复自动
        self.set_perf_level('auto')
        return results


if __name__ == "__main__":
    analyzer = GPUPowerAnalyzer()
    results = analyzer.benchmark_all_levels()
```

#### 7.3 简答题要点

1. 四层架构：
   - 用户层（rocm-smi, sysfs工具）
   - 内核层（amdgpu_pm.c, amdgpu_dpm.c）
   - 固件层（SMU 固件）
   - 硬件层（VRM, PLL, 传感器）

2. V/F 曲线定义了每个频率点所需的最低稳定电压。温度升高时，晶体管漏电流增加，需要更高电压维持相同频率，曲线整体上移。

3. 双时间尺度 PPT 实现快速响应（抑制功耗尖峰）和长期稳定（控制平均功耗）两个目标。fast PPT 响应时间窗口短（10ms），slow PPT 窗口长（1s）。

4. RDNA 3 引入了分片式 GPU 设计（GCD + MCD），每个芯片拥有独立的 SMU，通过 Infinity Fabric 互联。RDNA 2 为单芯片单 SMU 设计。

5. SMU 消息流程：驱动写入参数到 Mailbox 寄存器 → 写入消息 ID 触发 SMU 中断 → SMU 读取消息并执行 → SMU 写入响应到 Mailbox → 驱动轮询或等待中断获取结果。

### A.2 PPT功耗限制双窗口算法交互

```
PPTable 中定义的 PPT 配置:
┌─────────────────────────────────────────────────────────────┐
│  PPT0: 快速窗口 (Fast PPT)                                  │
│  ├─ 窗口大小: 1ms (可通过 smu->ppt_default_table 读取)      │
│  ├─ 功耗阈值: 200W (PowerLimit)                             │
│  ├─ 触发响应: 立即降频 (无延迟)                             │
│  └─ 适用场景: 瞬时电流尖峰保护 (过流保护)                   │
│                                                             │
│  PPT1: 慢速窗口 (Slow PPT)                                  │
│  ├─ 窗口大小: 10ms                                          │
│  ├─ 功耗阈值: 180W                                          │
│  ├─ 触发响应: 渐进式降频 (PID控制器逐步调整)                │
│  └─ 适用场景: 持续高负载下的热管理                          │
└─────────────────────────────────────────────────────────────┘

SMU内部PPT执行循环 (每1ms执行一次):
    ┌──────────────────────────────┐
    │ 读取 ADC 电流/电压采样       │
    └──────────┬───────────────────┘
               ▼
    ┌──────────────────────────────┐
    │ 计算瞬时功耗 P_inst = V×I    │
    └──────────┬───────────────────┘
               ▼
    ┌─────────────────────────────────────────────────┐
    │ 更新快速窗口移动平均 P_fast = α×P_inst          │
    │                     + (1-α)×P_fast_prev          │
    └──────────┬──────────────────────────────────────┘
               ▼
    ┌─────────────────────────────────────────────────┐
    │ 更新慢速窗口移动平均 P_slow = β×P_inst          │
    │                     + (1-β)×P_slow_prev          │
    └──────────┬──────────────────────────────────────┘
               ▼
    ┌─────────────────────────────────────────────────┐
    │ 比较 P_fast > PPT0_Threshold?                   │
    │  ├─ 是: 立即降频到最低安全P-State               │
    │  └─ 否: 继续检查慢速窗口                        │
    └──────────┬──────────────────────────────────────┘
               ▼
    ┌─────────────────────────────────────────────────┐
    │ 比较 P_slow > PPT1_Threshold?                   │
    │  ├─ 是: 逐步降低P-State (每步~50mV)             │
    │  │     直到 P_slow < PPT1_Threshold              │
    │  └─ 否: 恢复频率 (如之前被限制)                 │
    └─────────────────────────────────────────────────┘
```

### A.3 Thermal Throttle温度控制流程

```
温度传感器 (TS) 采样 (每~10ms)
    │
    ▼
SMU 读取 junction/mem/hotspot 温度
    │
    ▼
    ┌──────────────────────────────────────────────────┐
    │  温度阈值判断                                      │
    │                                                    │
    │  T > Tj_max (110°C)?                              │
    │    ├─ 是 → 紧急关机 (Tj_max + 5°C Hysteresis)    │
    │    │                                             │
    │  T > Tj_throttle (100°C)?                        │
    │    ├─ 是 → 硬降频 (Hard Throttle)                │
    │    │      └─ GFXCLK 强制降至最低档                │
    │    │      └─ 忽略所有性能请求                     │
    │    │                                             │
    │  T > Tj_soft_throttle (90°C)?                    │
    │    ├─ 是 → 软降频 (Soft Throttle)                │
    │    │      └─ 每 100ms 降低一个P-State             │
    │    │      └─ 允许用户请求但优先级较低             │
    │    │                                             │
    │    └─ 否 → 正常运行 (允许提升P-State)            │
    └──────────────────────────────────────────────────┘
    │
    ▼
温度降低后的恢复流程:
    ┌─────────────────────────────────────────────┐
    │ T < Tj_throttle - Hysteresis (95°C)?        │
    │  └─ 是 → 解除硬降频限制                     │
    │         └─ 恢复正常P-State管理               │
    │                                             │
    │ T < Tj_soft_throttle - Hysteresis (85°C)?  │
    │  └─ 是 → 解除软降频限制                     │
    │         └─ 允许 P-State 逐级回升             │
    └─────────────────────────────────────────────┘
```

```

    │ SMU 接收唤醒事件                     │

    │ ├─ 恢复 MCLK (如已关闭)             │
    │ └─ 重新初始化温度传感器              │

               ▼
    ┌─────────────────────────────────────┐
    │ amdgpu_device_ip_resume_phase1()    │
    │ ├─ SMU IP: 恢复DPM配置             │
    │ ├─ PSP IP: 重新初始化安全处理器      │
    │ └─ GMC IP: 恢复页表 + TLB刷新       │
    └──────────┬──────────────────────────┘
               ▼
    ┌─────────────────────────────────────┐
    │ amdgpu_device_ip_resume_phase2()    │
    │ ├─ GFX IP: 恢复上下文 + 退出CG/FG   │
    │ ├─ SDMA IP: 恢复DMA引擎             │
    │ ├─ VCN IP: 恢复编解码               │
    │ └─ Display IP: 恢复显示输出          │
    └──────────┬──────────────────────────┘
               ▼
    ┌─────────────────────────────────────┐
    │ GPU 恢复到 BACO 前状态的 P-State     │

    └─────────────────────────────────────┘
```

### B.3 电源管理关键性能指标 (KPI)

```
评估GPU电源管理质量的KPI定义:

1. P-State切换延迟
   定义: 从一个P-State切换到另一个的时间
   测量: echo low > power_dpm_force_performance_level
         → 测量 dmesg 中 SMU 响应时间戳差值
   优秀: <100µs  一般: 100-500µs  差: >500µs

2. PPT响应时间
   定义: 功耗超过阈值到频率开始降低的时间
   测量: 使用功率分析仪(外置) + 示波器
   优秀: <1ms  一般: 1-5ms  差: >5ms

3. 温度控制精度
   定义: 满载稳态温度与目标温度的偏差
   测量: cat /sys/class/drm/card0/device/hwmon/hwmon*/temp1_input
   优秀: ±2°C  一般: ±5°C  差: >±10°C

4. BACO入口/出口时间
   定义: 从触发到完成的完整状态转换时间
   测量: ftrace 追踪 pm_runtime_suspend/resume
   入口优秀: <50ms  一般: 50-200ms  差: >200ms
   出口优秀: <100ms  一般: 100-500ms  差: >500ms

5. 功耗效率 (Performance-per-Watt)
   定义: FPS / 平均功耗
   测量: 运行基准测试 + 记录功耗
   优秀: > 代际平均的110%  一般: 90-110%  差: < 90%
```

## 附录C：电源管理故障排查实战演练

### C.1 故障场景1：GPU频率锁定在最低档

**症状**：`cat /sys/class/drm/card0/device/pp_dpm_sclk` 显示频率固定在最低档，性能下降严重。

**排查步骤**：
```bash
# 1. 检查当前性能级别
cat /sys/class/drm/card0/device/power_dpm_force_performance_level
# 可能输出: auto / low / high / manual

# 2. 检查温度是否触发了thermal throttle
cat /sys/class/drm/card0/device/hwmon/hwmon*/temp1_input
# 输出: 55000 (55°C) → 温度正常，非热节流

# 3. 检查功耗限制是否生效
cat /sys/class/drm/card0/device/hwmon/hwmon*/power1_cap
cat /sys/class/drm/card0/device/hwmon/hwmon*/power1_input
# 比较当前功耗是否接近限制值

# 4. 检查dmesg中关于DPM报错
dmesg | grep -i "dpm\|SMU\|power\|throttl"
# 可能输出: SMU: Failed to set DPM level

# 5. 检查SMU固件状态
cat /sys/kernel/debug/dri/0/amdgpu_smu_info | grep -i "status\|state"

# 6. 重置电源管理状态
echo auto > /sys/class/drm/card0/device/power_dpm_force_performance_level
echo high > /sys/class/drm/card0/device/power_dpm_force_performance_level
```

**典型根因**：
- 驱动bug导致SMU命令失败
- SMU固件崩溃（需GPU重置）
- 功耗限制设置过低
- 电源线未插好（笔记本 + eGPU场景）

### C.2 故障场景2：PPT限制导致性能不达标

**症状**：运行3D应用时性能低于预期，GPU未达最高频率。

**排查步骤**：
```bash
# 1. 监控实时频率和功耗
watch -n 0.5 'echo "---频率---"; cat /sys/class/drm/card0/device/pp_dpm_sclk; \
     echo "---功耗---"; cat /sys/class/drm/card0/device/hwmon/hwmon*/power1_input; \
     echo "---限制---"; cat /sys/class/drm/card0/device/hwmon/hwmon*/power1_cap'

# 2. 调整功耗上限（如硬件支持）
echo 200000000 > /sys/class/drm/card0/device/hwmon/hwmon*/power1_cap
# 注意: 单位微瓦, 200000000µW = 200W

# 3. 使用amdgpu_top观察实时状态
amdgpu_top -d 1

# 4. 检查VRM供电能力
cat /sys/kernel/debug/dri/0/amdgpu_pm_info | grep -i "power\|voltage\|current"

# 5. 运行压力测试并记录
cat > /tmp/ppt_test.sh << 'EOF'
#!/bin/bash
echo "PPT压力测试 - $(date)"
for i in $(seq 1 30); do
    POWER=$(cat /sys/class/drm/card0/device/hwmon/hwmon*/power1_input)
    FREQ=$(cat /sys/class/drm/card0/device/pp_dpm_sclk | grep "*" | head -1)
    TEMP=$(cat /sys/class/drm/card0/device/hwmon/hwmon*/temp1_input)
    echo "$i: ${POWER}µW, ${FREQ}, ${TEMP}m°C"
    sleep 1
done
EOF
chmod +x /tmp/ppt_test.sh
/tmp/ppt_test.sh
```

### C.3 故障场景3：温度异常偏高

**症状**：空闲时GPU温度高于60°C，或满载瞬间温度飙升到100°C以上。

**排查步骤**：
```bash
# 1. 检查所有温度传感器
for s in /sys/class/drm/card0/device/hwmon/hwmon*/temp*_input; do
    NAME=$(basename $s | sed 's/_input//')
    VAL=$(cat $s 2>/dev/null)
    echo "$NAME: $VAL°C" 
done

# 2. 检查风扇是否正常
cat /sys/class/drm/card0/device/hwmon/hwmon*/fan1_input
# 0 RPM → 风扇停转或传感器故障

# 3. 检查风扇控制模式
cat /sys/class/drm/card0/device/hwmon/hwmon*/pwm1_enable
# 0=无, 1=手动, 2=自动(温度曲线)

# 4. 手动强制风扇全速测试
echo 1 > /sys/class/drm/card0/device/hwmon/hwmon*/pwm1_enable
echo 255 > /sys/class/drm/card0/device/hwmon/hwmon*/pwm1
sleep 10
cat /sys/class/drm/card0/device/hwmon/hwmon*/temp1_input
# 如果温度下降 → 自动风扇控制问题
# 如果温度不变 → 散热器接触不良或风扇物理故障

# 5. 检查thermal throttle触发情况
cat /sys/kernel/debug/dri/0/amdgpu_pm_info | grep -i "throttle\|thermal"
```

### C.4 故障场景4：BACO/BAMACO入口失败

**症状**：笔记本dGPU无法进入低功耗状态，电池续航短。

**排查步骤**：
```bash
# 1. 检查Runtime PM控制模式
cat /sys/bus/pci/devices/0000:01:00.0/power/control
# 必须为 auto 才能自动进入Runtime PM

# 2. 检查当前Runtime PM状态
cat /sys/bus/pci/devices/0000:01:00.0/power/runtime_status
# active → 未挂起

# 3. 检查是否有进程持有refcount
cat /sys/bus/pci/devices/0000:01:00.0/power/runtime_usage
# >0 → 有使用者阻止挂起

# 4. 主动触发挂起
echo auto > /sys/bus/pci/devices/0000:01:00.0/power/control
```

**题目2**：解释PPT双窗口算法的设计哲学，为什么需要快速和慢速两个窗口？

**参考答案**：
双窗口算法的设计是为了同时应对两种不同的功耗过载场景：
- **快速窗口(1ms)**：保护硬件免受瞬时电流尖峰损害。当GPU负载突然增加（如从桌面切换到游戏），电流可能瞬间超过安全阈值。快速窗口能立即检测并响应（~µs级别降频），防止VRM和电源线过流。
- **慢速窗口(10ms)**：管理持续负载下的热平衡。当GPU持续高负载运行时，温度逐渐上升，需要通过渐进式降频将功耗和温度控制在稳态范围内。慢速窗口使用PID算法平滑调整，避免频繁震荡。

### D.4 综合设计题

**题目6**：设计一个GPU电源管理的自动化测试框架，至少包含以下测试维度：P-State切换正确性、PPT限制精度、BACO入口/出口成功率、温度控制稳定性。

**参考答案**：
```bash
#!/bin/bash
# gpu_pm_autotest.sh - GPU电源管理自动化测试框架

PASS=0
FAIL=0
LOG_FILE="/tmp/gpu_pm_test_$(date +%Y%m%d_%H%M%S).log"

log() {
    echo "$(date +%H:%M:%S) [$1] $2" | tee -a "$LOG_FILE"
}

check_result() {
    if [ $1 -eq 0 ]; then
        PASS=$((PASS + 1))
        log "PASS" "$2"
    else
        FAIL=$((FAIL + 1))
        log "FAIL" "$2"
    fi
}

# 测试1: P-State切换正确性
test_pstate_switch() {
    log "INFO" "=== 测试1: P-State切换 ==="
    
    for level in low high auto manual; do
        echo $level > /sys/class/drm/card0/device/power_dpm_force_performance_level 2>/dev/null
        RET=$?
        sleep 1
        
        CURRENT=$(cat /sys/class/drm/card0/device/power_dpm_force_performance_level 2>/dev/null)
        if [ "$CURRENT" = "$level" ]; then
            check_result 0 "P-State切换至 $level"
        else
            check_result 1 "P-State切换至 $level (当前=$CURRENT)"
        fi
    done
    
    # 测试manual模式下设置指定P-State
    echo manual > /sys/class/drm/card0/device/power_dpm_force_performance_level
    echo 0 > /sys/class/drm/card0/device/pp_dpm_sclk 2>/dev/null
    check_result $? "设置SCLK P-State为0"
}

# 测试2: PPT限制精度
test_ppt_accuracy() {
    log "INFO" "=== 测试2: PPT限制精度 ==="
    
    # 仅在支持功耗限制的设备上执行
    if [ ! -f /sys/class/drm/card0/device/hwmon/hwmon*/power1_cap ]; then
        log "SKIP" "不支持PPT调整"
        return
    fi
    
    # 设置50%功耗限制
    CURRENT_CAP=$(cat /sys/class/drm/card0/device/hwmon/hwmon*/power1_cap)
    HALF_CAP=$((CURRENT_CAP / 2))
    
    echo $HALF_CAP > /sys/class/drm/card0/device/hwmon/hwmon*/power1_cap
    sleep 2
    
    ACTUAL_CAP=$(cat /sys/class/drm/card0/device/hwmon/hwmon*/power1_cap)
    ERROR=$(( (HALF_CAP - ACTUAL_CAP) * 100 / HALF_CAP ))
    ERROR=${ERROR#-}  # 绝对值
    
    if [ $ERROR -le 5 ]; then
        check_result 0 "PPT限制精度: 目标=$((HALF_CAP/1000000))W, 实际=$((ACTUAL_CAP/1000000))W, 误差=${ERROR}%"
    else
        check_result 1 "PPT限制精度: 目标=$((HALF_CAP/1000000))W, 实际=$((ACTUAL_CAP/1000000))W, 误差=${ERROR}%"
    fi
    
    # 恢复
    echo $CURRENT_CAP > /sys/class/drm/card0/device/hwmon/hwmon*/power1_cap
}

# 测试3: 温度控制稳定性
test_thermal_stability() {
    log "INFO" "=== 测试3: 温度控制稳定性 ==="
    
    TEMP_BEFORE=$(cat /sys/class/drm/card0/device/hwmon/hwmon*/temp1_input)
    
    # 加载GPU
    echo high > /sys/class/drm/card0/device/power_dpm_force_performance_level
    sleep 30
    
    TEMP_AFTER=$(cat /sys/class/drm/card0/device/hwmon/hwmon*/temp1_input)
    TEMP_RISE=$(( (TEMP_AFTER - TEMP_BEFORE) / 1000 ))
    
    if [ $TEMP_RISE -le 50 ]; then
        check_result 0 "温度控制: 上升${TEMP_RISE}°C (30秒负载)"
    else
        check_result 1 "温度控制: 上升${TEMP_RISE}°C (可能散热问题)"
    fi
    
    # 恢复
    echo auto > /sys/class/drm/card0/device/power_dpm_force_performance_level
}

# 测试4: BACO入口/出口成功率
test_baco_transition() {
    log "INFO" "=== 测试4: BACO转换 (10次循环) ==="
    
    for i in $(seq 1 10); do
        # 触发Runtime PM suspend
        echo auto > /sys/bus/pci/devices/0000:01:00.0/power/control 2>/dev/null
        sleep 2
        
        STATUS=$(cat /sys/bus/pci/devices/0000:01:00.0/power/runtime_status 2>/dev/null)
        if [ "$STATUS" = "suspended" ]; then
            PASS=$((PASS + 1))
            log "PASS" "BACO入口 #$i 成功"
        else
            FAIL=$((FAIL + 1))
            log "FAIL" "BACO入口 #$i 失败 (status=$STATUS)"
        fi
        
        # 触发resume
        echo on > /sys/bus/pci/devices/0000:01:00.0/power/control 2>/dev/null
        sleep 1
    done
}

# 运行所有测试
log "INFO" "GPU电源管理自动化测试开始"
test_pstate_switch
test_ppt_accuracy
test_thermal_stability

# BACO测试仅在dGPU上执行
if lspci | grep -qi "AMD\|ATI" && lspci -s 01:00.0 | grep -qi "VGA"; then
    test_baco_transition
fi

# 汇总
log "INFO" "========================================="
log "INFO" "测试完成: 通过=$PASS 失败=$FAIL"
log "INFO" "详细日志: $LOG_FILE"
log "INFO" "========================================="

exit $FAIL
```

### D.5 架构演进分析题

**题目7**：分析从GCN到RDNA 3的电源管理架构演进路线，指出最重要的3个架构变革，并说明理由。

**参考答案**：

**最有影响力的3个架构变革**：

1. **从硬件DPM到软DPM (SMU固件主导)**：
   - GCN时代：P-State切换由纯硬件状态机控制，固件只做监控
   - RDNA时代：SMU固件拥有完整控制权，驱动通过命令接口与SMU通信
   - 理由：软DPM使固件更新可以修复PM问题、优化算法，无需更换硬件

2. **多窗口PPT算法引入**：
   - Vega首次引入双窗口PPT (快速1ms + 慢速10ms)
   - RDNA 3扩展到3个PPT窗口 (GFX核心 + VRAM + SoC)
   - 理由：精确的功耗控制允许GPU在安全范围内更激进地提升性能，显著提高Performance-per-Watt

3. **细粒度电源门控和GFXOFF**：
   - GCN/Vega：只能整体门控GFX域
   - RDNA 1引入GFXOFF：空闲时完全关闭GFX引擎
   - RDNA 3：每个IP模块独立门控 (GFX/SDMA/VCN/JPEG)
   - 理由：移动设备和笔记本GPU对空闲功耗极其敏感，GFXOFF和细粒度门控使dGPU在笔记本上真正可用
