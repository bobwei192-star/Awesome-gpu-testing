# 第321天：电源管理与调度器的交互

## 概述

GPU 电源管理和任务调度之间存在紧密的耦合关系。调度器决定 GPU 何时执行什么任务, 而电源管理决定 GPU 以什么性能状态执行这些任务。两者协同工作的效率直接影响系统的吞吐量、响应时间和能效比。

本章深入分析 AMDGPU 驱动中电源管理与调度器的交互机制, 涵盖硬件调度器 (PFP/ME/CE)、软件调度器 (amdgpu_sched)、电源状态转换对调度的影响、以及各种协同优化策略。

```text

  └─────────────────────────────────────────────────────────────┘

硬件调度流程:
  1. 软件调度器将 job 写入 ring buffer (位于 GPU 可见的内存)
  2. MEC/PFP 从 ring buffer 预取命令 (PFP 负责预取, ME 负责执行)
  3. 命令被解析后分发给对应的执行单元 (CU/TA/TM/RB 等)
  4. 执行完成后, MEC 写回完成信号 (fence) 到内存
  5. 软件调度器轮询或等待中断, 确认 job 完成
```

### 2.3 内核代码分析: 调度器与电源管理的交互

```c
// drivers/gpu/drm/amd/amdgpu/amdgpu_sched.c
// 简化代码展示调度器如何感知电源状态

struct amdgpu_sched {
    struct drm_gpu_scheduler sched;
    struct amdgpu_device *adev;
    spinlock_t power_lock;

    /* 电源感知调度字段 */
    u32 load_avg;                    /* 滑动平均负载 (0~100) */
    ktime_t last_power_sample;       /* 上次功耗采样时间 */
    u64 busy_cycles;                 /* 繁忙周期计数 */
    u64 total_cycles;                /* 总周期计数 */
    u32 hysteresis_count;            /* 滞回计数器 */
    enum amd_pm_state_type desired_power_state; /* 期望电源状态 */
};

/* 调度器负载采样 - 每次 job 完成时调用 */
static void amdgpu_sched_sample_load(struct amdgpu_sched *sched)
{
    ktime_t now = ktime_get();
    u64 elapsed_us = ktime_to_us(ktime_sub(now, sched->last_power_sample));
    u32 instant_load;

    if (elapsed_us < 10000)  /* 最小采样间隔 10ms */
        return;

    /* 计算瞬时负载: (繁忙时间 / 总时间) * 100 */
    instant_load = (sched->busy_cycles * 100) /
                   max(sched->total_cycles, 1ULL);

    /* EWMA 指数加权移动平均 */
    sched->load_avg = (sched->load_avg * 7 + instant_load * 3) / 10;

    /* 重置计数器 */
    sched->busy_cycles = 0;
    sched->total_cycles = 0;
    sched->last_power_sample = now;

    /* 根据负载调整电源状态 */
    amdgpu_sched_apply_power_policy(sched);
}

/* 应用电源策略 */
static void amdgpu_sched_apply_power_policy(struct amdgpu_sched *sched)
{
    struct amdgpu_device *adev = sched->adev;
    enum amd_pm_state_type new_state;

    if (sched->load_avg > 80) {
        sched->hysteresis_count++;
        if (sched->hysteresis_count >= 3) {
            new_state = POWER_STATE_TYPE_PERFORMANCE;
            sched->hysteresis_count = 0;
        }
    } else if (sched->load_avg < 40) {
        sched->hysteresis_count++;
        if (sched->hysteresis_count >= 5) {
            new_state = POWER_STATE_TYPE_BATTERY;
            sched->hysteresis_count = 0;
        }
    } else {
        new_state = POWER_STATE_TYPE_BALANCED;
        sched->hysteresis_count = 0;
    }

    if (new_state != sched->desired_power_state) {
        sched->desired_power_state = new_state;
        amdgpu_dpm_set_power_state(adev, new_state);
    }
}

/* 电源状态变化通知调度器 */
void amdgpu_sched_power_state_changed(struct amdgpu_device *adev)
{
    /* 电源状态变化后, 调整 job 提交节奏 */
    struct drm_gpu_scheduler *sched;
    u32 new_freq = adev->pm.current_mclk;
    u32 old_freq = adev->pm.last_mclk;

    /* 如果频率下降, 增加 job 提交间隔 */
    if (new_freq < old_freq) {
        sched->submit_interval_us =
            (sched->submit_interval_us * old_freq) / new_freq;
    }

    adev->pm.last_mclk = new_freq;
}
```

## 3. 电源感知调度策略

### 3.1 频率感知的 Job 调度

```text
调度器根据当前 GPU 频率动态调整 job 提交策略:

  ┌─────────────────────────────────────────────────────────────┐
  │  频率区间       │ 调度策略           │  适用场景             │
  ├────────────────┼───────────────────┼───────────────────────┤
  │ sclk > 2000MHz │ 批量提交           │ 高性能计算/游戏       │
  │               │ (max 16 jobs)      │                       │
  ├────────────────┼───────────────────┼───────────────────────┤
  │ 1000~2000MHz   │ 正常提交           │ 一般工作负载          │
  │                │ (max 8 jobs)       │                       │
  ├────────────────┼───────────────────┼───────────────────────┤
  │ sclk < 1000MHz │ 限制提交           │ 低负载/节能模式       │
  │                │ (max 4 jobs)       │                       │
  ├────────────────┼───────────────────┼───────────────────────┤
  │ sclk 切换中    │ 暂停新 job 提交     │                     │
  │                │ (等待稳定)          │                       │
  └────────────────┴───────────────────┴───────────────────────┘

批量提交的好处:
  - 减少调度开销 (每个 job 的提交延迟)
  - 提高硬件利用率 (MEC 可以预取更多命令)
  - 降低电源管理中断频率

限制提交的好处:
  - 避免在低频率下堆积过多 job
  - 降低响应延迟 (第一个 job 等待时间缩短)
  - 有利于快速回归 idle 状态
```

### 3.2 功耗感知的任务优先级

```text
调度器可为任务分配电源优先级:

  ┌─────────────────────────────────────────────────────────────┐
  │  优先级级别     │ 功耗权重    │ 典型用例                    │
  ├────────────────┼────────────┼─────────────────────────────┤
  │ AMDGPU_PRIO_HIGH│ 1.5x      │ 实时渲染、交互式任务         │
  │ AMDGPU_PRIO_NORM│ 1.0x      │ 普通计算任务                  │
  │ AMDGPU_PRIO_LOW │ 0.5x      │ 后台批处理、离线渲染          │
  └────────────────┴────────────┴─────────────────────────────┘

实现方式:

static enum amd_pm_state_type
amdgpu_sched_job_to_power_state(struct amdgpu_job *job)
{
    switch (job->base.sched_priority) {
    case DRM_SCHED_PRIORITY_HIGH_HW:
    case DRM_SCHED_PRIORITY_HIGH:
        return POWER_STATE_TYPE_PERFORMANCE;
    case DRM_SCHED_PRIORITY_NORMAL:
        return POWER_STATE_TYPE_BALANCED;
    case DRM_SCHED_PRIORITY_LOW:
        return POWER_STATE_TYPE_BATTERY;
    default:
        return POWER_STATE_TYPE_BALANCED;
    }
}

/* 基于任务队列的电源状态决策 */
static enum amd_pm_state_type
amdgpu_sched_queue_power_state(struct amdgpu_ring *ring)
{
    struct amdgpu_job *job;
    enum amd_pm_state_type state = POWER_STATE_TYPE_BATTERY;

    /* 检查队列中所有等待的 job */
    list_for_each_entry(job, &ring->sched.job_list, node) {
        enum amd_pm_state_type job_state =
            amdgpu_sched_job_to_power_state(job);
        if (job_state > state)
            state = job_state;
    }

    /* 如果队列中有高优先级任务, 提升电源状态 */
    return state;
}
```

### 4.1 Ring Buffer 与电源管理

```text

  ┌─────────────────────────────────────────────────────────────┐
  │                    Ring Buffer 结构                         │
  │                                                             │
  │  ┌────┬────┬────┬────┬────┬────┬────┬────┬────┬────┐      │
  │  │J0  │J1  │J2  │J3  │J4  │J5  │    │    │    │    │      │
  │  └────┴────┴────┴────┴────┴────┴────┴────┴────┴────┘      │
  │   ↑                    ↑                                    │
  │   wptr (写指针)        rptr (读指针)                       │
  │   (软件更新)           (硬件更新)                           │
  │                                                             │
  │  电源管理影响:                                              │
  │    - wptr - rptr = 队列深度                                  │
  │    - 队列深度 > 阈值: 认为 GPU 繁忙, 提升频率                │
  │    - 队列深度 = 0: 认为 GPU 空闲, 降低频率                   │
  │    - 阈值可动态调整:                                        │
  │       高频状态: 深度阈值 = 4                                 │
  │       低频状态: 深度阈值 = 2                                 │

Ring buffer 电源状态轮询 (内核代码示意):

static void amdgpu_ring_poll_power(struct amdgpu_ring *ring)
{
    u32 rptr = ring->sched.hw_rptr_count;
    u32 wptr = ring->wptr;
    u32 depth;

    /* 计算队列深度 */
    if (wptr >= rptr)
        depth = wptr - rptr;
    else
        depth = (ring->buf_mask + 1) - rptr + wptr;

    /* 根据队列深度调整电源状态 */
    if (depth > 4) {
        /* 队列深度大: 提升频率 */
        amdgpu_dpm_set_power_state(ring->adev,
            POWER_STATE_TYPE_PERFORMANCE);
    } else if (depth == 0) {
        /* 队列为空: 准备进入低功耗 */
        amdgpu_dpm_set_power_state(ring->adev,
            POWER_STATE_TYPE_BATTERY);
    }
}
```

### 4.2 电源栅栏 (Power Fence)

```text
Power Fence 是一种确保电源状态转换不会影响正在执行 job 的同步机制:

  ┌─────────────────────────────────────────────────────────────┐
  │  Power Fence 工作机制:                                      │
  │                                                             │
  │  1. 调度器请求电源状态转换                                   │
  │  2. 电源管理发出 Power Fence, 阻塞新的 job 提交              │
  │  3. 等待正在执行的 job 完成                                  │
  │  4. GPU 进入安全窗口 (Safe Window)                           │
  │  5. 执行电源状态转换                                         │
  │  6. 释放 Power Fence, 恢复 job 提交                          │
  │                                                             │
  │  时序图:                                                    │
  │                                                             │
  │  调度器                       电源管理                      │
  │    │                             │                          │
  │    │─── set_power_state() ──────▶│                          │
  │    │                             │                          │
  │    │◄── return fence(f) ────────│                          │
  │    │                             │                          │
  │    │  等待 fence 信号            │  等待 GPU 空闲            │
  │    │                             │                          │
  │    │                             │── P-state switch ──▶     │
  │    │                             │                          │
  │    │◄── fence signal ───────────│                          │
  │    │                             │                          │
  │    │  继续提交新的 job           │                          │
  │    ▼                             ▼                          │
  └─────────────────────────────────────────────────────────────┘

内核代码实现:

struct dma_fence *amdgpu_power_fence(struct amdgpu_device *adev)
{
    struct amdgpu_ring *ring;
    struct dma_fence *fence;
    int r;

    /* 创建了一个在所有 ring 完成时触发的 fence */
    fence = dma_fence_allocate_private_stub(adev->ddev);
    if (!fence)
        return ERR_PTR(-ENOMEM);

    /* 等待所有正在执行的 ring 完成 */
    for (i = 0; i < AMDGPU_MAX_RINGS; i++) {
        ring = adev->rings[i];
        if (!ring || !ring->sched.thread)
            continue;

        /* 发出一个空的 sync job 等待完成 */
        r = amdgpu_ring_alloc(ring, 32);
        if (r)
            continue;

        amdgpu_ring_emit_fence(ring, ...);
        amdgpu_ring_commit(ring);
    }

    return fence;
}
```

### 4.3 电源管理中断与调度器

```text

  ┌─────────────────────────────────────────────────────────────┐
  │  电源管理中断类型:                                          │
  │                                                             │
  │  中断源          │ 触发条件              │ 调度器响应        │
  ├─────────────────┼──────────────────────┼───────────────────┤
  │ PPT (PowerTune) │ 功耗超过阈值          │ 降低频率/减少job  │
  │ TDC (电流)      │ 电流超过限制          │ 限制计算密集型job │
  │ TMP (温度)      │ 温度超过阈值          │ 整个调度降级      │
  │ PPC (电源状态)  │ P-state 切换完成      │ 调整提交策略      │

static irqreturn_t amdgpu_power_irq_handler(int irq, void *dev_id)
{
    struct amdgpu_device *adev = dev_id;
    u32 irq_src = RREG32(mmSMUIO_IRQ_SRC);

    switch (irq_src) {
    case SMUIO_IRQ_SRC_PPT:
        /* PowerTune 触发: 立即降频 */
        amdgpu_dpm_set_power_state(adev,
            adev->pm.current_power_state - 1);
        /* 通知调度器削减 job 提交 */
        amdgpu_sched_throttle(adev, true);
        break;

    case SMUIO_IRQ_SRC_TDC:
        /* 电流超限: 减少并行 job 数量 */
        amdgpu_sched_set_max_jobs(adev,
            adev->sched.max_jobs / 2);
        break;

    case SMUIO_IRQ_SRC_TMP:
        /* 温度超限: 全局降级 */
        amdgpu_sched_set_policy(adev,
            SCHED_POLICY_POWER_SAVE);
        break;

    case SMUIO_IRQ_SRC_PPC:
        /* P-state 切换完成: 恢复正常调度 */
        amdgpu_sched_throttle(adev, false);
        break;
    }

    return IRQ_HANDLED;
}
```

```text
static enum amd_pm_state_type
amdgpu_sched_multi_ring_policy(struct amdgpu_device *adev)
{
    bool gfx_busy = false, compute_busy = false;
    bool sdma_busy = false, vcn_busy = false;
    int busy_count = 0;

    /* 检查各引擎的负载 */
    if (amdgpu_ring_is_busy(adev->gfx_ring)) {
        gfx_busy = true;
        busy_count++;
    }
    if (amdgpu_ring_is_busy(adev->compute_ring)) {
        compute_busy = true;
        busy_count++;
    }
    if (amdgpu_ring_is_busy(adev->sdma_ring)) {
        sdma_busy = true;
        busy_count++;
    }

    /* 决策逻辑 */
    if (busy_count >= 3)
        return POWER_STATE_TYPE_PERFORMANCE;  /* 全速 */
    else if (busy_count == 2)
        return POWER_STATE_TYPE_BALANCED;     /* 平衡 */
    else if (busy_count == 1)
        return POWER_STATE_TYPE_BATTERY;      /* 节能 */
    else
        return POWER_STATE_TYPE_BATTERY;      /* 最低 */
}
```

```text
static void amdgpu_sched_partition_cus(struct amdgpu_device *adev,
                                       u32 workload_weight)

    u32 active_cus;
    u32 gated_cus;

    if (workload_weight > 80)
```

### 6.3 延迟感知调度 (Latency-Aware Scheduling)

```text
不同电源状态下的调度延迟分析:

  ┌─────────────────────────────────────────────────────────────┐
  │  P-state    │ GFXCLK    │ 提交延迟(μs)  │ 完成延迟(μs)     │
  ├─────────────┼───────────┼───────────────┼─────────────────┤
  │ P0 (Boost)  │ 2500 MHz  │ 0.5           │ 80              │
  │ P1          │ 2200 MHz  │ 0.6           │ 95              │
  │ P2          │ 2000 MHz  │ 0.8           │ 110             │
  │ P3          │ 1800 MHz  │ 1.0           │ 130             │
  │ P4          │ 1500 MHz  │ 1.5           │ 180             │
  │ P5          │ 1200 MHz  │ 2.0           │ 250             │
  │ P6          │ 800 MHz   │ 3.5           │ 400             │
  │ P7 (最低)   │ 400 MHz   │ 6.0           │ 800             │
  └─────────────────────────────────────────────────────────────┘

  观察:
    - 提交延迟 (从 ioctl 到命令写入 ring): 低频时增加 ~12x
    - 完成延迟 (从提交到 GPU 完成): 低频时增加 ~10x
    - 延迟与频率呈近似线性反比关系

延迟感知调度算法:

/* 计算 job 的预估完成时间 */
static u64 amdgpu_sched_estimate_completion(struct amdgpu_job *job,
                                            u32 freq_mhz)
{
    u64 cycles_needed = job->estimated_cycles;
    u64 time_us = (cycles_needed * 1000000) / (freq_mhz * 1000000);
    return time_us;
}

/* 延迟感知调度决策 */
static bool amdgpu_sched_latency_aware_pick(struct amdgpu_device *adev,
                                            struct amdgpu_job *job)
{
    u32 current_freq = amdgpu_get_sclk(adev);
    u64 completion_time;
    u64 deadline = job->deadline_ns;

    completion_time = amdgpu_sched_estimate_completion(job,
                                                       current_freq);

    /* 如果当前频率下无法满足截止时间, 提升频率 */
    if (completion_time > deadline) {
        u32 required_freq = (job->estimated_cycles * 1000000) /
                            deadline;

        /* 找到能满足截止时间的最低 P-state */
        enum amd_pm_state_type target =
            amdgpu_sched_freq_to_pstate(adev, required_freq);

        if (target < adev->pm.current_power_state) {
            amdgpu_dpm_set_power_state(adev, target);
            return true;  /* 提升了频率 */
        }
    }

    return false;  /* 当前频率已足够 */
}
```

### 7.3 工作负载特征识别

```text
调度器通过分析 job 特征来优化电源管理:

#define MAX_WORKLOAD_CLASSES 8

enum amdgpu_workload_class {
    WORKLOAD_GRAPHICS,       /* 图形渲染 */
    WORKLOAD_COMPUTE,        /* 通用计算 */
    WORKLOAD_MEDIA,          /* 视频编解码 */
    WORKLOAD_COPY,           /* 内存拷贝 */
    WORKLOAD_MIXED,          /* 混合负载 */
    WORKLOAD_IDLE,           /* 空闲 */
    WORKLOAD_UNKNOWN = 0xFF,
};

struct amdgpu_workload_classifier {
    /* 累计统计 */
    u64 class_histogram[MAX_WORKLOAD_CLASSES];
    u32 current_class;
    ktime_t class_start_time;

    /* 转换统计 */
    u32 transitions;
    ktime_t last_transition;

    /* 置信度 */
    u32 confidence;
};

/* 工作负载分类 */
static enum amdgpu_workload_class
amdgpu_sched_classify_workload(struct amdgpu_job *job)
{
    /* 根据 job 的 ring 类型和命令特征分类 */
    if (job->ring->type == AMDGPU_RING_TYPE_GFX)
        return WORKLOAD_GRAPHICS;
    else if (job->ring->type == AMDGPU_RING_TYPE_COMPUTE)
        return WORKLOAD_COMPUTE;
    else if (job->ring->type == AMDGPU_RING_TYPE_SDMA)
        return WORKLOAD_COPY;
    else if (job->ring->type == AMDGPU_RING_TYPE_VCN)
        return WORKLOAD_MEDIA;

    return WORKLOAD_UNKNOWN;
}

/* 基于工作负载的电源策略 */
static enum amd_pm_state_type
amdgpu_sched_workload_policy(enum amdgpu_workload_class wl)
{
    switch (wl) {
    case WORKLOAD_GRAPHICS:
        /* 图形负载: 需要高 GFXCLK, 对延迟敏感 */
        return POWER_STATE_TYPE_PERFORMANCE;

    case WORKLOAD_COMPUTE:
        /* 计算负载: 高 GFXCLK + 高 MCLK, 延迟容忍 */
        return POWER_STATE_TYPE_BALANCED;

    case WORKLOAD_COPY:
        /* 拷贝负载: 仅需 MCLK, GFXCLK 可降至最低 */
        return POWER_STATE_TYPE_BATTERY;

    case WORKLOAD_MEDIA:
        /* 视频负载: 仅需 VCN 时钟, 其他可门控 */
        return POWER_STATE_TYPE_BATTERY;

    default:
        return POWER_STATE_TYPE_BALANCED;
    }
}
```

```bash
#!/bin/bash
# sched_power_tune.sh - 调度器电源管理调优脚本

usage() {
    echo "用法: $0 [mode]"
    echo "  mode: balanced  - 平衡模式 (默认)"
    echo "        performance - 性能模式"
    echo "        powersave   - 节能模式"
    echo "        custom      - 自定义参数"
    exit 1
}

MODE=${1:-balanced}
CARD=${2:-card0}
BASE="/sys/class/drm/$CARD/device"

apply_balanced() {
    echo "应用平衡模式..."
    echo "auto" > "$BASE/power_dpm_state"
    echo "auto" > "$BASE/power_dpm_force_performance_level"
    echo 10 > "$MODULE/sched_job_interval_us"
    echo 3 > "$MODULE/sched_hysteresis"
    echo 0 > "$MODULE/sched_power_policy"
}

apply_performance() {
    echo "应用性能模式..."
    echo "performance" > "$BASE/power_dpm_state"
    echo "high" > "$BASE/power_dpm_force_performance_level"
    echo 1 > "$MODULE/sched_job_interval_us"
    echo 1 > "$MODULE/sched_hysteresis"
    echo 1 > "$MODULE/sched_power_policy"
}

apply_powersave() {
    echo "应用节能模式..."
    echo "low" > "$BASE/power_dpm_state"
    echo "low" > "$BASE/power_dpm_force_performance_level"
    echo 50 > "$MODULE/sched_job_interval_us"
    echo 8 > "$MODULE/sched_hysteresis"
    echo 2 > "$MODULE/sched_power_policy"
}

apply_custom() {
    echo "应用自定义参数..."
    echo "$CUSTOM_STATE" > "$BASE/power_dpm_state"
    echo "$CUSTOM_LEVEL" > "$BASE/power_dpm_force_performance_level"
    echo "$CUSTOM_INTERVAL" > "$MODULE/sched_job_interval_us"

echo "当前配置:"
```

```bash
#!/bin/bash
# sched_bench.sh - 调度器电源管理基准测试
# 测试不同电源策略对调度性能的影响

RESULT_FILE="/tmp/sched_bench_$(date +%s).csv"

echo "策略,提交延迟(μs),完成延迟(μs),吞吐量(jobs/s),功耗(W)" > "$RESULT_FILE"

bench_policy() {
    local policy=$1
    local label=$2

    echo "测试: $label"

    # 设置策略
    echo "$policy" > /sys/class/drm/card0/device/power_dpm_state
    sleep 1  # 等待稳定

    # 运行 GPU 基准测试 (使用 rocm-smi 或 clinfo 负载)
    local duration=10
    local start_time=$(date +%s%N)
    local jobs=0

    while [ $(( $(date +%s%N) - start_time )) -lt $((duration * 1000000000)) ]; do
        # 提交一个简单的 GPU 操作
        rocm-smi --showpower &>/dev/null
        ((jobs++))
    done

    local elapsed_ms=$(( ($(date +%s%N) - start_time) / 1000000 ))
    local throughput=$((jobs * 1000 / elapsed_ms))

    # 读取当前功耗
    local power=$(cat /sys/class/drm/card0/device/hwmon/hwmon*/power1_average 2>/dev/null)
    local power_w=$((power / 1000000))

    echo "$label,0,0,$throughput,$power_w" >> "$RESULT_FILE"
    echo "  吞吐量: $throughput jobs/s, 功耗: ${power_w}W"
}

bench_policy "performance" "performance"
bench_policy "auto" "auto"
bench_policy "low" "low"

echo ""
echo "测试完成! 结果保存在 $RESULT_FILE"
cat "$RESULT_FILE" | column -t -s ','
```

### 10.3 PCIe 子系统交互

```text
PCIe 总线状态与 GPU 调度-电源管理之间存在重要交互:

  ┌─────────────────────────────────────────────────────────────┐
  │  PCIe 链路状态对调度的影响:                                   │
  │                                                             │
  │  PCIe 状态    │ 描述                │ 调度器影响            │
  ├──────────────┼────────────────────┼───────────────────────┤
  │ L0 (全速)    │ 正常操作            │ 最低延迟, 最大带宽     │
  │ L0s (轻度)   │ 快速进入/退出       │ 额外 ~1μs 延迟        │
  │ L1 (中度)    │ 时钟停止            │ 额外 ~10μs 恢复延迟    │
  │ L2 (深度)    │ 主电源关闭          │ 额外 ~100μs+ 恢复延迟  │
  │ L3 (关闭)    │ 完全断电            │ 需要完整重新初始化     │
  └─────────────────────────────────────────────────────────────┘

  PCIe ASPM (Active State Power Management) 与调度:

  ASPM 启用时, PCIe 链路在空闲时自动降入节能状态。
  这对调度器的影响:

    1. SDMA 操作: 大量数据传输前需要等待 PCIe 链路唤醒
    2. 小数据包: 频繁的 L0s/L1 切换增加延迟
    3. 中断传输: MSI/MSI-X 中断可能因链路状态而延迟

  调度器优化策略:

  static void amdgpu_sched_aspm_hint(struct amdgpu_device *adev,
                                     bool data_transfer)
  {
      /* 大量数据传输前阻止 PCIe 进入 L1 */
      if (data_transfer) {
          /* 通过 PCIe 配置空间禁用 ASPM */
          pci_disable_link_state(adev->pdev,
              PCIE_LINK_STATE_L0S | PCIE_LINK_STATE_L1);
      } else {
          /* 传输完成后恢复 ASPM */
          pci_enable_link_state(adev->pdev,
              PCIE_LINK_STATE_L0S | PCIE_LINK_STATE_L1);
      }
  }

  PCIe 带宽与电源状态的协同:

  GPU P-state    PCIe Gen    PCIe 带宽     调度器策略
  P0 (最高)      Gen5 x16    64 GB/s      最大传输粒度
  P3 (中等)      Gen4 x16    32 GB/s      中等传输粒度
  P6 (低)        Gen3 x16    16 GB/s      合并小传输

  调度器根据当前的 PCIe 代际调整 DMA 传输的批量大小:
    - Gen5: 每批最多 256 MB
    - Gen4: 每批最多 128 MB
    - Gen3: 每批最多 64 MB
    - Gen2: 每批最多 32 MB
```

## 11. 总结与展望

### 13.2 性能测试

```text
电源-调度协同的性能测试基准:

  基准测试 1: 延迟灵敏度 (Latency Sensitivity)
    测量在不同 P-state 下, 从 job 提交到开始执行的延迟
    方法:
      - 固定 GPU 在特定 P-state
      - 提交短 job (10μs 执行时间)
      - 测量 fence 信号延迟
      - 每个 P-state 重复 10000 次

  基准测试 2: 电源状态切换开销 (P-state Switching Overhead)
    测量 P-state 切换的延迟和功耗开销
    方法:
      - 循环切换 P0 ↔ P6
      - 记录每次切换延迟
      - 记录切换期间的瞬时功耗
      - 计算平均切换开销

  基准测试 3: 能效曲线 (Energy Efficiency Curve)
    测量不同频率下的能效比
    方法:
      - 固定每个 P-state 运行相同负载
      - 测量 FPS 和平均功耗
      - 计算 FPS/W 能效比
      - 绘制频率-能效曲线

  基准测试 4: 调度器公平性 (Scheduler Fairness)
    验证多进程情况下的电源调度公平性
    方法:
      - 同时运行 N 个 GPU 计算进程
      - 每个进程运行相同负载
      - 测量每个进程的完成时间
      - 计算公平性指数 (Jain's Fairness Index)
```

### 13.3 压力测试

```text
电源-调度协同的压力测试方案:

  压力测试 1: 快速 P-state 切换
    - 在 1 秒内触发 100 次 P-state 切换
    - 同时在 GFX ring 持续提交 job
    - 验证无 fence 超时或 GPU hang

  压力测试 2: 极端负载切换
    - 空载 ↔ 满载以 100ms 间隔交替
    - 持续 30 分钟
    - 监控温度和功耗是否在安全范围内

  压力测试 3: 多进程竞争
    - 32 个进程同时请求 GPU 资源
    - 每个进程随机提交计算/拷贝/显示 job
    - 持续 1 小时
    - 验证调度器电源策略的稳定性

  压力测试 4: 热节流触发
    - 在高温环境下运行满载负载
    - 验证热节流触发后调度器正确降低频率
    - 验证温度稳定后调度器恢复频率
    - 验证无温度振荡 (Thermal Throttling Oscillation)

所有测试结果应记录到测试报告中, 包括:
  - 测试环境 (GPU 型号, 驱动版本, 系统配置)
  - 测试参数和步骤
  - 性能数据 (FPS, 延迟, 功耗, 温度)
  - 通过/失败判定
  - 异常现象记录
```
